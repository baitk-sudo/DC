# 组件设计分析 — 第 6 段：签约与合同管理

本段覆盖**签约全流程**：从待签任务展示、合同/协议查看与签署，到合同生效、归档，以及签约完成后与**进件同步策略（SIGNED）**的衔接（客户、授信、用信、评级、额度、合同、进件等模块的更新）。支持多合同类型（授信合同、借款合同、放款申请书等）与多轮签约场景。

**签约组件定位**：签约组件为系统中的**公共能力组件**，统一支撑**授权协议签订**与**合同签订**；签约操作支持**面客渠道**（客户在手机/PC 等自助签约）与**办贷端 PAD 现场签约**（客户经理携带 PAD 到场、客户在 PAD 上完成签署）。两种入口共用同一套待签任务、合同实例与状态回写逻辑，仅签约入口（渠道标识）不同。

**本段涉及的组件**（办贷管理端 v2 + 渠道端）：

- 签约组件（公共能力：渠道端展示与签署 + 办贷端 PAD 现场签署 + 办贷端任务与状态管理）
- 合同管理组件
- 合同（协议）模版管理组件
- 风控组件（签约前再次风控，可选）
- 消息中心组件、人脸识别服务、安全组件（签约环节验证）
- 进件管理组件（签约完成后更新进件状态并触发同步策略）
- 客户管理、授信管理、用信管理、评级管理、额度管理组件（作为 SIGNED 同步策略的被调用方）

**已对照产品**：湖北农信-公积金贷款（线上/线上转线下）、湖南农信-公积金贷款、长沙银行-税E贷、长沙银行-政采贷。

## 一、签约组件（公共能力：渠道端 + 办贷端 PAD + 办贷端任务与状态）

### 1.1 职责与边界

| 项目 | 说明 |
|------|------|
| **定位** | **公共能力组件**：统一支撑系统中**授权协议签订**与**合同签订**；不区分业务阶段（授权准入、授信签约、用信签约、放款申请书等），均通过同一套任务模型与签署流程完成。 |
| **签约入口** | **面客渠道**：客户在手机 APP、微信、Web 等自助发起签约，查看待签任务列表、加载合同/协议、完成验证与签署。**办贷端 PAD 现场签约**：客户经理使用办贷端 PAD 登录，选择进件/客户后调起该客户待签任务，客户在 PAD 上完成身份验证与电子签署；签署结果与面客渠道一致回传至办贷端，更新任务状态、合同状态并驱动进件同步策略。 |
| **职责** | **签约入口（面客 + 办贷 PAD）**：展示待签任务列表与已签任务、加载待签合同/协议内容、引导客户完成身份验证（短信/人脸等）与电子签署，并将签署结果回传办贷端。**办贷端**：维护签约任务（与进件、合同、客户关联）、任务状态（待签/已签/过期等）、接收各入口的签署结果并更新任务状态、通知合同管理组件更新合同状态。支持多类签约任务：授权协议、授信合同、借款合同、放款申请书、股东会决议等。 |
| **边界** | 不生成合同内容，合同实例与模板由合同管理、合同模版管理组件提供；不执行业务数据同步，签约完成后由进件管理根据同步策略调用各业务组件；不执行风控规则，仅在签约流程中按配置调用风控组件做签约前校验。 |

### 1.2 签约入口：面客渠道 vs 办贷端 PAD 现场签约

| 维度 | 面客渠道 | 办贷端 PAD 现场签约 |
|------|----------|----------------------|
| 使用人 | 客户本人 | 客户经理持 PAD，客户在 PAD 上签署 |
| 场景 | 客户远程自助签约 | 客户经理上门或现场办理，客户现场在 PAD 签字 |
| 待签任务来源 | 与办贷端一致（同一套任务表，按客户/进件查询） | 客户经理选择进件/客户后，签约组件按进件或客户返回待签任务列表 |
| 签署流程 | 列表 → 选任务 → 验证 → 看合同 → 签署 → 回传 | 客户经理在 PAD 上选进件/客户 → 调起待签 → 客户在 PAD 上验证、看合同、签署 → 回传 |
| 状态与同步 | 签署完成后与办贷端 PAD 签约共用同一套状态更新与进件 SIGNED 同步策略 | 同上，无差异 |

**设计要点**：签约组件对外暴露的「待签任务列表」「发起签约」「提交签署结果」等接口需支持**渠道标识**（如 `channel=PORTAL` / `channel=PAD`），以便审计与统计；业务逻辑与进件状态、合同状态、同步策略的触发保持一致，不因入口不同而分支。

### 1.3 多产品 / 多模式支持点

- **两类签约内容**：**授权协议**（授权准入阶段等）与**合同**（授信合同、借款合同、放款申请书、股东会决议等），均由签约组件统一承载签署流程，合同/协议内容与模板由合同管理、合同模版管理组件提供。
- **多任务类型**：支持授权协议、授信合同、借款合同、放款申请书、股东会决议等任务类型，按产品与流程阶段配置不同任务类型与合同类型对应关系。
- **多轮签约**：同一进件/授信下可能有多轮签约（如先签授信合同、后签借款合同、提款时再签放款申请书），签约组件按任务顺序展示，支持按任务类型与合同类型区分。
- **签约前校验**：可按产品配置签约前是否再次执行风控准入、是否强制人脸/短信验证，由签约组件调用风控、消息中心、人脸识别等通用能力；面客与 PAD 现场签约共用同一套校验配置。

### 1.4 关键接口与上下游关系

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 待签任务列表查询 | 按客户/进件返回待签任务列表，含任务类型、合同摘要、截止时间等；**面客渠道**与**办贷 PAD** 共用，调用方传入渠道标识（如 channel） | 下游：签约任务表、合同管理（合同摘要）；上游：面客渠道 / 办贷 PAD |
| 发起签约任务 | 从任务列表进入某一待签任务，初始化签约流程；支持由客户（面客）或客户经理在 PAD 上为当前客户调起 | 下游：合同管理（合同实例与渲染数据）；上游：面客渠道 / 办贷 PAD |
| 加载待签合同内容 | 从合同管理组件获取合同实例及填充数据，从合同模版管理获取模板结构用于展示 | 下游：合同管理组件、合同（协议）模版管理组件；上游：签约组件（任一签约入口） |
| 签约前风控/身份验证 | 按配置调用风控组件、短信验证码、人脸识别、安全组件 | 下游：风控组件、消息中心、人脸识别、安全组件；上游：签约流程 |
| 提交签署结果 | 客户完成签署后，签约入口（面客或 PAD）将签署结果（签名数据、时间戳、渠道标识等）回传办贷端 | 下游：签约组件（办贷）、合同管理组件（合同生效）；上游：面客渠道 / 办贷 PAD |
| 更新签约任务状态 | 办贷端将任务状态更新为已签，并触发合同管理将合同状态更新为已签署/生效 | 下游：合同管理组件；上游：签约组件（办贷） |
| 生成签约任务（办贷） | 第 4 段直通或第 5 段审批通过后，由进件管理/工作流/合同管理触发，创建待签任务并关联合同、进件、客户；任务可被面客渠道或办贷 PAD 拉取并执行签署 | 下游：签约任务表；上游：进件管理、工作流、合同管理 |

### 1.5 对已分析产品的支撑检查

| 产品 | 本段相关步骤 | 说明 |
|------|--------------|------|
| 湖北/湖南 公积金 | 客户在线签定借款合同：发起签约、风控、人脸、查看合同、短信、签授权协议、签借款合同，更新客户/用信/授信/评级/合同/签约/进件状态 | 签约组件支持借款合同签署及签署后状态回写；晚同步时进件管理在 SIGNED 触发各模块同步 |
| 长沙银行-税E贷 | 客户在线签定授信合同：流程同上，签授信合同后更新用信/授信/评级/合同/签约/进件；后续提款时可能再签放款申请书 | 签约组件支持授信合同类型及多轮签约（授信合同→放款申请书） |
| 长沙银行-政采贷 | 客户在线签定借款合同：额度激活后贷款申请生成借款合同与签约任务，客户签署借款合同后更新合同/签约/进件 | 签约组件支持借款合同类型，与合同管理、进件管理协同完成状态更新 |

## 二、合同管理组件

### 2.1 职责与边界

| 项目 | 说明 |
|------|------|
| **职责** | 维护合同实例（授信合同、借款合同、放款申请书、授权协议等）的全生命周期：生成合同（草稿/待签）、关联进件/客户/授信/用信、提供合同内容与填充数据给签约组件展示、接收签署结果后更新合同状态为已签署/生效并归档；支持合同版本、变更与查询。 |
| **边界** | 合同模板结构与变量由合同（协议）模版管理组件维护，合同管理按模板与业务数据生成合同实例；不执行签约流程，仅提供合同数据并接收签署结果；不直接驱动客户/授信/用信/评级/额度等模块的同步，这些由进件管理在 SIGNED 时按同步策略调用各业务组件。 |

### 2.2 多产品 / 多模式支持点

- **多合同类型**：支持授信合同、借款合同、放款申请书、授权协议、股东会决议等类型，按产品与流程阶段配置生成哪类合同；同一进件/授信下可存在多份合同（如先授信合同后借款合同）。
- **合同与进件/授信/用信关联**：合同实例关联进件号、客户、授信、用信等，便于签约时展示与签署后同步策略按进件维度调用各模块 upsert。
- **签署后状态与归档**：合同状态包括草稿、待签、已签署、生效、归档等；签署完成后合同管理更新状态并通知签约组件，进件管理在进件状态迁至 SIGNED 时可按策略再次调用合同管理的「终态确认」或仅依赖合同管理已在签署时完成的生效操作。

### 2.3 关键接口与上下游关系

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 生成合同实例 | 按产品、进件、授信、用信等生成合同（草稿/待签），关联进件号与客户、授信、用信 | 下游：合同表；上游：进件管理（同步策略）、工作流、授信管理（第 5 段审批通过后） |
| 查询合同内容与填充数据 | 为签约组件提供合同实例内容及变量填充结果，供渠道展示与签署 | 下游：无；上游：签约组件 |
| 合同生效/归档 | 接收签约组件回传的签署结果，将合同状态更新为已签署、生效，并归档 | 下游：合同表、归档存储；上游：签约组件 |
| 按进件更新合同（同步策略） | 进件状态 SIGNED 时，进件管理可调用合同管理的 upsertByIntake 或终态确认接口，确保合同与业务数据一致 | 下游：合同表；上游：进件管理组件 |

### 2.4 对已分析产品的支撑检查

| 产品 | 本段相关步骤 | 说明 |
|------|--------------|------|
| 湖北/湖南 公积金 | 签借款合同，合同生效，晚同步下签约时写客户/授信/用信/评级/合同 | 合同管理提供合同实例与生效能力，与签约、进件同步策略协同 |
| 税E贷 | 签授信合同，后续提款签放款申请书；授信合同与放款申请书为不同合同类型 | 合同管理支持多合同类型及同一客户/授信下多份合同 |
| 政采贷 | 签借款合同（额度激活后贷款申请生成），合同生效后更新合同/签约/进件 | 合同管理支持借款合同类型及与进件、用信关联 |

## 三、合同（协议）模版管理组件

### 3.1 职责与边界

| 项目 | 说明 |
|------|------|
| **职责** | 维护各类合同与协议的模板（条款结构、变量定义、版本管理），按合同类型、产品、版本提供模板内容；为合同管理组件生成合同实例提供模板与变量规则，为签约组件展示合同提供模板结构。 |
| **边界** | 不生成具体合同实例，不存储客户/授信/用信等业务数据；仅提供模板与变量定义，合同内容生成由合同管理组件负责。 |

### 3.2 多产品 / 多模式支持点

- **按合同类型与产品配置模板**：授信合同、借款合同、放款申请书、各类授权协议等可有不同模板，同一类型下不同产品可配置不同模板或版本。
- **变量与条款可配置**：支持占位变量与条款库，合同管理生成实例时按业务数据填充，便于多产品复用与合规条款维护。

### 3.3 关键接口（本段相关）

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 按合同类型/编码/版本查询模板 | 返回模板结构、条款与变量定义，供合同管理生成实例或签约展示 | 下游：无；上游：合同管理组件、签约组件（展示用） |

### 3.4 对已分析产品的支撑检查

| 产品 | 本段相关步骤 | 说明 |
|------|--------------|------|
| 各产品 | 加载查看合同模板、签约时展示合同内容 | 合同模版管理为合同管理、签约组件提供模板，满足多合同类型与多产品 |

## 四、风控组件、消息中心、人脸识别、安全组件（本段）

### 4.1 本段职责与边界

| 组件/能力 | 本段职责 | 边界 |
|-----------|----------|------|
| **风控组件** | 签约前再次风控准入（可选），按产品配置是否执行及规则包，返回通过/拒绝 | 不改变合同或签约状态，仅输出风控结论供签约流程决定是否继续 |
| **消息中心** | 签约环节发送并验证短信验证码 | 不承载业务逻辑，仅提供验证码能力 |
| **人脸识别服务** | 签约环节人脸识别与活体检测 | 不承载业务逻辑，仅输出识别结果 |
| **安全组件** | 提款等场景的提款密码校验等（本段若涉及放款申请书签署后的提款，由放款段落使用） | 本段签约环节主要依赖短信、人脸 |

### 4.2 对已分析产品的支撑检查

| 产品 | 本段相关步骤 | 说明 |
|------|--------------|------|
| 各产品 | 签约前客户风控准入、验证短信验证码、人脸识别操作 | 风控、消息中心、人脸识别按配置被签约组件调用，满足 |

## 五、进件管理组件（本段：签约完成与同步策略）

### 5.1 本段职责与边界

| 项目 | 说明 |
|------|------|
| **本段职责** | 在客户完成合同签署后，签约组件或合同管理将「签约完成」事件通知进件管理（或进件管理轮询/订阅签约状态）；进件管理将进件状态更新为 SIGNED，并依据《进件中心-业务数据同步策略设计》中的同步策略表，对本产品在 SIGNED 状态下配置的业务模块执行同步（客户、评级、授信、用信、额度、合同等），调用各模块的 upsertByIntake 或终态确认接口。 |
| **边界** | 同步动作与范围由同步策略配置决定，进件管理不写死模块列表；各业务模块的幂等与数据一致性由各模块自身保证。 |

### 5.2 多产品 / 多模式支持点

- **晚同步**：湖北等机构在 SIGNED 时才向客户、评级、授信、用信、额度、合同等模块写入或终态确认，进件管理在 SIGNED 时按策略一次性调用上述模块。
- **早同步**：湖南/长沙等机构在 RECEIVED/APPROVED 已初始化，SIGNED 时仅做更新/终态确认，进件管理在 SIGNED 时按策略调用各模块的「更新/终态确认」动作。
- **按产品配置**：不同产品可在同步策略表中配置不同的 SIGNED 触发模块与动作，进件管理按产品+状态查询策略并执行。

### 5.3 关键接口（本段相关）

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 进件状态更新为 SIGNED | 签约完成后更新进件单状态，并发布领域事件或触发同步策略执行 | 下游：同步策略执行→客户/评级/授信/用信/额度/合同等；上游：签约组件或合同管理（签约完成通知） |
| 执行 SIGNED 同步策略 | 查询同步策略表，对配置的模块依次调用 upsertByIntake 或终态确认 | 下游：客户管理、评级管理、授信管理、用信管理、额度管理、合同管理；上游：进件管理（状态迁移时） |

### 5.4 对已分析产品的支撑检查

| 产品 | 本段相关步骤 | 说明 |
|------|--------------|------|
| 湖北 公积金 | 签约后创建/更新正式客户、用信、授信、评级、合同，更新签约状态与进件状态；晚同步 | 进件管理在 SIGNED 执行晚同步策略，与文档设计一致 |
| 湖南/长沙 公积金、税E贷、政采贷 | 签约后更新用信、授信、评级、合同、进件状态；早同步下为更新/终态确认 | 进件管理在 SIGNED 执行早同步产品的更新/终态策略，满足 |

## 六、客户管理、授信管理、用信管理、评级管理、额度管理（本段：同步策略被调用方）

### 6.1 本段角色

上述组件在本段不主动发起签约流程，而是在**进件状态迁至 SIGNED 时**，被进件管理组件按同步策略依次调用，执行「按进件号幂等」的写入或终态确认：

| 模块 | 在 SIGNED 时的典型动作（由同步策略配置决定） |
|------|----------------------------------------------|
| 客户管理 | 创建/更新正式客户，临时客户转正式，补全客户信息 |
| 授信管理 | 写入或更新授信结论、有效期，终态确认 |
| 用信管理 | 写入或更新用信记录，终态确认 |
| 评级管理 | 写入或更新评级结果 |
| 额度管理 | 正式占用额度或更新额度状态 |
| 合同管理 | 合同生效、归档，与客户/授信/用信关联（若签约时已生效，可为终态确认或无需二次调用） |

各组件需提供 upsertByIntake 或等效接口，并保证幂等，详见《进件中心-业务数据同步策略设计》。

### 6.2 对已分析产品的支撑检查

| 产品 | 说明 |
|------|------|
| 湖北 公积金 | 晚同步：SIGNED 时一次性写入客户、授信、用信、评级、合同、额度，各模块提供接口即可 |
| 湖南/长沙 公积金、税E贷、政采贷 | 早同步：SIGNED 时对各模块执行更新/终态确认，各模块支持按进件号幂等更新即可 |

## 七、本段组件协作关系小结

```text
[渠道] 待签任务列表 <-- 签约组件（办贷：任务与合同关联）
   |
   v
客户发起签约 --> 签约组件（渠道）--> 合同管理（合同内容）--> 合同模版管理（模板）
   |
   v
风控（可选）、短信、人脸 --> 客户签署 --> 签约组件提交签署结果
   |
   v
签约组件（办贷）--> 更新任务状态 --> 合同管理（合同生效/归档）
   |
   v
进件管理（进件状态 SIGNED）--> 查询同步策略 --> 客户/评级/授信/用信/额度/合同 等 upsertByIntake 或终态确认
```

## 八、多合同类型与多轮签约小结

| 场景 | 合同类型 | 典型产品 | 说明 |
|------|----------|----------|------|
| 直通或审批后签借款合同 | 借款合同 | 湖北/湖南 公积金 | 进件 R1 或审批通过后生成借款合同与签约任务，客户签一份借款合同即完成 |
| 先签授信合同后提款 | 授信合同 + 放款申请书 | 税E贷 | 审批通过后生成授信合同与签约任务，客户签授信合同；提款时再签放款申请书 |
| 额度激活后签借款合同 | 借款合同 | 政采贷 | 贷款申请通过后生成借款合同与签约任务，客户签借款合同 |
| 授权协议 | 授权协议 | 各产品 | 在授权准入或申请内签署，签约组件支持该任务类型，合同管理可归档授权记录 |

合同管理组件与签约组件通过**合同类型 + 任务类型**配置化，支持上述多产品、多轮签约，无需为每种合同写死流程。

## 九、与前序及后续段落衔接

- **与第 5 段（三合一任务与审批）**：审批通过后生成合同与签约任务，本段从「客户或客户经理看到待签任务」开始（面客渠道或办贷 PAD），到签署完成、进件状态 SIGNED 及同步策略执行结束。
- **与《进件中心-业务数据同步策略设计》**：本段进件管理在 SIGNED 时的行为与该文档完全一致，实现晚同步/早同步下签约节点对各业务模块的驱动。
- **与后续放款段落**：客户签约完成后，若为借款合同或授信合同+放款申请书，将进入提款/放款流程；放款时可能再次使用签约组件签署放款申请书，逻辑与本段一致，仅合同类型与业务阶段不同。

## 十、签约组件作为公共能力对前序任务的影响

将签约组件明确为**公共能力组件**、并支持**面客渠道 + 办贷端 PAD 现场签约**后，对前序设计与文档的衔接要求如下，便于实施时一致落地：

| 前序任务/文档 | 影响与需同步的设计点 |
|---------------|----------------------|
| **第 3 段（授权准入与规则触发）** | 签约组件在授权阶段承担「授权协议签订」，应统一视为公共能力：授权协议签署入口可为**面客渠道**（客户在 APP/Web 自助签）或**办贷端 PAD**（客户经理现场让客户在 PAD 上签授权协议）。授权配置、合同模版、签约任务生成与状态回写逻辑不变，仅调用签约能力时需支持渠道标识（面客 / PAD）。 |
| **第 3.5 段（授信/贷款申请与提额）** | 申请提交后若涉及授权或后续签约，生成的签约任务同样可下发给面客或办贷 PAD，无需在申请组件内区分签约入口；签约组件统一按「待签任务」对外服务。 |
| **第 4 段（进件接收与风控准入）** | 直通分支下「生成签约任务」后，任务可被**面客渠道**或**办贷 PAD** 拉取并执行签署；进件管理、工作流仅负责生成任务并关联进件/合同/客户，不限定签约必须在某一端完成。路由与分配逻辑若需区分「线上自助签约」与「现场 PAD 签约」，可在进件/工作流中通过产品配置或任务属性标识，签约组件仍以同一套接口与状态机响应。 |
| **第 5 段（三合一任务与审批）** | 审批通过后生成的合同与签约任务，同样支持客户在**面客渠道**签约或客户经理带 **PAD 现场**签约；授信管理、工作流、合同管理、签约组件的职责不变，仅需约定：待签任务生成后由签约组件统一对外提供「待签列表」「发起签约」「提交签署结果」，面客与 PAD 共用同一套任务与状态。 |
| **合同管理 / 合同模版管理** | 不因签约入口（面客 / PAD）而区分合同实例或模板；合同管理仅按进件、客户、合同类型提供合同内容与生效接口，签约组件负责多入口的展示与回写。 |
| **进件同步策略（SIGNED）** | 无论签约发生在面客渠道还是办贷 PAD，签约完成后均应由签约组件/合同管理将「签约完成」事件通知进件管理，进件管理将进件状态更新为 SIGNED 并执行同步策略；**不区分签约入口**，避免重复或遗漏同步。 |

**小结**：前序各段中凡涉及「签约」「待签任务」「授权协议签署」的设计，均默认由**同一签约组件**以公共能力形式提供；面客与办贷 PAD 仅为两种**签约入口**，共用任务模型、状态与进件同步策略，无需为 PAD 单独再建一套签约流程。

本文件为「按业务办理过程顺序」组件分析的第 6 段；后续段落可继续分析**放款与受托支付审批**、**还款**、**贷后与预警**等。
