# 组件设计分析 — 第 1 段：营销获客与预授信

按**业务办理过程顺序**进行组件分析，本段覆盖流程主线中的**营销获客 / 预授信白名单**环节，确保该段内组件对已分析的全部产品支撑不遗漏。

**本段涉及的组件**（办贷管理端 v2 + 渠道端）：

- 营销方案管理组件（办贷 + 渠道展示）
- 预授信组件（办贷 + 渠道）
- 征信与外部数据管理组件（办贷）
- 风控组件（办贷，本段侧重白名单校验与预授信规则）

**已对照产品**：湖北农信-公积金贷款（线上/线上转线下）、湖南农信-公积金贷款、长沙银行-税E贷、长沙银行-政采贷。

**本段涉及的限界上下文**：产品与营销、外部数据、风控（见《限界上下文与集成关系-ContextMap.md》）。

---

## 一、营销方案管理组件

### 1.1 职责与边界

| 项目 | 说明 |
|------|------|
| **职责** | 定义并管理「营销产品」与可售产品的关系、对客展示信息、申请/进件要素配置；为渠道提供产品列表与详情、申请页要素、营销活动与溯源能力（如二维码/链接）。 |
| **边界** | 不定义产品本身的金融参数（期限、担保方式、利率结构等），由**产品管理组件**提供；不执行风控或额度测算，仅提供配置与查询服务；不存储客户级额度或合同数据，额度/合同查询由**额度管理**/**合同管理**提供。 |

### 1.2 多产品 / 多模式支持点

- **营销产品与可售产品**：一个营销产品映射一个可售产品，支持多营销产品对应同一可售产品（不同活动、不同渠道）。新产品仅需在营销方案中新增映射与展示配置。
- **对客展示**：产品名称、简介、额度范围、利率说明、申请入口等按营销产品配置，支持按渠道、客户标签、白名单过滤可见产品。
- **申请要素**：贷款申请要素、授信申请要素、进件管理查看要素分别配置，避免为不同产品写死表单。
- **营销溯源**：支持按营销方案/客户经理生成唯一二维码或链接，记录扫码/点击事件，用于转化统计与业绩归因。

### 1.3 关键接口与上下游关系

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 产品列表查询（对客） | 按渠道、客户标识、标签返回可见营销产品列表 | 下游：产品管理（可售产品）、额度/合同（可用额等可选）；上游：渠道 |
| 产品详情查询（对客） | 返回营销产品详情及展示文案、申请入口 | 下游：产品管理（产品参数）、利率与费用（优惠券入口）；上游：渠道 |
| 申请页要素查询 | 按营销产品+申请类型（贷款申请/授信申请）返回表单要素、必填项、校验规则 | 下游：产品管理；上游：渠道贷款申请/授信申请组件 |
| 进件查看要素配置 | 按营销产品返回进件管理端需展示/补录的要素 | 下游：进件管理组件使用；上游：办贷进件管理 |
| 营销二维码/链接生成 | 按营销方案+客户经理等生成唯一链接，并记录扫码事件 | 下游：无；上游：渠道、客户经理端 |
| 营销产品与可售产品映射维护 | 办贷端配置营销产品、映射关系、上下架 | 下游：产品管理（可售产品列表）；上游：产品管理 |

### 1.4 对已分析产品的支撑检查

| 产品 | 本段内相关步骤 | 组件支撑 | 是否遗漏 |
|------|----------------|----------|----------|
| 湖北农信-公积金-线上/线下 | 查询产品信息：产品列表、产品详情；查询利率优惠券、额度可用额、合同可用额、最高可贷额 | 营销方案提供列表与详情封装；利率/额度/合同/产品管理由各组件提供，营销方案作为入口或聚合层 | 无遗漏；需支持按渠道/产品过滤列表 |
| 湖南农信-公积金 | 分享营销产品二维码、客户微信扫码；客户查询产品信息（同上） | 营销方案支持二维码生成与扫码记录、产品列表与详情 | 无遗漏 |
| 长沙银行-税E贷 | 客户查询产品信息（同上）；授信申请页初始化来自营销方案 | 同上；另需支持「授信申请」类型申请页要素配置 | 无遗漏 |
| 长沙银行-政采贷 | 客户查询产品信息（同上）；授信申请、贷款申请页要素 | 同上；需同时支持授信申请与贷款申请要素配置 | 无遗漏 |

**结论**：营销方案管理组件在职责上可覆盖本段及后续「申请页、进件查看」需求；需确保**产品列表/详情/申请要素/进件要素**均按营销产品+产品类型配置化，避免硬编码。

## 二、预授信组件

### 2.1 职责与边界

| 项目 | 说明 |
|------|------|
| **职责** | 统一管理「预授信记录」，支持两类模式：**产品预授信**（批处理生成名单、客户经理代客预授信、外部/政采平台推送白名单）与 **客户预授信**（客户在指定产品下主动发起预授信申请）；提供预授信试算、额度展示与结果查询能力。 |
| **边界** | 不直接实现风控规则或额度测算逻辑，而是调用**风控组件**执行产品/客户预授信规则；不直接对接外部系统，白名单推送由**征信与外部数据管理组件**接入后再由本组件消费；预授信结果不等同于正式授信或额度，正式授信在**授信管理组件**，额度占用在**额度管理组件**。 |

### 2.2 多产品 / 多模式支持点

- **预授信类型**：在统一预授信模型中引入 `precredit_type` 枚举，至少包括：
  - `PRODUCT`：产品预授信（批处理、客户经理代客、外部/政采平台推送产生的名单/额度）
  - `CUSTOMER`：客户预授信（客户在某产品下主动申请，按客户+产品维度产生记录）
- **白名单来源**：支持多来源配置（客户经理维护、文件导入、外部数据平台推送、政采平台推送、公积金平台等），按产品/营销方案配置使用哪些来源，均归并为 `PRODUCT` 类型记录。
- **名单粒度**：支持按产品、客户、额度、有效期等维度管理预授信记录；支持「仅名单」与「名单+预授信额度」两种模式。
- **批处理生成**：批处理任务按规则（如缴存、纳税、政采数据）加工生成产品预授信名单，规则可配置，不写死在产品中。
- **客户经理代客**：办贷端支持客户经理代客户发起预授信，可生成产品预授信名单（PRODUCT）或直接生成客户预授信记录（CUSTOMER），按配置决定。
- **渠道侧预授信申请（客户预授信）**：客户在指定产品下发起预授信申请时，预授信组件按客户+产品维度调用风控/额度测算返回预授信额度并写入 `CUSTOMER` 类型记录；是否支持由产品配置决定。

### 2.3 关键接口与上下游关系

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 白名单维护（办贷） | 客户经理新增/编辑/导入/删除产品预授信白名单条目，按产品与有效期 | 下游：风控（可选校验）；上游：办贷端 |
| 白名单推送接收 | 接收外部数据管理组件转发的推送数据，去重、落库为 `PRODUCT` 类型记录、可选触达风控 | 下游：风控组件（校验）；上游：征信与外部数据管理组件 |
| 批处理生成名单 | 按配置规则从数据源加工生成产品预授信名单并写入（`PRODUCT` 类型） | 下游：风控/额度测算（若需）；上游：批处理组件调度 |
| 代客预授信申请（办贷） | 客户经理代客发起，按配置生成产品预授信名单或客户预授信记录 | 下游：风控组件（预授信规则）；上游：办贷端 |
| 客户预授信申请（渠道） | 客户在指定产品下发起预授信申请，形成 `CUSTOMER` 类型预授信记录并返回预授信额度 | 下游：风控/额度测算；上游：渠道预授信组件 |
| 预授信额度查询（渠道） | 按客户+产品维度查询预授信结果（优先 CUSTOMER 类型，其次 PRODUCT 类型）并展示 | 下游：无；上游：渠道预授信组件、申请页 |
| 是否在白名单/预授信结果查询 | 进件或申请时查询客户是否在产品预授信名单或已有客户预授信结果 | 下游：无；上游：进件管理、渠道申请页 |

### 2.4 对已分析产品的支撑检查

| 产品 | 本段内相关步骤 | 组件支撑 | 是否遗漏 |
|------|----------------|----------|----------|
| 湖北农信-公积金-线上/线下 | CSV 未单独列出预授信阶段 | 可选能力；若产品有预授信，由预授信组件+风控支持 | 无遗漏 |
| 湖南农信-公积金 | 场景1 客户经理维护预产品授信白名单；场景2 外部数据平台推送产品授信白名单 | 预授信组件支持两种来源；外部推送经征信与外部数据管理接入 | 无遗漏 |
| 长沙银行-税E贷 | 场景1 客户经理维护；场景2 外部数据平台推送 | 同湖南农信 | 无遗漏 |
| 长沙银行-政采贷 | 政采平台推送产品授信白名单 | 预授信组件支持「政采平台」来源；推送由征信与外部数据管理对接政采平台后转预授信 | 无遗漏 |

**结论**：预授信组件需在设计中显式支持**多白名单来源**（含政采平台）及**批处理/代客/渠道申请**多种生成与使用方式；与征信与外部数据管理、风控的接口边界清晰，不直接对接外部系统。

## 三、征信与外部数据管理组件

### 3.1 职责与边界

| 项目 | 说明 |
|------|------|
| **职责** | 统一接入人行征信、百行征信、公安联网核查、工商司法、政采平台、税务、公积金等外部数据源；管理查询授权与查询记录留痕；接收并转发外部推送（如白名单推送）；对账与监控。 |
| **边界** | 不执行业务规则或决策，仅提供标准化数据服务与接入管理；风控、评级、预授信等组件消费其数据，不做反向依赖；不存储业务结论（如授信结论、额度），仅存储原始或加工后的外部数据及调用记录。 |

### 3.2 多产品 / 多模式支持点

- **多数据源**：按数据源类型配置接入（征信、公安、工商、政采、税务、公积金等），新产品只需配置是否使用某数据源及授权项。
- **授权与留痕**：按产品与业务环节配置所需授权项；每次查询留痕，满足合规与审计。
- **推送接入**：支持外部系统主动推送（如政采平台、外部数据平台）的协议与幂等处理，转发给预授信等消费方。
- **客户信息推送**：支持外部业务平台推送客户基础信息（如政采/公积金/合作平台客户），按配置决定是仅落为客户档案（由客户管理组件消费），还是触发自动注册流程（由用户注册组件消费）。
- **统一出口**：风控、评级、进件、预授信等均通过本组件获取外部数据，避免各业务重复对接。

### 3.3 关键接口与上下游关系

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 征信/外部数据查询 | 按授权与数据源类型查询，返回标准化结果 | 下游：人行/百行/第三方；上游：风控、评级、进件、授权校验 |
| 授权校验与查询记录 | 校验是否已授权、记录查询行为 | 下游：无；上游：风控、评级、预授信、进件 |
| 公安联网核查 | 身份证件联网核查 | 下游：公安接口；上游：进件、客户管理 |
| 工商/司法数据查询 | 企业信息、股东、司法风险等 | 下游：工商/司法数据源；上游：股东会决议、风控、预授信 |
| 政采/外部平台推送接收 | 接收推送、去重、落库并转发预授信等 | 下游：预授信组件；上游：政采平台、外部数据平台 |
| 白名单推送转发 | 将推送数据按配置转发给预授信组件 | 下游：预授信组件；上游：本组件推送接收 |
| 客户信息推送转发 | 将推送的客户信息规范化后，按配置转发给客户管理组件（建档）和/或用户注册组件（自动注册） | 下游：客户管理组件、用户注册组件；上游：本组件推送接收 |

### 3.4 对已分析产品的支撑检查

| 产品 | 本段内相关步骤 | 组件支撑 | 是否遗漏 |
|------|----------------|----------|----------|
| 湖北农信-公积金 | 本段无直接外部数据步骤；后续进件/风控会用到征信、公积金等 | 本段不直接出现；进件段风控准入需征信与外部数据 | 本段无遗漏 |
| 湖南农信-公积金 | 外部数据平台推送白名单 | 征信与外部数据管理接收推送并转发预授信 | 无遗漏 |
| 长沙银行-税E贷 | 外部数据平台推送白名单；后续股东会决议需工商司法 | 同湖南；股东会决议段使用工商司法 | 本段无遗漏 |
| 长沙银行-政采贷 | 政采平台推送白名单 | 征信与外部数据管理对接政采平台、接收推送并转发预授信 | 无遗漏 |

**结论**：本段内主要依赖为**白名单推送接入**（外部数据平台、政采平台）；需在设计中明确推送协议、幂等与转发规则，确保预授信组件不直接对接外部系统。

## 四、风控组件（本段内职责）

### 4.1 职责与边界（本段）

| 项目 | 说明 |
|------|------|
| **本段职责** | 预授信白名单校验与预授信规则执行；客户经理维护或外部推送的白名单经风控做合规/风险校验（可选）；客户发起预授信申请时执行预授信规则并返回预授信额度（可选）。 |
| **边界** | 规则与规则包配置化，按产品/场景选择；不直接调用外部数据，通过**征信与外部数据管理组件**获取数据；准入结论（通过/转人工/拒绝）在进件段使用，本段仅产出预授信相关结果。 |

### 4.2 多产品 / 多模式支持点

- **规则包**：按产品或营销产品配置预授信规则包、白名单校验规则，新产品通过配置接入。
- **数据依赖**：通过征信与外部数据管理获取征信、税务、政采等数据，风控只做规则执行与结果输出。
- **与预授信协作**：预授信组件负责名单与额度展示，风控负责规则执行；接口上可为「预授信调用风控执行某规则包」或「风控回调预授信写结果」。

### 4.3 关键接口与上下游关系（本段）

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 白名单校验（预授信） | 对预授信名单条目或推送数据执行校验规则 | 下游：征信与外部数据管理（取数）；上游：预授信组件、批处理 |
| 预授信规则执行 | 按客户/进件/产品执行预授信规则，返回通过与否及预授信额度 | 下游：征信与外部数据管理、额度测算逻辑；上游：预授信组件、渠道预授信申请 |
| 规则包配置 | 按产品/场景配置规则包，供预授信与后续进件准入使用 | 下游：无；上游：授权配置管理（可选关联）、办贷配置 |

### 4.4 对已分析产品的支撑检查

| 产品 | 本段内相关步骤 | 组件支撑 | 是否遗漏 |
|------|----------------|----------|----------|
| 湖北农信-公积金 | 无预授信阶段 | — | 无遗漏 |
| 湖南农信-公积金 | 场景1/2 预授信白名单，调用智能决策 | 风控组件执行白名单/预授信规则 | 无遗漏 |
| 长沙银行-税E贷 | 场景1/2 预授信白名单，调用智能决策 | 同上 | 无遗漏 |
| 长沙银行-政采贷 | 政采平台推送白名单，调用智能决策 | 风控对推送名单做校验或预授信规则 | 无遗漏 |

**结论**：本段中风控组件以「预授信规则与白名单校验」为主，与预授信、征信与外部数据管理边界清晰；进件段风控准入在后续段落分析。

## 五、本段组件协作关系小结

```text
[渠道]                     [办贷 v2]
   |                            |
   |  产品列表/详情/申请要素      |  营销方案管理组件  <-- 产品管理（可售产品）
   |  <------------------------  |
   |  预授信申请/额度展示         |  预授信组件  ------> 风控组件（规则）
   |  <------------------------  |       ^
   |  扫码/链接                   |       | 白名单推送
   |                            |  征信与外部数据管理组件  <-- 政采/外部平台/人行等
```

- **营销方案管理**：为渠道提供产品与申请页能力，依赖产品管理，不依赖预授信。
- **预授信**：依赖风控做规则、依赖征信与外部数据管理接收推送与取数；不直接对接外部系统。
- **征信与外部数据管理**：统一接入与推送，为风控、预授信、后续进件/评级提供数据。
- **风控**：本段仅承担预授信相关规则与白名单校验，数据通过征信与外部数据管理获取。

---

## 六、与后续段落的衔接

- **客户注册/认证**（第 2 段）：本段不涉及；营销方案可能按「是否登录/是否认证」过滤产品列表。
- **授权准入**（第 3 段）：授权配置管理组件定义各阶段授权项；本段不承担授权，仅营销展示与预授信。
- **进件与同步**：进件接收时可查询「是否在白名单/是否有预授信结果」用于路由或展示；预授信组件仅提供查询，不进件单存储。

本文档为「按业务办理过程顺序」组件分析的第 1 段；第 2 段为**客户注册与认证**，涉及用户注册组件、客户管理组件、系统管理组件等，后续可单独成文。
