# 进件中心与业务数据同步策略设计

> 本文档针对「进件数据何时同步到后台业务模块」的机构差异，给出统一设计思路与配置化方案，便于在分析完所有产品流程后统一回头评审。

---

## 一、问题与背景

### 1.1 核心差异

不同机构对「进件数据何时写入/同步到后台业务模块（客户、评级、授信、用信、额度、合同等）」的期望不一致：

| 模式 | 典型代表 | 同步时机 | 业务含义 |
|------|----------|----------|----------|
| **晚同步** | 湖北农信 | **合同签订完成后**才将关键数据同步到客户、授信、用信、额度、评级、合同等业务模块 | 进件在签约前仅作为「申请暂存 + 审批过程载体」；正式业务数据在签约生效时一次性落地 |
| **早同步** | 湖南农信 | **进件接收后**即初始化客户、评级、授信、用信等信息到后台业务模块；后续线下三合一、审批、签约是在这些已初始化的业务记录上更新 | 业务模块在进件阶段就能看到「草稿/初始」数据，便于贷中、贷后、营销、风控联动；签约/放款时做最终确认与更新 |

### 1.2 设计目标

- **同一套进件中心实现**，通过**配置化的同步策略**支持「晚同步」与「早同步」两种（及未来可能出现的其他）模式。
- 新产品、新机构接入时，以**配置为主、代码零改或极少改**，减少二次开发。
- 各业务组件（客户、评级、授信、用信、额度、合同等）具备**按进件号幂等 upsert** 能力，支持在不同业务阶段被多次调用（初始化 → 更新 → 终态确认）。

---

## 二、进件中心整体定位与生命周期

### 2.1 进件单作为「申请与流程载体」

- 进件中心维护**进件单（Intake）** 作为唯一权威的「申请视图」：
  - 原始申请数据（渠道、产品、客户标识、金额、期限等）
  - 流程状态、审批轨迹、决策结果（通过/转人工/拒绝）
  - 与各业务模块的同步状态（哪些模块已在何时落地/更新）

- 业务模块（客户、授信、用信、额度、合同等）是**经营与核算视图**，数据来源由「进件同步策略」决定：在哪些进件状态下、对哪些模块执行「创建/初始化/更新」。

### 2.2 进件生命周期（建议状态枚举）

以下状态为建议的最小集合，可根据实际产品扩展子状态或分支：

| 状态码 | 含义 | 说明 |
|--------|------|------|
| `RECEIVED` | 已接收 | 进件已落库，尚未进入审批/风控 |
| `UNDER_REVIEW` | 审批/调查中 | 风控准入中、或人工三合一/审批中 |
| `APPROVED` | 审批通过 | 授信/用信审批已通过，可生成合同与签约任务 |
| `SIGNED` | 合同签约完成 | 客户已完成合同签署，业务数据可视为「正式生效」 |
| `DISBURSED` | 已放款 | 资金已划付 |
| `REJECTED` | 已拒绝 | 任一环节拒绝，流程终止 |

状态迁移应由**工作流/审批流组件**驱动（如审批通过 → 进件状态改为 `APPROVED`），进件中心在状态变更时负责**发布领域事件**并**执行同步策略**。

## 三、同步策略：配置驱动、事件触发

### 3.1 策略配置表（概念模型）

将「哪个业务模块、在哪个进件状态下、执行何种同步动作」抽象为配置表，而不是写死在代码中。

| 配置项 | 说明 | 示例 |
|--------|------|------|
| 产品编号/产品类型 | 不同产品可配置不同策略 | 公积金贷、经营贷、抵押贷 |
| 业务模块 | 被同步的目标模块 | 客户、评级、授信、用信、额度、合同 |
| 触发时机 | 绑定进件状态或领域事件 | RECEIVED / APPROVED / SIGNED / DISBURSED |
| 同步动作 | 新建 / 初始化 / 更新 / 补全 | 初始化、更新、终态确认 |
| 幂等键 | 与业务模块约定，按进件号+业务主键 upsert | intakeId + customerId / creditId / contractId |
| 是否启用 | 支持按机构/产品关闭某条策略 | 是/否 |

**说明**：

- **新建**：仅当业务侧尚无该进件对应记录时创建。
- **初始化**：写入进件当前已有的数据，形成「草稿/初始」记录，后续可被更新。
- **更新**：在已有记录上补充或修正（如审批结论、线下三合一填写结果）。
- **终态确认**：将业务记录从草稿/临时状态置为正式生效（如合同生效、额度正式占用）。

同一「产品 + 业务模块」可在不同触发时机配置多条记录，例如：客户在 `RECEIVED` 时初始化，在 `SIGNED` 时更新并终态确认。

### 3.2 进件中心在状态迁移时的行为

1. **更新进件单状态**（可由工作流回调或进件中心内部流转）。
2. **发布领域事件**，例如：`IntakeReceived`、`IntakeApproved`、`IntakeSigned`、`IntakeDisbursed`。
3. **查询同步策略表**：根据当前产品、当前进件状态（或事件类型），得到「本状态下需触发的业务模块及同步动作」列表。
4. **按策略依次调用**各业务组件提供的「按进件号幂等 upsert」接口，传入进件号与当前进件快照/业务所需字段。
5. **记录同步结果**（成功/失败/跳过），用于对账与排查；失败时可支持重试与补偿。

## 四、各业务组件的接口契约（按进件同步）

为支持「早同步 + 多次更新」与「晚同步 + 一次性写入」两种模式，各业务组件应提供**按进件维度的幂等写入接口**。

### 4.1 通用约定

- 接口形态建议：`upsertXxxByIntake(intakeId, payload)` 或等价 REST/服务调用。
- **幂等性**：以 `intakeId` + 业务侧主键（或业务侧生成的与 intake 绑定的唯一键）为幂等键，重复调用不产生多条记录，仅更新。
- **数据来源**：payload 由进件中心根据当前进件单数据及流程上下文集成后传入；业务组件不直接读进件库，保持单向依赖（进件 → 业务）。

### 4.2 建议提供的接口（示例）

| 业务模块 | 建议接口（示例） | 说明 |
|----------|------------------|------|
| 客户 | upsertCustomerByIntake(intakeId, customerData) | 创建或更新客户；支持临时客户 → 正式客户 |
| 评级 | upsertRatingByIntake(intakeId, ratingData) | 写入/更新评级结果、评分、模型版本等 |
| 授信 | upsertCreditByIntake(intakeId, creditData) | 初始化授信申请或写入授信结论、有效期等 |
| 用信 | upsertDrawdownByIntake(intakeId, drawdownData) | 初始化用信申请或写入用信结论、金额、期限等 |
| 额度 | reserveOrUpdateQuotaByIntake(intakeId, quotaData) | 预占、正式占用或释放额度 |
| 合同 | createOrUpdateContractByIntake(intakeId, contractData) | 生成合同草稿或更新为已签署/生效 |

具体字段与校验规则由各组件在接口文档中定义；进件中心只负责在「正确的时机」携带「当前已有的数据」调用。

## 五、两种模式的配置示例

### 5.1 湖北农信模式（晚同步：合同签订完成后同步）

| 业务模块 | 触发时机 | 同步动作 |
|----------|----------|----------|
| 客户 | SIGNED | 新建/更新（正式客户） |
| 评级 | SIGNED | 新建/更新 |
| 授信 | SIGNED | 新建/更新 |
| 用信 | SIGNED | 新建/更新 |
| 额度 | SIGNED | 正式占用/更新 |
| 合同 | SIGNED | 终态确认（合同生效） |

效果：在 `RECEIVED`～`APPROVED` 阶段，各业务模块不写入或仅做最小化临时记录；签约完成（`SIGNED`）时，进件中心按策略一次性调用各模块 upsert，完成「申请 → 正式业务」的落地。

### 5.2 湖南农信模式（早同步 + 多次更新）

| 业务模块 | 触发时机 | 同步动作 |
|----------|----------|----------|
| 客户 | RECEIVED | 初始化（系统创建客户） |
| 客户 | APPROVED / SIGNED | 更新/补全（线下补充信息、正式确认） |
| 评级 | RECEIVED | 初始化（调用评级模型写入结果） |
| 评级 | APPROVED / SIGNED | 更新（线下三合一后的评级结果） |
| 授信 | RECEIVED | 初始化（授信申请信息） |
| 授信 | APPROVED | 更新（审批结论、授信额度、有效期） |
| 授信 | SIGNED | 终态确认 |
| 用信 | RECEIVED | 初始化（用信申请信息） |
| 用信 | APPROVED / SIGNED | 更新/终态确认 |
| 额度 | RECEIVED | 初始化（额度测算结果、预占可选） |
| 额度 | APPROVED / SIGNED | 正式占用/更新 |
| 合同 | APPROVED | 初始化（生成合同草稿/待签版本） |
| 合同 | SIGNED | 终态确认（合同生效） |

效果：进件接收后业务模块即可见「初始/草稿」数据；后续审批、签约、放款各阶段按策略对同一业务记录做更新与终态确认，避免重复建单，依赖各模块幂等 upsert。

## 六、与工作流/审批流组件的配合

- **工作流/审批流组件**负责：流程节点、审批人、会签/转签、超时、退回等；在关键节点完成时回调进件中心或发布「审批通过/拒绝」等事件。
- **进件中心**负责：
  - 维护进件单状态与领域事件；
  - 根据同步策略在相应状态/事件下调用业务组件；
  - 不把「何时同步」的逻辑散落在各个审批节点代码中，而是集中在进件中心 + 策略表。

这样，同一套工作流定义可复用于不同产品，差异仅体现在「该产品在进件中心配置的同步策略」上。

## 七、数据一致性与可观测性

- **一致性**：以进件单为权威来源，业务模块为派生视图；通过幂等 upsert 与重试机制，避免重复写入与脏数据。若需强一致，可考虑进件中心与关键业务模块在「同一状态迁移」内使用本地事务或 saga 编排。
- **可观测性**：建议记录「进件状态变更日志」与「同步执行日志」（进件号、目标模块、触发时机、成功/失败、重试次数），便于对账、排查与后续「所有产品分析完后」的回头评审。
