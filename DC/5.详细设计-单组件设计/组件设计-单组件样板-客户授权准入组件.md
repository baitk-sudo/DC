## 文档说明与样板用途

- **客户授权准入组件**是各类业务申请（预授信、授信申请、贷款申请、进件接收、签约前授权等）中**统一的授权展示与准入中枢**：  
  负责展示授权文件、组织授权签署、多因子认证，并在授权后触发授权准入规则执行，将结果返回给上游业务流程。
- 本文档采用与《进件管理组件 — 单组件详细设计》一致的**固定章节结构**：  
  **职责与边界 → 前置调用场景与条件 → 后续调用 → 关键接口与数据 → 多产品/多模式支持 → 与其他段落的协同说明**。  
  后续为其他渠道端组件做单组件设计时可直接复用该结构，仅替换内容。

## 一、职责与边界（简述）

| 项目 | 说明 |
|------|------|
| **职责** | 为客户提供**授权准入**的统一入口与交互流程：<br>在产品详情页或申请流程中，根据授权配置管理组件返回的结果，展示本阶段所需授权文件（征信、公积金、税务、政采数据等授权协议）；<br>引导客户阅读授权协议、完成短信/人脸等多因子认证并签署授权协议；<br>在授权签署完成后，携带客户/产品/授权信息调用风控组件执行**授权准入规则包**，并将通过/拒绝/转人工等结论回传给调用方（产品详情页、授信/贷款申请组件、进件管理组件等）。 |
| **边界** | 不决定“需要哪些授权文件/执行哪些规则包”，由**授权配置管理组件（办贷端）**负责配置；<br>不直接维护授权协议模板内容，仅按授权文件编码从**合同（协议）模版管理组件**加载；<br>不直接维护授权记录与合同实例的落库细节，授权签署与归档由**签约组件 + 合同管理组件**负责；<br>不自行编写规则，仅将授权配置返回的规则包 ID 传递给**风控组件**执行，并展示结果。 |

## 二、前置调用场景与调用条件

客户授权准入组件本身处于渠道端，但会被不同的上游入口以不同模式调用。以下按**调用方场景（A/B/C）**和**业务阶段**整理前置条件，帮助实现与测试确保不会误用授权能力。

### 2.1 前置调用者与业务阶段对应关系（A/B/C 总览小图）

> 这里只用“总览小图”的方式收束 A/B/C 三类入口，详细步骤见后续各小节。

| 场景 | 前置调用者 | 典型入口位置 | 业务阶段（示例） | 授权准入组件在本场景中的角色 |
|------|------------|--------------|------------------|------------------------------|
| **场景 A：独立授权阶段** | 渠道前端（产品详情页 / 专门“授权准入”入口） | 税E贷、政采贷等产品详情页上的「先授权再申请」按钮 | `AUTHORIZATION`（独立授权准入阶段） | 承载完整的独立授权流程：展示多种授权文件、组织授权签署、执行授权准入规则；将结果返回产品详情页或上游流程，通常作为是否允许进入授信申请/贷款申请的前置判断。 |
| **场景 B：申请内授权** | 授信申请组件 / 贷款申请组件（渠道端） | 公积金贷款、部分消费贷的**申请页面内部**嵌入的授权区域 | `CREDIT_APPLY` / `LOAN_APPLY`（申请阶段内授权） | 由申请页面调用授权准入组件的能力（通常以弹窗/子流程形式）完成本阶段所需授权文件的展示、签署与准入规则执行；结果作为是否允许提交申请、是否需要补授权的依据。 |
| **场景 C：进件/签约前补授权** | 进件管理组件 / 签约组件（办贷端触发，渠道端执行） | 在进件接收或签约前发现授权不足时，由办贷端或 PAD 发起的**补授权**流程 | `INTAKE_ACCESS` / `SIGNING` / `APPLICATION_PRE_SUBMIT` 等 | 在进件/签约前由办贷端触发授权准入组件（面客渠道或 PAD），完成缺失授权的补录与准入规则校验，结果回传进件管理或签约组件以决定是否继续后续流程。 |

> **与进件管理组件的链路关系提示**：  
> - 场景 A/B：授权准入组件的结果将被授信申请/贷款申请组件记录，并通过 `authRecordId`/授权结果字段传给进件管理组件，用于进件接收阶段的前置条件校验。  
> - 场景 C：进件管理组件在进件接收或状态迁移前，若发现授权不足，可触发补授权流程，调用本组件完成授权签署与准入，再继续后续状态迁移。

### 2.2 场景 A：独立授权阶段（产品详情页 → 授权准入组件）

| 序号 | 前置条件 | 说明 |
|------|----------|------|
| A.1 | 已选定产品与客户 | 产品详情页上已明确当前产品（`productId`）与客户登录态（至少有 `customerId` 或可解析为客户的身份信息）；对公客户需已完成对公注册与基础认证（见第 2 段客户注册与认证）。 |
| A.2 | 授权配置管理已配置对应阶段 | 授权配置管理组件中，按「产品 + 客户类型 + 业务阶段 = `AUTHORIZATION`」已配置需要签署的授权文件列表及需执行的授权准入规则包 ID 列表。 |
| A.3 | 渠道可识别业务阶段与入口 | 授权准入组件需从调用参数中识别业务阶段为 `AUTHORIZATION`，并可选识别渠道与场景（APP/H5、产品详情页、营销落地页等）以便做埋点与差异化展示。 |
| A.4 | 安全能力/签约能力可用 | 短信、人脸识别、安全组件已按产品/机构配置好认证策略；签约组件/合同管理组件中已存在授权协议模板的签署任务配置。 |

**本场景下授权准入组件的入口能力**：  
接收“独立授权阶段”的发起请求，查询授权配置管理组件获取本产品在 `AUTHORIZATION` 阶段需要的授权文件与规则包，组织授权协议展示与签署，并在签署完成后调用风控组件执行授权准入规则，将结果返回上游产品详情页或后续流程。

---

### 2.3 场景 B：申请内授权（授信/贷款申请页内嵌授权）

| 序号 | 前置条件 | 说明 |
|------|----------|------|
| B.1 | 已进入授信/贷款申请流程 | 授信申请组件或贷款申请组件已初始化申请页面，具备产品标识、客户标识及基本申请上下文。 |
| B.2 | 授权配置管理中的阶段配置就绪 | 授权配置管理组件中，按「产品 + 客户类型 + 业务阶段 = `CREDIT_APPLY` / `LOAN_APPLY`」已配置本申请阶段需要的授权文件与规则包（如“申请内授权”）。 |
| B.3 | 调用方可区分“新授权”与“补授权” | 对于已存在历史授权记录的客户，调用方应能在调用授权准入组件前查询（或由授权准入组件查询）既有授权记录，用于判断是否需要重新授权或仅做提示。 |
| B.4 | 授权检查模式已配置 | 授权配置管理中针对本产品/阶段已配置授权检查模式（强制/仅提示/仅记录），授权准入组件需在执行后根据模式向申请组件返回“是否允许提交申请”的建议。 |

**本场景下授权准入组件的入口能力**：  
由授信/贷款申请组件在提交前（或页面初始化时）调用，用于展示申请阶段所需授权文件并完成签署与准入校验；返回的授权结果（含 `authRecordId`、授权结论）由申请组件透传给进件管理组件，用作进件接收时的前置条件校验依据。

---

### 2.4 场景 C：进件/签约前补授权（办贷端触发，渠道端执行）

| 序号 | 前置条件 | 说明 |
|------|----------|------|
| C.1 | 已存在进件或签约上下文 | 办贷端（进件管理组件/签约组件）已能定位到目标进件/签约任务，具备进件 ID、产品、客户标识以及需要补授权的数据源范围。 |
| C.2 | 授权配置管理中存在进件/签约阶段配置 | 授权配置管理组件中，按「产品 + 客户类型 + 业务阶段 = `INTAKE_ACCESS`/`SIGNING` 等」已配置需要的授权文件与规则包，用于进件准入/签约前授权校验。 |
| C.3 | 面客渠道或办贷 PAD 可用 | 办贷端可通过短信、二维码、PAD 等方式将补授权任务发给客户，客户在面客渠道或 PAD 上完成授权流程。 |
| C.4 | 回调与结果落库路径清晰 | 授权准入组件完成补授权与准入规则执行后，需将新的授权记录及准入结果回传给进件管理组件或签约组件（办贷端），以驱动进件状态机或签约流程继续执行。 |

**本场景下授权准入组件的入口能力**：  
接受来自办贷端的补授权触发请求，生成授权任务（通常以内嵌签约任务为载体），驱动客户在面客渠道完成补授权签署与准入校验，将结果同步给办贷端业务组件。

### 2.5 前置条件汇总表（跨场景检查清单）

| 类别 | 序号 | 条件 | 适用场景 |
|------|------|------|----------|
| 数据完整性 | 1 | 调用授权准入组件前必须明确产品、客户、业务阶段（如 `AUTHORIZATION`/`CREDIT_APPLY`/`LOAN_APPLY`/`INTAKE_ACCESS` 等） | A、B、C |
| 配置就绪 | 2 | 授权配置管理组件中存在对应的「产品 + 客户类型 + 业务阶段」配置，返回授权文件列表与规则包 ID 列表 | A、B、C |
| 配置就绪 | 3 | 合同（协议）模版管理组件中已配置授权文件编码对应的协议模板；签约组件已支持“授权协议”任务类型 | A、B、C |
| 配置就绪 | 4 | 短信、人脸识别、安全组件等通用安全能力已按产品/机构配置好策略，可供本组件调用 | A、B、C |

## 三、客户授权准入组件的后续调用

客户授权准入组件本身是**渠道前端交互组件**，但在一次完整授权流程中会调用多个办贷端能力组件。以下从“授权流程阶段”角度整理后续调用关系。

### 3.1 按授权流程阶段的后续调用矩阵

> 授权准入组件在整个流程中不落库具体授权内容与规则执行结果，仅作为“编排与展示层”，将必要的上下文传递给下游组件。

| 授权阶段/步骤 | 授权准入组件主动调用的下游组件 | 端别 | 说明 |
|---------------|--------------------------------|------|------|
| **初始化：确定本次授权需求** | 授权配置管理组件 | 办贷端 | 按「产品 + 客户类型 + 业务阶段」查询需签署的授权文件列表及需执行的规则包 ID 列表，决定需要展示哪些协议以及后续要执行哪些规则。 |
| **加载授权协议内容** | 合同（协议）模版管理组件 | 办贷端 | 按授权文件编码加载授权协议模板供前端展示，支持多协议组合展示。 |
| **多因子认证（短信/人脸/密码等）** | 消息中心组件、人脸识别组件、安全组件 | 办贷端 | 根据系统管理/安全策略执行短信验证码、人脸识别、交易密码等验证，验证结果仅作为是否允许进入签署环节的前置。 |
| **授权协议签署** | 签约组件 → 合同管理组件/授权记录表 | 办贷端 | 为本次授权生成“授权协议”签约任务，客户在渠道端或 PAD 完成签署；签约结果由签约组件回写合同管理/授权记录表。 |
| **执行授权准入规则** | 风控组件 → 征信与外部数据管理组件（如需） | 办贷端 | 在确认授权签署完成、外部数据查询授权合法后，按授权配置返回的规则包 ID 列表调用风控组件执行授权准入规则；风控组件可进一步调用征信与外部数据管理组件获取必要外部数据。 |
| **结果回传给业务流程** | 上游业务组件（授信申请、贷款申请、进件管理、产品详情页等） | 渠道端/办贷端 | 将授权结果（通过/拒绝/转人工）与授权记录标识（如 `authRecordId`）返回给上游调用方，用于后续是否允许提交申请/创建进件/继续签约等决策。 |

### 3.2 对下游组件的接口与数据要求

- **授权配置管理组件（办贷端）**：  
  提供按「产品 + 客户类型 + 业务阶段」查询授权配置的接口，返回：
  - 需签署的授权文件列表（含授权文件编码、名称、是否必签、适用对象/关联人类型集合）；
  - 需执行的风控规则包 ID 列表（`rule_package_ids`）；
  - 授权检查模式（强制/仅提示/仅记录）等配置字段。

- **合同（协议）模版管理组件（办贷端）**：  
  提供按授权文件编码/版本查询授权协议模板内容的接口，支持渠道端渲染授权文本。

- **签约组件 & 合同管理组件（办贷端）**：  
  - 支持“授权协议”这一类签约任务类型，与授信合同、借款合同区分；  
  - 支持从渠道端发起授权协议签署流程（包括短信/人脸等验证），并将签署结果回写到合同管理组件或专门的授权记录表；  
  - 为授权准入组件提供“根据授权任务 ID/授权记录 ID 查询签署结果”的接口。

- **风控组件 & 征信与外部数据管理组件（办贷端）**：  
  - 风控组件需支持“授权准入”这一规则执行场景，入参包含产品、客户、业务阶段、授权记录标识与规则包 ID 列表；  
  - 征信与外部数据管理组件需在存在有效授权的前提下，按授权配置支持拉取征信、公积金、税务、政采等外部数据。

## 四、关键接口与数据（授权准入组件提供）

本节关注**授权准入组件本身对上游提供的接口/能力**，以 REST/接口视角描述。内部实现可采用前后端分离 + BFF 等模式，本节不限定技术栈。

| 接口/能力 | 说明 | 上游调用方（均标注端别） |
|-----------|------|--------------------------|
| 初始化授权准入（`initAuthorization`） | 根据产品、客户、业务阶段初始化本次授权流程，返回需展示的授权文件列表及初始上下文 | 渠道前端页面（产品详情页/申请页）、进件管理组件（办贷端，通过中台/BFF 间接调用） |
| 提交授权签署结果（`submitAuthSigningResult`） | 渠道端在完成授权协议签署后，携带签署结果及授权记录标识回调授权准入组件 | 签约组件前端/渠道容器 |
| 执行授权准入规则（`executeAuthorizationAccess`） | 在确认授权协议签署完成后，调用风控组件执行授权准入规则，并返回结论给调用方 | 渠道前端页面、进件管理组件、授信/贷款申请组件 |
| 查询授权结果（`queryAuthorizationResult`） | 按客户/产品/业务阶段查询最近一次授权结果及是否仍在有效期内，支持业务流程判断是否需要补授权 | 授信申请组件、贷款申请组件、进件管理组件、签约组件 |

### 4.1 初始化授权准入 — 请求与返回字段定义

#### 请求字段

| 字段块 | 字段名 | 字段中文名 | 类型 | 是否必填 | 说明 |
|--------|--------|------------|------|----------|------|
| **请求头 Header** | `requestId` | 渠道请求唯一标识 | string | 是 | 渠道侧生成的唯一请求号，用于幂等控制（同一 requestId + 场景视为同一初始化请求）。 |
|  | `productId` | 产品编号 | string | 是 | 对应产品管理中的产品编码。 |
|  | `customerId` | 客户号 | string | 否 | 若已有客户号则优先传入；否则需在 `customerSnapshot` 中提供足够的识别信息。 |
|  | `customerSnapshot` | 客户快照 | object | 否 | 证件类型/证件号/姓名/手机号等最小识别信息，用于在无客户号时定位客户或创建临时客户；`customerId` 与 `customerSnapshot` 至少一项必填。 |
|  | `customerType` | 客户类型 | enum | 是 | 数据字典（英文:中文）：PERSONAL:个人、CORPORATE:对公。对公授权须满足对公注册与基础认证已完成。 |
|  | `businessStage` | 业务阶段 | enum | 是 | 统一语言中定义的阶段枚举，如：AUTHORIZATION、CREDIT_APPLY、LOAN_APPLY、INTAKE_ACCESS、SIGNING、APPLICATION_PRE_SUBMIT 等。 |
|  | `channel` | 渠道标识 | string | 是 | APP/H5/WX_MP/PC_WEB 等，用于埋点与差异化交互。 |
|  | `channelScene` | 渠道场景 | string | 否 | 如 PRODUCT_DETAIL、LOAN_APPLY_PAGE、AUTH_STAGE 等，用于精细化配置。 |

#### 返回字段

| 字段名 | 字段中文名 | 类型 | 说明 |
|--------|------------|------|------|
| `authorizationContextId` | 本次授权上下文编号 | string | 标识一次授权流程上下文，用于后续签署与规则执行的关联。 |
| `requiredAuthFiles` | 需签署的授权文件列表 | array | 每项包含授权文件编码、名称、是否必签、适用对象/关联人类型集合等信息，来自授权配置管理组件。 |
| `rulePackageIds` | 授权准入规则包 ID 列表 | array\<string> | 需执行的规则包 ID，由授权配置管理组件提供，供后续调用风控组件使用。 |

### 4.2 执行授权准入规则 — 内部总体设计（流程级伪代码）

授权准入组件本身不执行规则，而是统一组织对授权配置管理、签约、风控的调用。以下为“授权准入整体流程”的简化伪代码，用于约束对外接口行为和下游调用顺序。

```text
1. 初始化授权上下文（initAuthorization）
   - 按产品 + 客户类型 + 业务阶段调用授权配置管理组件，获取授权文件列表 + 规则包 ID 列表 + 检查模式。
   - 将结果返回前端，展示授权协议列表。

2. 触发授权协议签署
   - 为本次授权生成“授权协议”签约任务（可由签约组件/合同管理组件承载）。
   - 前端组织短信/人脸等多因子认证，调用签约组件完成授权协议签署。

3. 确认授权签署完成
   - 由签约组件回调或查询合同管理/授权记录表，确认本次授权需要的协议均已签署成功。
   - 生成或获取 `authRecordId`（授权记录编号），作为后续风控与进件/申请的引用号。

4. 调用风控组件执行授权准入规则
   - 将 authRecordId + rulePackageIds + 客户/产品/业务阶段等上下文传入风控组件。
   - 风控组件按规则包执行授权准入规则，必要时通过征信与外部数据管理组件拉取外部数据。
   - 获得授权准入结果（通过 / 拒绝 / 转人工 / 补充资料等）。

5. 返回授权准入结果给业务流程
   - 授权准入组件根据授权检查模式（ENFORCE/REMIND/NONE）和风控结果，向上游业务组件返回统一结果结构：
     - 是否授权签署齐备；
     - 授权准入风控结论；
     - 是否允许继续后续步骤（如提交申请/创建进件/继续签约）；
     - authRecordId 供后续进件管理、风控、报表等引用。
```

## 五、多产品 / 多模式支持点（对现有 5 个产品的支撑）

参考《组件设计-第3段-授权准入与规则触发.md》中的多产品分析，这里只突出与端到端示例相关的支撑点。

| 产品 | 场景类型 | 授权模式 | 授权准入组件支撑要点 |
|------|----------|----------|----------------------|
| 湖北农信-公积金贷款（线上） | 贷款申请（B） | 申请内授权 | 贷款申请页内嵌授权区域，由申请组件调用授权准入能力：按 `LOAN_APPLY` 阶段查询授权配置，展示征信、公积金授权协议，完成签署与准入校验；授权结果通过 `authRecordId` 传给进件管理组件，用于进件接收阶段校验授权是否有效。 |

| 湖南农信-公积金贷款 | 贷款申请（B） | 申请内授权 + 可能存在预授信结果参考 | 授权准入组件在 `LOAN_APPLY` 阶段执行授权，与是否有预授信结果无直接耦合；若产品要求先预授信再授权，可通过授权配置中的阶段定义与规则包配置体现。 |
| 长沙银行-税E贷 | 授信申请前独立授权（A） | 独立授权阶段 + 申请前强制授权 | 在 `AUTHORIZATION` 阶段提供独立的“客户授权准入”入口：展示多种授权协议（征信、税务、政采等），完成签署与准入规则执行；授权结果成为是否允许进入授信申请（场景 A → 进件管理）的前置条件。 |
| 长沙银行-政采贷 | 授信申请前独立授权（A） | 独立授权阶段 + 申请前强制授权 | 与税E贷类似，授权准入组件在独立授权阶段提供统一能力；仅授权文件种类、规则包配置有所差异，由授权配置管理组件负责。 |

## 六、与其他段落/组件的协同关系小结

- **与第 1 段（营销获客与预授信）**：  
  预授信阶段若需要征信/公积金/政采授权，可通过授权配置管理在 `CUSTOMER_PRE_CREDIT` / `PRODUCT_PRE_CREDIT` 阶段配置授权文件与规则包，由预授信组件调用本组件完成授权与准入校验。

- **与第 2 段（客户注册与认证）**：  
  授权准入中用到的短信、人脸、密码等认证手段，均为第 2 段中定义的通用安全能力与系统管理策略的复用；本组件不自建安全能力，仅消费这些组件。

- **与第 3 段（授权配置管理、风控组件、合同模版管理、签约组件等）**：  
  本文档可视为对第 3 段中“客户授权准入组件”部分的**单组件放大版**，在职责边界、前置条件、调用链、接口字段层面补充更多细节，但不改变第 3 段中已有的设计结论。

- **与第 4 段（进件接收与风控准入）及《进件管理组件 — 单组件详细设计》**：  
  授权准入组件产生的授权记录与准入结果，是进件接收阶段 `A.4/B.4/C.4` 条中“先授权”前置条件的主要支撑；进件管理组件通过 `authRecordId` 及授权配置管理组件的配置校验授权是否有效，并在必要时触发补授权（调用本组件的场景 C）。

- **与签约组件/合同管理组件**：  
  授权准入阶段中“授权协议”属于签约组件的一个任务类型；本组件负责触发与编排，签约组件负责签署流程与结果回写，合同管理组件负责模板管理与合同/授权记录落库。
