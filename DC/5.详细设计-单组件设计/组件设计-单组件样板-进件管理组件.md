## 文档说明与样板用途

- **进件管理组件**是授信/贷款/提额等申请的**统一接收与流程中枢**，上游来自渠道申请组件、工作流、签约、放款等多类调用方，下游依赖风控、授权配置、工作流及客户/评级/授信/用信/额度/合同等业务组件，前置条件与后续调用均需在单组件设计中写清。
- 本文档采用**固定章节结构**（职责与边界 → 前置调用场景与条件 → 后续调用 → 关键接口与数据 → 多产品支持 → 样板说明），后续为其他组件做单组件设计时可直接复用该结构，仅替换内容。

---

## 一、职责与边界（简述）

| 项目 | 说明 |
|------|------|
| **职责** | 统一接收**授信进件、贷款/用信进件、提额进件**；<br>维护进件单生命周期与状态机（RECEIVED → UNDER_REVIEW → APPROVED → SIGNED → DISBURSED / REJECTED）；<br>在状态迁移时按**同步策略配置表**触发对各业务模块的同步；<br>根据风控准入结果与产品/进件类型/渠道配置驱动路由（R1 直通 / R3 转线下）；<br>R3 转线下时按配置确定**办贷任务类型**（三合一/二合一/单独）并传给工作流；<br>执行进件自动分配，与工作流协同生成待办任务。 |
| **边界** | 不执行业务规则或额度测算（由风控、评级、额度等组件负责）；<br>不定义审批流程与审批人（由工作流/审批流负责）；<br>不直接生成合同内容或签约任务内容（由合同管理、签约组件提供）；<br>「何时、对谁、做什么」同步由同步策略配置决定，进件管理只负责按配置调用各业务组件的 upsertByIntake 类接口。 |

## 二、前置调用场景与调用条件

进件管理组件需在**多种调用场景**下被正确调用，且每次调用前必须满足与**调用方及进件类型**相匹配的前置条件。以下按**调用者**细分场景与条件，并说明对**配置组件**的依赖，确保满足复杂调用场景要求。

### 2.1 前置调用者与进件类型对应关系

| 前置调用者 | 端别 | 进件类型（建议枚举） | 典型业务含义 |
|------------|------|----------------------|--------------|
| 授信申请组件 | 渠道端 | CREDIT_APPLY | 客户发起授信申请，生成授信进件（如税E贷、政采贷） |
| 贷款申请组件 | 渠道端 | LOAN_APPLY | 客户发起贷款/用信申请，生成贷款进件（如公积金贷、政采贷额度项下贷款） |
| 提额组件 | 渠道端/办贷端入口 | LIMIT_INCREASE | 客户或办贷端发起授信提额/贷款提额，生成提额进件 |
| 工作流/审批流组件 | 办贷端 | — | 不创建新进件，对已有进件做状态回调、办贷任务类型传入 |
| 签约组件 | 渠道端、办贷端 PAD | — | 不创建新进件，签约完成驱动进件状态迁至 SIGNED 并触发同步策略 |
| 放款组件 | 办贷端 | — | 不创建新进件，放款完成驱动进件状态迁至 DISBURSED |

进件类型定义见《统一语言-领域词汇表》；与总览中「授信申请、贷款申请、提额」三组件与进件类型的对应关系一致。

### 2.2 按调用者的前置调用场景与条件

以下各场景中，**列出的条件均为在该场景下调用进件管理前须至少满足的条件**（与关系）；若产品在授权配置中要求「先授权」，须在进件创建或进入关键状态前通过授权配置管理组件 + 授权记录完成校验。

---

#### 场景 A：授信申请组件（渠道端）→ 进件管理（办贷端）（生成授信进件）

| 序号 | 前置条件 | 说明 |
|------|----------|------|
| A.1 | 传入数据完整 | 至少包含：产品标识；<br>进件类型 = `CREDIT_APPLY`；<br>客户标识（或可解析为客户的足够信息）；<br>授信申请所需基本业务字段（金额、期限、用途等按产品配置）。 |
| A.2 | 产品与进件配置就绪 | 产品管理组件（办贷端）已配置授信申请要素与校验规则；<br>同步策略表中对该产品 + 进件类型 + 各状态有明确配置（或显式不触发）；<br>进件状态机定义完整。 |
| A.3 | 客户可识别 | 调用时须能唯一识别客户：已有 `customerId`，或本次申请过程中采集到的身份信息可由客户管理组件（办贷端）创建/补录临时客户；<br>对公客户须已完成对公注册与基础认证（用户注册组件（渠道端） + 客户管理组件（办贷端））。 |
| A.4 | 先预授信/先授权（若产品要求） | 若产品要求先预授信：须已通过预授信组件获得有效预授信结果（可在授信申请页展示与复用）；<br>若产品要求先授权：须已通过客户授权准入组件（渠道端）或授权配置管理组件（办贷端）校验，且授权结果可被进件或风控在进件接收阶段引用（同一产品 + 客户 + 阶段下已存在有效未到期授权则无需重复授权）；<br>若未配置上述前置：按「直接授信」模式提交授信进件。 |

**进件管理在本场景的入口能力**：接收授信申请数据，落库进件单，进件类型=CREDIT_APPLY，初始状态 RECEIVED，并触发 RECEIVED 对应的同步策略（若为早同步产品）。

---

#### 场景 B：贷款申请组件（渠道端）→ 进件管理（办贷端）（生成贷款/用信进件）

| 序号 | 前置条件 | 说明 |
|------|----------|------|
| B.1 | 传入数据完整 | 至少包含：产品标识；<br>进件类型 = `LOAN_APPLY`；<br>客户标识；<br>贷款/用信申请所需基本业务字段（金额、期限、用途、目标授信/额度项如适用）。 |
| B.2 | 产品与进件配置就绪 | 产品管理组件（办贷端）已配置贷款申请要素与规则；<br>同步策略表中对该产品 + `LOAN_APPLY` + 各状态有配置；<br>进件状态机定义完整。 |
| B.3 | 客户可识别 | 同 A.3；<br>对公须已完成对公注册与基础认证。 |
| B.4 | 先授权后申请（若产品要求） | 若产品要求先授权后申请：须已通过客户授权准入组件（渠道端）或授权配置管理组件（办贷端）校验；<br>授权结果可被进件或风控在进件接收阶段引用（同一产品 + 客户 + 阶段下已存在有效未到期授权则无需重复授权）；<br>是否存在预授信阶段及如何使用预授信由产品的业务流程建模决定。 |

**申请前规则包（可选）**：若产品在《产品流程-组件调用场景视图与配置表》**2.4 风控与路由配置表**中启用了「申请前（可选）」规则包（反欺诈、黑名单、产品参数准入等），由**贷款申请组件（渠道端）**在**调用 createIntake 之前**按产品配置调用**风控组件**执行；未通过则**不得提交进件**。进件管理**不校验**申请前规则包是否已执行，仅可选接收请求中的 `preCheckResultRef`（申请前规则结果引用号）用于审计与追溯。同一产品的线上与线上转线下模式在阶段 1～3 一致，申请前规则包配置可复用（见该配置表中 HBGJJ-ONLINE / HBGJJ-O2O 等）。

**进件管理在本场景的入口能力**：接收贷款申请数据，落库进件单，进件类型=LOAN_APPLY，初始状态 RECEIVED，并触发 RECEIVED 对应同步策略（若为早同步）。

---

#### 场景 C：提额组件（渠道端/办贷端入口）→ 进件管理（办贷端）（生成提额进件）

| 序号 | 前置条件 | 说明 |
|------|----------|------|
| C.1 | 传入数据完整 | 至少包含：产品标识；<br>进件类型 = `LIMIT_INCREASE`；<br>客户标识；<br>提额对象标识（授信单/额度项/用信单等）；<br>提额申请所需基本业务字段。 |
| C.2 | 产品与进件配置就绪 | 产品管理组件（办贷端）/额度管理组件（办贷端）已开放提额入口并配置提额规则；<br>同步策略表中对该产品 + `LIMIT_INCREASE` + 各状态有配置；<br>进件状态机定义完整。 |
| C.3 | 提额对象可唯一定位 | 授信提额需存在授信额度记录；<br>贷款提额需存在目标贷款/用信记录；<br>对公提额对象须已完成对公客户注册与基础认证。 |
| C.4 | 先授权或贷后检查（若产品要求） | 若需先授权：通过客户授权准入组件（渠道端）完成授权校验（同一产品 + 客户 + 阶段下已存在有效未到期授权则无需重复授权）；<br>若需先完成特定贷后检查：通过贷后管理/预警组件（办贷端）验证后再提交提额进件。 |

**进件管理在本场景的入口能力**：接收提额申请数据，落库进件单，进件类型=LIMIT_INCREASE，初始状态 RECEIVED，并触发 RECEIVED 对应同步策略（若配置）。

---

#### 场景 D：工作流/审批流组件（办贷端）→ 进件管理（办贷端）（状态回调与办贷任务驱动）

| 序号 | 前置条件 | 说明 |
|------|----------|------|
| D.1 | 进件已存在且可识别 | 进件 ID 或业务主键明确；<br>进件单处于可被工作流操作的合法状态（如 `UNDER_REVIEW`）。 |
| D.2 | 流程模板与办贷任务类型已定义 | 工作流中已配置进件路由、办贷流程模板（三合一/二合一/单独）；<br>R3 转线下时：办贷任务类型由进件管理根据产品 + 进件类型 + 渠道等配置解析并传入工作流；<br>工作流仅负责按该任务类型启动对应模板。 |
| D.3 | 回调上下文完整 | 回调需携带：进件 ID；<br>新状态（如 `APPROVED`）；<br>必要业务结论摘要（如需写回进件或驱动同步）。 |

**进件管理在本场景的职责**：接收工作流节点完成回调，更新进件状态（如 APPROVED）；根据新状态查询同步策略表并执行对客户/评级/授信/用信/额度/合同等的 upsertByIntake；在 R3 分支下向工作流传入办贷任务类型（三合一/二合一/单独），由工作流启动对应办贷流程模板。

---

#### 场景 E：签约组件（渠道端/办贷端 PAD）→ 进件管理（办贷端）（签约完成驱动 SIGNED）

| 序号 | 前置条件 | 说明 |
|------|----------|------|
| E.1 | 进件已存在且与签约任务关联 | 签约任务或合同实例与进件 ID 可关联；<br>进件当前状态允许迁至 `SIGNED`（通常为 `APPROVED` 或等效）。 |
| E.2 | 签约结果已落库 | 合同管理组件（办贷端）/签约组件（渠道端/办贷端 PAD）已完成签署结果回写；<br>进件管理可依据签约完成事件或合同状态驱动状态迁移。 |
| E.3 | 同步策略表就绪 | 对该产品 + 进件类型：在 `SIGNED` 状态下有明确的同步目标与动作配置；<br>例如晚同步产品在 `SIGNED` 一次性写客户/授信/用信/额度/合同等。 |

**进件管理在本场景的职责**：将进件状态更新为 SIGNED；发布领域事件（如 IntakeSigned）；按同步策略表执行 SIGNED 下的各模块 upsertByIntake（晚同步为一次性写入，早同步为更新/终态确认）。

---

#### 场景 F：放款组件（办贷端）→ 进件管理（办贷端）（放款完成更新 DISBURSED）

| 序号 | 前置条件 | 说明 |
|------|----------|------|
| F.1 | 进件已存在且与用信/放款关联 | 放款对应的用信记录与进件可关联；<br>进件当前状态允许迁至 `DISBURSED`。 |
| F.2 | 放款交易已成功 | 放款组件（办贷端）已完成核心/账务放款并回写用信状态；<br>进件管理仅负责状态更新与可选同步。 |
| F.3 | 同步策略表就绪 | 对该产品 + 进件类型：在 `DISBURSED` 状态下若配置了同步动作（如终态确认、报表驱动），则配置完整可执行。 |

**进件管理在本场景的职责**：将进件状态更新为 DISBURSED；按同步策略执行 DISBURSED 下配置的同步动作（若有）。

### 2.3 与配置组件的关系（满足复杂场景的配置依赖）

进件管理不实现业务规则与流程定义，而是依赖以下**配置组件**与配置表，才能在不同产品、不同调用方下正确行为：

| 配置来源 | 端别 | 作用 | 进件管理如何使用 |
|----------|------|------|-------------------|
| **产品管理组件** | 办贷端 | 产品、进件类型、申请要素、枚举与校验规则 | 校验传入产品与进件类型合法；<br>配合同步策略与路由配置使用。 |
| **同步策略表**（与《进件中心-业务数据同步策略设计》一致） | 办贷端配置 | 按产品 + 进件类型 + 进件状态决定对哪些业务模块、在何时、执行何种同步动作 | 在 `RECEIVED` / `APPROVED` / `SIGNED` / `DISBURSED` 等状态迁移时查询并执行 `upsertByIntake`。 |
| **授权配置管理组件** | 办贷端 | 产品 + 客户类型 + 业务阶段 → 授权文件与规则包；<br>**进件阶段规则包 ID** | 进件接收后按「产品 + 进件类型/阶段」查询进件准入规则包 ID，再调用风控组件（办贷端）执行进件风控准入；<br>不决定「签什么协议」，仅消费规则包 ID。 |
| **工作流/审批流组件** | 办贷端 | 进件后流程模板、办贷任务类型（三合一/二合一/单独）、分配规则 | 进件管理根据产品 + 进件类型 + 渠道解析办贷任务类型并传入工作流；<br>工作流按模板执行节点并回调进件管理更新状态。 |
| **进件状态机定义** | 办贷端配置 | 允许的状态与迁移 | 进件管理只执行已定义的状态迁移，避免非法状态。 |

上述依赖在**任意前置调用场景**下均需就绪：即无论是授信申请、贷款申请、提额、工作流回调、签约完成还是放款完成，进件管理在执行动作前都会依赖「产品+进件类型」下的同步策略、状态机及（进件接收场景下）授权配置中的进件规则包 ID。

### 2.4 前置条件汇总表（跨场景检查清单）

以下为进件管理组件**被调用前**须满足的**通用与场景相关**条件汇总，便于实现与测试时逐项保证复杂调用场景可满足。

| 类别 | 序号 | 条件 | 适用场景 |
|------|------|------|----------|
| 数据完整性 | 1 | 传入的申请/进件数据完整：至少包含产品、进件类型、客户标识、基本业务字段 | A、B、C（创建进件）；D、E、F 为进件 ID + 事件/状态 |
| 配置就绪 | 2 | 对应产品在同步策略表中有配置（含各触发状态与目标模块） | 全部 |
| 配置就绪 | 3 | 进件状态机定义完整，且与同步策略、工作流模板一致 | 全部 |
| 配置就绪 | 4 | 进件接收场景下：**授权配置管理组件（办贷端）**中可查询到「产品+进件类型/阶段」对应的进件准入规则包 ID | A、B、C（进件接收后风控准入） |
| 先决业务条件 | 5 | 如产品在授权配置中要求「先授权」，在进件创建或进入关键状态前可通过**客户授权准入组件（渠道端）**/**授权配置管理组件（办贷端）**及授权记录完成校验 | A、B、C |
| 先决业务条件 | 6 | 客户可唯一识别；对公场景下已完成对公注册与基础认证 | A、B、C |
| 先决业务条件 | 7 | 提额场景：提额对象（授信/额度/用信）可唯一定位 | C |

## 三、进件管理组件的后续调用

进件管理在**不同状态与分支**下会调用不同下游组件；下游不直接反向调用进件管理（除工作流回调、签约/放款完成回写状态外）。以下按**进件生命周期与分支**整理后续调用关系。

### 3.1 按状态/场景的后续调用矩阵

> **约定**：在前置调用条件满足的前提下，进件管理组件**必须先持久化进件记录**（通常初始状态为 RECEIVED），之后再按配置触发风控与后续同步。风控返回拒绝/不通过时，仅更新进件状态及后续路由，不抹除已存在的进件记录，便于审计与追溯。

| 进件阶段/分支 | 进件管理主动调用的下游组件 | 端别 | 说明 |
|---------------|----------------------------|------|------|
| **进件接收（RECEIVED）** | 授权配置管理组件 | 办贷端 | 按产品+进件类型/阶段查询**进件准入规则包 ID**，供风控执行进件准入 |
| 进件接收（RECEIVED） | 客户管理组件 | 办贷端 | 早同步时 upsertCustomerByIntake（初始化/补录）；晚同步时仅补录临时客户（若配置） |
| 进件接收（RECEIVED） | 风控组件 | 办贷端 | 使用授权配置返回的规则包 ID 执行**进件风控准入**，获取 R1/R3/拒绝等结果 |
| 进件接收（RECEIVED） | 客户/评级/授信/用信/额度管理组件 | 办贷端 | **早同步**时按同步策略执行 RECEIVED 下的 upsertByIntake（初始化） |
| **路由：R1 直通** | 工作流/审批流组件、签约组件、合同管理组件 | 办贷端；签约为渠道端+办贷端 PAD | 生成签约任务、进件分配至签约池；合同与签约内容由合同管理、签约组件提供 |
| **路由：R3 转线下** | 工作流/审批流组件 | 办贷端 | 传入**办贷任务类型**（三合一/二合一/单独），由工作流启动对应办贷流程模板；进件自动分配 |
| **状态 APPROVED** | 客户/评级/授信/用信/额度/合同管理组件 | 办贷端 | 按同步策略执行 APPROVED 下的 upsertByIntake（更新或生成合同草稿/待签） |
| **状态 SIGNED** | 客户/评级/授信/用信/额度/合同管理组件 | 办贷端 | 按同步策略执行 SIGNED 下的 upsertByIntake（晚同步为一次性写入，早同步为更新/终态确认） |
| **状态 DISBURSED** | 按策略表（若有） | 办贷端 | 若有 DISBURSED 触发动作，则调用对应模块 |
| **全生命周期** | 监管报送与报表组件 | 办贷端 | 通过数据视图或事件间接消费进件数据，进件管理不直接调用报表写入，仅保证进件与状态可被查询 |

### 3.2 对下游组件的接口与数据要求

- **授权配置管理组件（办贷端）**：提供按「产品 + 客户类型 + 业务阶段/进件类型」查询进件阶段**风控规则包 ID 列表**的接口；进件管理只消费规则包 ID，不解析规则内容。
- **风控组件（办贷端）**：提供按规则包 ID 执行进件准入的接口，入参含进件 ID、产品、客户标识等，返回准入结果（如 R1/R3/拒绝）。
- **工作流/审批流组件（办贷端）**：支持按进件 ID、办贷任务类型（或流程模板 ID）启动流程实例；支持节点完成时回调进件管理更新状态。
- **客户/评级/授信/用信/额度/合同管理组件（办贷端）**：均需提供**按进件号幂等**的 upsert 接口（如 upsertXxxByIntake(intakeId, payload)），具体见《进件中心-业务数据同步策略设计》第四节；进件管理在状态迁移时按策略表调用，不直接读业务库。

#### 内部总体设计（统一流程 + Java 风格伪代码）

进件接收采用**一套统一实现**，按「产品 + 进件类型 + 状态」查配置表区分行为，不在代码中按产品名分支。以下为内部主流程说明及 Java 风格伪代码（含注释）。

##### 流程步骤概要

1. **幂等控制**：按 `requestId + productId + intakeType` 查已有进件；存在且 payload 一致则直接返回既有进件号。  
2. **基础校验**：必填字段、进件类型枚举、产品存在且启用；不通过则返回业务错误，不落进件。  
3. **创建进件单**：生成进件号，落库，状态 = `RECEIVED`。  
4. **执行 RECEIVED 同步策略**：查同步策略表，对客户/授信/用信/额度等模块按配置执行 `upsertByIntake`（早同步产品在此初始化）。  
5. **进件准入**：查授权配置获取进件阶段规则包 ID；若配置要求先授权则校验授权记录（已授权且未到期则无需重新授权）；调用风控组件执行进件准入规则包，得到 R1/R3/拒绝。  
6. **路由**：拒绝则更新进件状态为 `REJECTED` 并记录原因；R1 直通或 R3 转线下则执行**进件自动分配**（R1 分配至签约池/自助任务池，R3 分配至客户经理队列），由工作流按分配规则执行；R1 并启动线上签约流程，R3 并启动线下办贷流程模板。  
7. **返回响应**：进件号、进件类型、产品、初始状态、创建时间。  

##### Java 风格伪代码（含注释）

```java
/**
 * 进件接收入口：统一处理授信进件(CREDIT_APPLY)、贷款/用信进件(LOAN_APPLY)、提额进件(LIMIT_INCREASE)。
 * 行为差异由同步策略表、授权配置、风控与路由配置驱动，不按产品名写 if-else。
 */
public CreateIntakeResponse createIntake(CreateIntakeRequest request) {

    // ----- 1. 幂等控制：同一 requestId+productId+intakeType 视为同一笔请求 -----
    String requestKey = buildRequestKey(request.getRequestId(), request.getProductId(), request.getIntakeType());
    Intake existing = intakeRepository.findByRequestKey(requestKey);
    if (existing != null) {
        // 可选：校验关键业务字段与本次请求是否一致，防止同一 requestId 传入不同数据
        if (isPayloadConsistent(existing, request)) {
            return toResponse(existing);
        }
        throw new BusinessException("IDEMPOTENT_CONFLICT", "同一请求键下存在进件且数据不一致");
    }

    // ----- 2. 基础参数校验：不通过则直接返回，不落进件 -----
    validateBasic(request);  // 必填、枚举、产品存在且启用、按 intakeType 的必填业务字段

    // ----- 3. 解析客户标识：已有 customerId 或待按快照创建临时客户 -----
    CustomerKey customerKey = customerResolver.resolve(request.getCustomerId(), request.getCustomerSnapshot(), request.getCustomerType());

    // ----- 4. 生成进件号并持久化进件单，初始状态 RECEIVED -----
    String intakeId = idGenerator.nextId();
    Instant now = clock.instant();
    // 业务字段（applyAmount/term/purpose/targetCreditId/guaranteeInfo/extraFields）由 buildPayloadSnapshot 纳入快照，供同步策略与风控使用
    // 前置结果引用（preCreditRefId / authRecordId / preCheckResultRef）仅落库供审计与追溯，进件管理不据此校验
    Intake intake = Intake.builder()
            .intakeId(intakeId)
            .requestId(request.getRequestId())
            .productId(request.getProductId())
            .intakeType(request.getIntakeType())   // CREDIT_APPLY / LOAN_APPLY / LIMIT_INCREASE
            .channel(request.getChannel())
            .channelScene(request.getChannelScene())   // 可选，渠道场景（埋点与精细化配置）
            .sourceSystem(request.getSourceSystem())
            .customerKey(customerKey)
            .customerType(request.getCustomerType())
            .status(IntakeStatus.RECEIVED)
            .createdAt(now)
            .payloadSnapshot(buildPayloadSnapshot(request))   // 含 applyAmount、term、purpose、targetCreditId、guaranteeInfo、extraFields 等
            .preCreditRefId(request.getPreCreditRefId())           // 预授信结果引用，可选
            .authRecordId(request.getAuthRecordId())               // 授权记录编号，可选；步骤 6 若配置要求先授权会校验授权有效性，此处仅落库
            .preCheckResultRef(request.getPreCheckResultRef())     // 申请前规则包结果引用，可选；申请前规则包由贷款申请组件在提交前执行，进件管理不校验
            .build();
    intakeRepository.insert(intake);

    // ----- 5. 触发 RECEIVED 状态下的同步策略（早同步产品在此初始化客户/授信/用信/额度等） -----
    executeSyncStrategy(intake.getProductId(), intake.getIntakeType(), IntakeStatus.RECEIVED, intakeId, buildBusinessSnapshot(request, customerKey));

    // ----- 6. 进件准入：授权配置取规则包 → 校验先授权（若配置要求）→ 执行进件风控 -----
    List<String> rulePackageIds = authConfigClient.getIntakeRulePackageIds(
            intake.getProductId(), intake.getCustomerType(), "INTAKE_ACCESS", intake.getIntakeType());

    if (authConfigClient.isAuthRequiredForIntake(intake.getProductId(), intake.getCustomerType(), "INTAKE_ACCESS")) {
        // 同一产品+客户+阶段下，已存在有效未到期授权则无需重新授权
        boolean hasValidAuth = authService.hasValidUnexpiredAuth(
                customerKey, intake.getProductId(), "INTAKE_ACCESS", request.getAuthRecordId());
        if (!hasValidAuth) {
            handleAuthNotSatisfied(intake, request);  // 按 ENFORCE/REMIND/NONE 决定拒绝或仅提示
            throw new BusinessException("AUTH_REQUIRED", "进件阶段要求先授权且当前未满足");
        }
    }

    RiskResult riskResult = riskEngine.executeIntakeRules(
            rulePackageIds, intakeId, intake.getProductId(), customerKey, buildRiskInputSnapshot(request, intake));
    // riskResult: R1 / R3 / REJECT + 原因等

    // ----- 7. 路由与进件分配：根据风控结果 + 产品配置决定 R1 / R3 / 拒绝；R1/R3 分支执行进件自动分配并启动后续流程 -----
    RouteDecision route = routeConfig.decide(intake.getProductId(), intake.getIntakeType(), riskResult);

    if (route == RouteDecision.REJECT) {
        intake.setStatus(IntakeStatus.REJECTED);
        intake.setRejectReason(riskResult.getReason());
        intakeRepository.updateStatus(intake);
        executeSyncStrategy(intake.getProductId(), intake.getIntakeType(), IntakeStatus.REJECTED, intakeId, null);
        throw new BusinessException("INTAKE_REJECTED", "进件风控准入不通过");
    }

    if (route == RouteDecision.R1) {
        workflowClient.assignIntake(intake, route, null);            // 进件自动分配：分配至签约池/自助任务池，由工作流按分配规则执行
        workflowClient.startOnlineFlow(intake, riskResult);          // 生成签约任务、启动线上签约流程
    } else if (route == RouteDecision.R3) {
        String taskType = determineLoanTaskType(intake.getProductId(), intake.getIntakeType(), intake.getChannel());
        workflowClient.assignIntake(intake, route, taskType);        // 进件自动分配：分配至客户经理队列，由工作流按分配规则执行
        workflowClient.startOfflineFlow(intake, taskType, riskResult);  // 启动办贷流程模板（三合一/二合一/单独）
    }

    // ----- 8. 返回响应 -----
    return CreateIntakeResponse.builder()
            .intakeId(intakeId)
            .intakeType(intake.getIntakeType())
            .productId(intake.getProductId())
            .initialStatus(intake.getStatus())
            .createdAt(intake.getCreatedAt())
            .build();
}
```

##### 说明与约定

- **幂等键**：`requestId + productId + intakeType`；同一键下若已存在进件且本次 payload 一致则直接返回原进件号。  
- **前置条件**：必填字段、产品与进件类型合法性在步骤 2 完成；不满足则抛业务异常，不写库。  
- **进件必存**：步骤 4 执行后即有一条 `RECEIVED` 进件；风控拒绝只更新状态为 `REJECTED`，不删除记录。  
- **前置结果引用（preCreditRefId / authRecordId / preCheckResultRef）**：在步骤 4 建进件时随请求体一并落库，**仅用于审计与追溯**。其中 `authRecordId` 在步骤 6 若配置要求「先授权」时会参与授权有效性校验；`preCheckResultRef` 为贷款申请组件在提交前调用申请前规则包产生的结果引用，进件管理**不校验**该引用是否存在或是否通过，只持久化便于事后排查。  
- **业务字段与快照**：`applyAmount`、`term`、`purpose`、`targetCreditId`、`guaranteeInfo`、`extraFields` 通过 `buildPayloadSnapshot(request)` 进入进件快照并落库；`buildBusinessSnapshot(request, customerKey)`、`buildRiskInputSnapshot(request, intake)` 在同步策略与进件风控时从 request 及进件中组装所需业务数据，实现时需保证这些 helper 覆盖上述业务字段。  
- **进件自动分配**：步骤 7 中 R1/R3 分支均先调用工作流 `assignIntake`（进件分配），再调用 `startOnlineFlow`/`startOfflineFlow`。分配规则（按机构、产品、客户经理负载、客户归属等）在工作流/审批流组件侧配置并执行，进件管理仅传入进件、路由结果及（R3 时）办贷任务类型。  
- **产品差异**：同步策略、授权是否必选、规则包 ID、R1/R3 条件均来自配置表（产品+进件类型+阶段），代码中不按产品 ID 分支。  
  - **示例（湖南农信-公积金贷款 HNGJJ）**：在《产品流程-组件调用场景视图与配置表》的风控与路由配置表中约定：**风控返回 R1 且申请金额≤20 万时按 R1 直通线上签约**；**风控返回 R1 且申请金额>20 万时按 R3 分支转线下**；**风控返回 R3 时统一转线下**；**风控返回 R5 时拒绝并将进件状态置为 REJECTED**。上述规则完全通过路由配置表与 `routeConfig.decide(...)` 承载，无需在代码中写死 20 万阈值。  

### 4.3 进件状态更新（回调）— 请求与返回字段定义

工作流/审批流、签约组件、放款组件在节点完成或业务完成后，通过本接口回调进件管理更新进件状态（如 APPROVED / SIGNED / DISBURSED / REJECTED），进件管理按新状态执行同步策略。**统一一个状态更新接口**，不按产品拆多套。

#### 请求字段

| 字段名 | 字段中文名 | 类型 | 是否必填 | 说明 |
|--------|------------|------|----------|------|
| `intakeId` | 进件编号 | string | 是 | 需要更新的进件号。 |
| `currentStatus` | 当前进件状态 | enum | 否（推荐传） | 调用方认为的当前状态，用于防并发/误操作；不传则仅按状态机校验目标状态是否合法。数据字典（英文:中文）：RECEIVED:已接收、UNDER_REVIEW:审批/调查中、APPROVED:审批通过、SIGNED:签约完成、DISBURSED:已放款、REJECTED:已拒绝。 |
| `newStatus` | 目标进件状态 | enum | 是 | 本次要更新到的状态。数据字典（英文:中文）：同上。 |
| `sourceSystem` | 来源系统 | string | 是 | 触发本次状态更新的系统。数据字典（英文:中文）：WORKFLOW:工作流/审批流、SIGNING:签约组件、DISBURSEMENT:放款组件。 |
| `operatorId` | 操作人编号 | string | 否 | 办贷端人工操作时填客户经理/审批人；系统自动流转可空。 |
| `operationTime` | 操作时间 | datetime | 否 | 业务发生时间，用于审计；不传则用服务端当前时间。 |
| `bizContext` | 业务上下文 | object | 否 | 与状态更新相关的摘要：如审批结论、签约任务 ID、放款流水号等，供同步策略或下游使用。 |
| `comment` | 备注说明 | string | 否 | 状态变更说明（如“自动审批通过”“人工复核拒绝：征信不良”）。 |
| `requestId` | 请求唯一标识 | string | 否 | 回调幂等键；同一回调重试时保持不变，避免重复执行。 |

#### 返回字段

| 字段名 | 字段中文名 | 类型 | 说明 |
|--------|------------|------|------|
| `intakeId` | 进件编号 | string | 本次更新的进件号。 |
| `fromStatus` | 原状态 | enum | 更新前的进件状态。 |
| `toStatus` | 新状态 | enum | 本次更新后的进件状态。 |
| `updatedAt` | 更新时间 | datetime | 本次状态更新的时间。 |

进件单建议至少包含：进件 ID、产品、进件类型、客户标识、当前状态、创建时间、来源（渠道/办贷）、同步执行日志（可选但建议有）。

#### 内部处理逻辑（Java 风格伪代码，概要）

##### 流程步骤概要

0. **可选：requestId 幂等**：若调用方传入 `requestId`，先查幂等缓存；已处理过则直接返回缓存结果，避免重复执行同步策略。  
1. **查找进件**：按 `intakeId` 查询进件记录；不存在则返回业务错误（进件不存在）。  
2. **可选：校验当前状态**：若调用方传入 `currentStatus`，与进件当前状态比对，不一致则视为并发/误操作，返回状态冲突错误。  
3. **校验状态迁移合法性**：按配置的状态机定义，判断从当前状态到 `newStatus` 是否允许迁移；不允许则返回非法迁移错误。  
4. **更新进件状态及上下文**：将进件状态更新为 `newStatus`；**操作时间**取 `operationTime`（传则用调用方时间，不传则用服务端当前时间）；记录操作人、来源系统、备注及业务上下文快照，持久化。  
5. **在新状态下执行同步策略**：根据「产品 + 进件类型 + 新状态」查同步策略表，对客户/授信/用信/额度/合同等模块按配置执行 `upsertByIntake`（如 APPROVED/SIGNED/DISBURSED 下的更新或终态确认）。  
6. **返回结果**：返回进件号、原状态、新状态、更新时间。幂等缓存仅在步骤 4、5 均成功后才写入，保证「命中缓存即表示上次已完整完成状态更新与同步」。

##### Java 风格伪代码（含注释）

```java
/**
 * 进件状态更新入口：工作流/签约/放款等系统在节点完成后回调本方法，
 * 将进件从当前状态迁移到目标状态，并在新状态下执行同步策略。
 */
public UpdateIntakeStatusResponse updateIntakeStatus(UpdateIntakeStatusRequest request) {

    // 0. 可选：按 requestId 幂等（同一回调重试时 requestId 不变，避免重复执行同步策略）
    if (request.getRequestId() != null) {
        UpdateIntakeStatusResponse cached = statusUpdateCache.findByRequestId(request.getRequestId());
        if (cached != null) {
            return cached;   // 已处理过，直接返回原结果
        }
    }

    // 1. 查找进件
    Intake intake = intakeRepository.findById(request.getIntakeId());
    if (intake == null) {
        throw new BusinessException("INTAKE_NOT_FOUND", "进件不存在");
    }

    // 2. 可选：校验 currentStatus，防并发/误操作
    if (request.getCurrentStatus() != null
            && !request.getCurrentStatus().equals(intake.getStatus())) {
        throw new BusinessException("STATUS_MISMATCH",
                "当前进件状态与调用方传入不一致，可能存在并发更新");
    }

    // 3. 校验状态迁移是否合法（按配置的状态机定义）
    if (!stateMachine.canTransit(intake.getStatus(), request.getNewStatus())) {
        throw new BusinessException("ILLEGAL_STATUS_TRANSITION",
                "不允许从 " + intake.getStatus() + " 迁移到 " + request.getNewStatus());
    }

    IntakeStatus fromStatus = intake.getStatus();
    IntakeStatus toStatus = request.getNewStatus();

    // 4. 更新进件状态及上下文信息（operationTime 不传则用服务端当前时间）
    Instant updateTime = request.getOperationTime() != null ? request.getOperationTime() : clock.instant();
    intake.setStatus(toStatus);
    intake.setLastUpdatedAt(updateTime);
    intake.setLastUpdatedBy(request.getOperatorId());
    intake.setLastUpdateSource(request.getSourceSystem());
    intake.setLastUpdateComment(request.getComment());
    intake.setBizContextSnapshot(buildBizContextSnapshot(request.getBizContext()));

    intakeRepository.updateStatusAndContext(intake);

    // 5. 在新状态下执行同步策略（如 APPROVED/SIGNED/DISBURSED）；先执行再写幂等缓存，保证「返回成功即已落库且同步已执行」
    executeSyncStrategy(
            intake.getProductId(),
            intake.getIntakeType(),
            toStatus,
            intake.getIntakeId(),
            buildBusinessSnapshotForStatus(intake, request.getBizContext())
    );

    UpdateIntakeStatusResponse response = UpdateIntakeStatusResponse.builder()
            .intakeId(intake.getIntakeId())
            .fromStatus(fromStatus)
            .toStatus(toStatus)
            .updatedAt(intake.getLastUpdatedAt())
            .build();
    if (request.getRequestId() != null) {
        statusUpdateCache.put(request.getRequestId(), response);   // 幂等：仅在全流程成功后才缓存，重试时步骤 0 命中则说明上次已完整成功
    }

    // 6. 返回结果
    return response;
}
```

### 4.4 错误码与幂等策略说明

本小节给出进件管理组件在「进件接收」「进件状态更新」两个接口上的**错误码建议与幂等策略约定**，便于上游统一处理与排查。

#### 4.4.1 错误码分类（建议）

| 错误码 | 场景 | 说明 |
|--------|------|------|
| `VALIDATION_FAILED` | 请求参数校验失败 | 必填字段缺失、类型/格式不合法、进件类型或产品不存在/未启用等。 |
| `IDEMPOTENT_CONFLICT` | 幂等冲突 | 同一幂等键（如 requestId+productId+intakeType）下已存在进件，但本次请求的关键业务字段与历史记录不一致。 |
| `AUTH_REQUIRED` | 授权前置条件未满足 | 授权配置要求「先授权」，但当前客户/阶段下未找到有效未到期授权记录。 |
| `INTAKE_REJECTED` | 进件风控准入拒绝 | 进件阶段规则包执行结果为拒绝，或路由配置根据风控结果决定拒绝；进件状态被置为 REJECTED。 |
| `INTAKE_NOT_FOUND` | 进件不存在 | 状态更新或查询时，按传入的 intakeId 未找到进件记录。 |
| `STATUS_MISMATCH` | 状态不一致 | 状态更新接口中，调用方传入的 currentStatus 与当前进件状态不一致，可能存在并发或误操作。 |
| `ILLEGAL_STATUS_TRANSITION` | 非法状态迁移 | 试图从当前状态迁移到状态机未允许的目标状态。 |

> 以上错误码为建议值，最终编码与文案可按项目统一错误规范调整；关键是区分「参数/前置问题」「幂等冲突」「授权/风控导致的业务拒绝」与「状态机/并发问题」几类。

#### 4.4.2 幂等策略约定

- **进件接收（createIntake）**  
  - 幂等键：`requestId + productId + intakeType`。  
  - 行为：  
    - 若该键下**不存在**进件 → 创建新进件并返回。  
    - 若该键下已存在进件且关键字段与本次请求一致 → 直接返回已有进件（视为重复请求）。  
    - 若该键下已存在进件但关键字段不一致 → 返回 `IDEMPOTENT_CONFLICT` 错误，上游需排查调用逻辑。  
  - 建议：上游系统在**重试渠道请求**时务必使用相同的 requestId，避免重复建单。

- **进件状态更新（updateIntakeStatus）**  
  - 幂等键：`requestId`（可选） + `intakeId + newStatus`。  
  - 行为：  
    - 对于同一 `intakeId + newStatus`，重复调用且请求体一致，应保证状态更新和同步策略执行**幂等**（不产生重复副作用）；  
    - 若调用方传入 `currentStatus`，且与进件当前状态不一致，则返回 `STATUS_MISMATCH`，上游应停止重试并排查是否并发更新或重复回调。  
  - 建议：工作流/签约/放款等系统在**重试同一业务节点回调**时，保持 requestId 不变，便于日志关联与排查。

> 总体原则：**「参数/配置/前置条件错误」直接报错不建单或不改状态；「幂等冲突」报专门错误码提示调用方排查；「风控/授权拒绝」通过业务错误码返回，同时保留进件记录并更新状态便于审计。**

## 五、多产品/多模式支持要点

- **进件类型**：CREDIT_APPLY / LOAN_APPLY / LIMIT_INCREASE 等，不同进件类型可配置不同下游流程模板与同步策略。
- **同步时机配置化**：通过同步策略表支持晚同步（如湖北）、早同步（如湖南、长沙），无需改代码。
- **路由规则配置化**：按产品配置「风控准入结果 + 额度阈值 + 其他条件」决定 R1 直通或 R3 转人工；办贷任务类型按产品+进件类型+渠道配置。
- **分配规则**：进件自动分配时可按机构、产品、客户经理负载等配置，由工作流执行具体分配与任务生成。

## 六、单组件设计样板结构说明（供推广）

以下为本文档采用的**单组件设计章节结构**，后续为其他组件（如风控、签约、授信管理等）做单组件详细设计时，可复用相同结构，仅替换为对应组件的内容。

| 章节 | 目的 | 推广时替换为 |
|------|------|--------------|
| **文档说明与样板用途** | 说明文档定位及与总览/段落设计的关系 | 该组件的定位与依据文档 |
| **一、职责与边界（简述）** | 精炼职责与边界，与总览一致 | 目标组件的职责与边界 |
| **二、前置调用场景与调用条件** | 按调用者细分场景，逐场景列出前置条件；说明与配置组件关系；给出汇总检查清单 | 目标组件的前置调用方、每类调用场景与条件、配置依赖、检查清单 |
| **三、后续调用** | 按状态/场景的后续调用矩阵及对下游的接口/数据要求 | 目标组件的下游调用矩阵与接口要求 |
| **四、关键接口与数据** | 本组件对外提供的主要接口与核心数据 | 目标组件的接口与数据 |
| **五、多产品/多模式支持要点** | 配置化、多产品差异点 | 目标组件的多产品支持要点 |
| **六、单组件设计样板结构说明** | 列出样板章节及推广方式 | 可保留或改为「与其它组件设计文档的引用关系」 |

按上述结构编写单组件设计，可保证：**前置谁在什么条件下调我、我在什么情况下调谁、依赖哪些配置**三者清晰可查，便于实现、测试与评审，并与其他段落设计、总览文档保持一致。推广到其他组件时，**所有涉及组件均需标注（渠道端）或（办贷端）**，必要时用「渠道端/办贷端入口」「渠道端、办贷端 PAD」等区分双端存在的组件。
