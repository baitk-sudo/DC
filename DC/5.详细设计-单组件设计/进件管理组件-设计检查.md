## 二、分项检查

### 2.1 文档结构与样板

- **固定章节**：职责与边界 → 前置调用场景与条件 → 后续调用 → 关键接口与数据 → 多产品支持 → 样板说明，齐全。
- **依据文档**：总览、第3/4/5段、进件中心同步策略、场景视图与配置表、业务步骤级分析、产品场景对照（建议类）、词汇表及端到端示例引用正确。
- **端别标注**：渠道端/办贷端在职责、前置调用者、后续调用、接口上游中均有标注，符合约定。

### 2.2 职责与边界

- **职责**：统一接收授信/贷款/提额进件、维护状态机、按同步策略触发同步、按风控结果路由（R1/R3）、R3 时解析办贷任务类型并驱动工作流、进件自动分配，与第4段 1.1 及总览中「进件管理组件」行一致。
- **边界**：不执行业务规则与额度测算、不定义审批流程与审批人、不直接生成合同/签约内容、「何时对谁做什么」由同步策略配置决定，表述清晰无越界。

### 2.3 前置调用场景与条件

- **2.1 前置调用者与进件类型**：授信申请→CREDIT_APPLY、贷款申请→LOAN_APPLY、提额→LIMIT_INCREASE；工作流/签约/放款不创建进件仅回调，与总览及词汇表一致。
- **2.2 按调用者场景**：  
  - 场景 A（授信申请）、B（贷款申请）、C（提额）：数据完整、产品与进件配置、客户可识别、先预授信/先授权（若产品要求）等条件完整。  
  - 场景 B 单独说明了「申请前规则包」由贷款申请组件在 createIntake 前执行，进件管理不校验、仅可选落库 preCheckResultRef，与场景视图一致。  
  - 场景 D（工作流回调）、E（签约→SIGNED）、F（放款→DISBURSED）：进件已存在、上下文完整、同步策略就绪等条件明确。
- **2.3 配置依赖**：产品管理、同步策略表、授权配置管理（进件规则包 ID）、工作流/办贷任务类型、进件状态机，与第4段及总览一致。
- **2.4 前置条件汇总表**：数据完整性、配置就绪、先决业务条件按场景（A/B/C 等）划分，便于实现与测试逐项检查。

### 2.4 后续调用与依赖

- **3.1 按状态/场景的后续调用矩阵**：  
  - RECEIVED：授权配置（规则包 ID）、客户管理、风控、早同步下客户/评级/授信/用信/额度。  
  - R1：工作流、签约、合同管理；R3：工作流（办贷任务类型）。  
  - APPROVED/SIGNED/DISBURSED：按策略表调用各业务组件 upsertByIntake。  
  - 全生命周期：监管报送与报表通过数据视图/事件间接消费。  
  与第4段 1.3、进件中心同步策略设计一致。
- **3.2 对下游接口要求**：授权配置（规则包 ID 查询）、风控（进件准入）、工作流（启动模板+回调）、各业务组件（upsertByIntake 幂等），与《进件中心-业务数据同步策略设计》第四节接口契约一致。

### 2.5 关键接口与数据

- **进件接收（createIntake）**：  
  - 请求：requestId/productId/intakeType/channel、客户（customerId/customerSnapshot/customerType）、业务字段（applyAmount/term/purpose/targetCreditId 等）、可选 preCreditRefId/authRecordId/preCheckResultRef，定义完整。  
  - 返回：intakeId、intakeType、productId、initialStatus、createdAt，满足上游使用。  
  - 内部流程：幂等→校验→建进件→RECEIVED 同步→授权配置取规则包→先授权校验（若配置）→风控准入→路由与进件分配→返回，与 2.2/2.3/3.1 描述一致。  
  - 伪代码与说明（幂等键、前置结果引用仅落库/审计、authRecordId 在「先授权」时参与校验、业务字段与快照、进件自动分配、产品差异靠配置）清晰，可指导实现。
- **进件状态更新（updateIntakeStatus）**：  
  - 请求：intakeId、currentStatus（可选）、newStatus、sourceSystem、operatorId/operationTime/bizContext/comment/requestId（幂等），字段齐全。  
  - 流程：幂等→查进件→currentStatus 校验→状态机校验→更新状态与上下文→执行新状态同步策略→返回，与 3.1 及同步策略设计一致。
- **错误码与幂等**：VALIDATION_FAILED、IDEMPOTENT_CONFLICT、AUTH_REQUIRED、INTAKE_REJECTED、INTAKE_NOT_FOUND、STATUS_MISMATCH、ILLEGAL_STATUS_TRANSITION 分类合理；createIntake/updateIntakeStatus 幂等策略约定明确，便于上游与测试使用。

### 2.6 与总览、第4段、同步策略的一致性

- **总览**：前置调用者（授信/贷款/提额、工作流、签约、放款）、前置条件（数据完整、同步策略与状态机、先授权可校验）、后续调用（授权配置、风控、工作流、客户/评级/授信/用信/额度/合同、监管报送），单组件文档与总览中「进件管理组件」行一致。
- **第4段**：职责与边界、多产品支持点（进件类型、同步时机、路由规则、分配规则）、关键接口与上下游、对已分析产品的支撑（湖北/湖南/长沙银行），单组件文档与第4段 1.1～1.4 对齐。
- **进件中心-业务数据同步策略设计**：状态枚举（RECEIVED/UNDER_REVIEW/APPROVED/SIGNED/DISBURSED/REJECTED）、策略配置表概念、状态迁移时查表并调用 upsertByIntake、各业务组件接口契约（upsertXxxByIntake），单组件文档引用正确且行为一致。

## 三、可改进与建议

1. **进件查询接口**  
   - 现状：第四节表格中列出「进件查询」，但未像 createIntake/updateIntakeStatus 一样给出请求/返回字段及说明。  
   - 建议：若实现阶段需要明确契约，可增补「进件查询」的入参（intakeId/客户/产品/状态/时间范围等）与返回字段（列表/分页、轨迹等）简要说明；若仅内部或由他组件封装查询，可在文档中注明「以查询 API 文档或实现为准」。

2. **办贷任务类型枚举与配置表引用**  
   - 现状：文档多处提到「三合一/二合一/单独」及「产品+进件类型+渠道」解析办贷任务类型，未显式引用《产品流程-组件调用场景视图与配置表》或具体配置表名。  
   - 建议：在 2.3 或 3.1 中增加一句「办贷任务类型枚举及解析规则见《产品流程-组件调用场景视图与配置表》或产品×进件×渠道路由配置表」，便于实现与配置维护时溯源。

## 四、检查清单（实现与测试可复用）

实现或测试进件管理组件时，可按以下清单自检：

- [ ] 六类前置场景（A～F）的前置条件在实现中均有校验或约定（含申请前规则包由申请组件执行、进件管理不校验）。
- [ ] 进件接收幂等键为 requestId+productId+intakeType；同一键下 payload 一致返回原进件，不一致返回 IDEMPOTENT_CONFLICT。
- [ ] 进件接收后必先落库 RECEIVED，再执行 RECEIVED 同步策略、进件准入、路由与分配；拒绝时仅更新为 REJECTED，不删进件。
- [ ] 先授权校验（若配置）依赖授权配置管理 + 授权记录；authRecordId 落库并参与校验，preCheckResultRef 仅落库不校验。
- [ ] 状态更新接口支持 requestId 幂等；状态机校验、currentStatus 校验、新状态同步策略执行顺序与文档一致。
- [ ] 下游调用（授权配置、风控、工作流、客户/评级/授信/用信/额度/合同）的时机与同步策略表一致，且依赖《进件中心-业务数据同步策略设计》中的接口契约。
