# 湖南农信-公积金贷款 — 进件端到端示例（贷款申请场景）

[进件管理组件详细设计](../5.详细设计-单组件设计/组件设计-单组件样板-进件管理组件.md)

> 目标：用一条「湖南公积金贷款申请」的完整链路，串起**渠道端组件 → 进件管理组件（createIntake/updateIntakeStatus）→ 同步策略/状态机**，作为《进件管理组件 — 单组件详细设计》的配套示例。

---

## 一、整体链路鸟瞰

以「客户在渠道前端发起湖南公积金贷款申请」为主线（对应场景 B：贷款申请组件 → 进件管理，`LOAN_APPLY`）：

1. 客户在渠道前端进入公积金贷款产品页面。  
2. 渠道前端通过**营销方案管理组件**获取产品信息与贷款申请页面要素配置。  
3. 客户在**贷款申请组件**中填写贷款申请信息（金额、期限、用途等）。  
4. 贷款申请组件按产品配置可选调用：  
   - **预授信白名单/预授信结果检查**（如客户已在预授信名单或已有预授信结果，则可在页面提示/带出额度，必要时按配置校验是否允许继续）；  
   - **申请前规则包**（风控组件）；  
   - **客户授权准入组件**（湖南为申请内授权，必经前置）。  
5. 贷款申请组件在满足「预授信白名单/预授信结果检查（如要求）+ 申请前规则包（如启用）+ 申请内授权」后，调用**进件管理组件的进件接收接口（createIntake，`intakeType=LOAN_APPLY`）**，创建贷款进件，进件状态 = `RECEIVED`。  
6. 进件管理组件在 `RECEIVED` 状态下按**同步策略表**对客户/评级/授信/用信/额度等模块执行初始化（湖南为早同步）。  
7. 进件管理组件按**授权配置 + 进件规则包 ID**调用风控组件执行**进件准入**，得到 R1/R3/拒绝。  
8. 按路由配置：  
   - R1：启动线上流程或直通签约；  
   - R3：解析办贷任务类型（三合一/二合一/单独），启动线下三合一/审批流程。  
9. 后续通过**进件状态更新接口（updateIntakeStatus）**在 APPROVED / SIGNED / DISBURSED 等节点更新进件状态，并按同步策略表驱动业务模块终态确认。

## 二、关键节点与组件角色

### 2.1 渠道端：贷款申请前置

- **营销方案管理组件（渠道前端）**  
  - 提供湖南公积金贷款产品列表/详情及贷款申请页面要素配置（字段、必填、校验规则等）。  
  - 贷款申请组件据此渲染申请表单。

- **贷款申请组件（渠道端）**  
  - 职责：承载客户在渠道上发起湖南公积金贷款申请（场景 B，`LOAN_APPLY`）。  
  - 关键前置：  
    - 客户识别：已有 customerId，或通过用户注册+客户管理在申请过程中创建临时客户；  
    - 对公客户：需先完成对公注册与基础认证；  
    - 授权：湖南为**申请内授权**，在贷款申请页面内调用客户授权准入组件完成授权（授权配置决定授权文件与规则包）。  
  - 可选前置：  
    - 申请前规则包（反欺诈/参数准入）：按产品配置由贷款申请组件在提交前调用风控组件执行，未通过则不允许提交进件。

### 2.2 进件接收：createIntake（LOAN_APPLY）

- 上游：**贷款申请组件（渠道端）**  
  - 在前置条件满足后，构造 `CreateIntakeRequest` 调用进件管理组件：  
    - `requestId`：渠道请求唯一标识；  
    - `productId`：湖南公积金贷款产品编码；  
    - `intakeType`：`LOAN_APPLY`；  
    - `channel` / `sourceSystem`：渠道与来源系统；  
    - `customerId` 或 `customerSnapshot` + `customerType`；  
    - 业务字段：`applyAmount`、`term`、`purpose`、`targetCreditId`（如有授信/额度项）；  
    - 前置结果引用：`preCreditRefId`（如有预授信）、`authRecordId`（申请内授权记录）、`preCheckResultRef`（申请前规则包结果）。

- 进件管理组件内部处理（见单组件文档 4.1 内部总体设计）：  
  1. 幂等检查：按 `requestId + productId + intakeType` 查既有进件，避免重复建单。  
  2. 校验必填与产品/进件类型合法性。  
  3. 生成 `intakeId`，写入进件表，状态 = `RECEIVED`。  
  4. 查同步策略表（湖南为早同步）：在 `RECEIVED` 状态下对客户/评级/授信/用信/额度等模块执行初始化 `upsertByIntake`。  
  5. 调用授权配置管理组件获取**进件阶段规则包 ID**，并结合授权记录（已授权且未到期则不再重复授权），调用风控组件执行**进件准入**。  
  6. 按路由配置对风控结果做 R1/R3/拒绝决策：  
     - 拒绝：进件状态置为 `REJECTED`，记录原因；  
     - R1：启动线上流程（如自动签约/分配）；  
     - R3：确定办贷任务类型（如三合一）并调用工作流组件启动线下流程。

### 2.3 状态更新：updateIntakeStatus

湖南公积金的线下三合一/审批和签约、放款等节点，均通过 **`updateIntakeStatus`** 回调进件管理组件更新进件状态，并触发同步策略：

- **示例节点 1：三合一/审批通过**  
  - 来源系统：工作流组件（办贷端）。  
  - 调用 `updateIntakeStatus`：  
    - `intakeId`：湖南公积金贷款进件号；  
    - `currentStatus`: `UNDER_REVIEW`；  
    - `newStatus`: `APPROVED`；  
    - `sourceSystem`: `WORKFLOW`；  
    - `bizContext`: 审批结论摘要（如额度、期限、利率建议等）。  
  - 进件管理：  
    - 校验状态迁移合法性；  
    - 更新进件状态为 `APPROVED`；  
    - 根据同步策略表在 `APPROVED` 下对授信/用信/额度/合同等模块执行 `upsertByIntake`（如生成合同草稿/待签状态）。

- **示例节点 2：客户在线签借款合同**  
  - 来源系统：签约组件（渠道端/进件端）。  
  - 回调 `updateIntakeStatus`：`newStatus = SIGNED`。  
  - 进件管理：  
    - 在 `SIGNED` 状态下按同步策略对客户/授信/用信/额度/合同执行更新或终态确认（湖南为早同步，SIGNED 时多为“更新/终态确认”而非首次写入）。

- **示例节点 3：放款完成**  
  - 来源系统：放款组件（办贷端）。  
  - 回调 `updateIntakeStatus`：`newStatus = DISBURSED`。  
  - 进件管理：  
    - 将进件状态更新为 `DISBURSED`；  
    - 若同步策略表中对 DISBURSED 有配置，则执行终态确认或报表触发等动作。

## 三、与产品场景与配置的对应关系

- **场景视图**：  
  - 在《产品流程-组件调用场景视图与配置表》1.3 湖南农信中，阶段「3、客户发起贷款申请」和「4、系统进件接收」，分别对应：  
    - 渠道端的**贷款申请组件 → 进件管理组件**（本示例的上游）；  
    - 办贷端的**进件管理组件 → 授权配置/客户管理/风控/评级/授信/用信/合同/签约组件**（本示例中的内部同步与进件准入）。  

- **进件类型与前置条件**：  
  - 本示例使用 `LOAN_APPLY`，与《进件管理组件 — 单组件详细设计》中 2.1/2.2 场景 B 定义一致；  
  - 前置条件 B.1～B.4 中“申请内授权为必经前置”“预授信结果可优先参考但不强制”，与《进件管理组件-产品场景对照分析》中湖南公积金的建议配置一致。

- **同步策略与早同步模式**：  
  - 进件在 `RECEIVED` 状态下即初始化客户/评级/授信/用信等模块，之后在 `APPROVED/SIGNED/DISBURSED` 等状态下做更新/终态确认，符合《进件生命周期与同步策略关系分析》中对湖南模式的“早同步”描述。

---

## 四、小结

- 本示例展示了湖南公积金贷款申请在**一个进件维度**上的完整生命周期：从渠道贷款申请组件发起，到进件接收、同步策略驱动、进件准入风控与 R1/R3 路由，再到审批、签约、放款时通过状态更新回调进件管理。  
- 所有产品差异（是否先预授信、是否先授权、早/晚同步、R1/R3 条件等）都通过**授权配置、产品配置、同步策略表与路由配置**承载，进件管理的实现保持为一套统一流程。  
- 在后续为其他产品（如湖北公积金、税E贷、政采贷）设计端到端示例时，可复用本结构，仅替换组件链路与配置差异。

## 五、与 CSV 对齐的补充场景（白名单维护）

> 本节补充 `产品业务流程与组件综合分析.csv` 中湖南公积金贷款在**贷款申请前的白名单维护阶段**，便于端到端理解「客户经理维护白名单 → 客户扫码申请」的整体链路。

### 5.1 阶段 1：白名单维护

- **对应 CSV 行**：营销产品「湖南农信-公积金贷款」、业务阶段「1、白名单维护」  
- **前置条件要点**：
  - 对应阶段的规则包 ID 列表已通过授权配置管理/产品配置等确定；
  - 必要输入数据（客户/进件/授信/用信/押品/外部数据等）可获取或通过征信与外部数据管理组件拉取；
  - 外部数据源与风控引擎自身处于可用状态。
- **进入组件与内部处理逻辑**（来自 CSV）：
  - 前置入口：**办贷端**；
  - 进入组件：**名单管理组件**；
  - 内部调用链：  
    `名单管理组件 → 风控组件（申请前规则包：反欺诈、黑/白名单、产品参数准入）`
- **关键说明（CSV「说明」列）**：
  - 客户经理在办贷端维护**企业白名单**；
  - 在企业名单下增加客户白名单时，调用「申请前规则包」进行业务基础准入，**通过后**保存名单信息；
  - 后续客户在渠道侧发起湖南公积金贷款申请时，贷款申请组件可以基于该名单/预授信结果做提示或前置检查。

> 在本端到端示例中，为突出「贷款申请 → 进件 → 早同步 → R1/R3 → 线下三合一/审批」主线，白名单维护阶段未展开逐步演示；实际产品接入时，可在办贷端配合名单管理组件与风控组件，根据上述 CSV 约定落地。
