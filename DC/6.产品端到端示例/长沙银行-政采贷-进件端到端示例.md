# 长沙银行-政采贷 — 进件端到端示例（授信申请 + 额度项下贷款申请）

[进件管理组件详细设计](../5.详细设计-单组件设计/组件设计-单组件样板-进件管理组件.md)

> 目标：用一条「政采贷对公客户：政采平台白名单 → 强关联人授权准入 → 授信申请进件（早同步、转线下）→ 股东会决议 → 额度激活 → 额度项下贷款申请（用信/借款合同）→ 受托支付常见」的完整链路，串起**渠道端组件 → 进件管理组件（createIntake/updateIntakeStatus）→ 早同步策略/状态机 → 工作流与业务组件**，作为《进件管理组件 — 单组件详细设计》的配套示例。

---

## 一、整体链路鸟瞰

政采贷相对税E贷的关键差异：**政采平台白名单推送**、**强关联人授权**、**授信进件后转线下必选**、以及在授信/额度激活后存在**额度项下贷款申请（用信）**，并可能出现**授信合同 + 借款合同**等多轮签约。

以「对公客户完成政采白名单准入与强关联人授权后发起授信申请，线下办贷完成额度激活，再在额度项下发起贷款申请并签借款合同、提款放款」为主线：

1. **预授信（白名单推送）**：政采平台通过**征信与外部数据管理组件**推送白名单；**预授信组件**消费并落库；风控组件可按「政采白名单」规则校验。  
2. **对公客户注册认证**（必经）：渠道前端 **用户注册组件** → **客户管理组件**、系统管理/消息/人脸/安全；对公必须先注册与基础认证。  
3. **查询产品信息**：**营销方案管理组件** → 产品管理、利率与费用、**预授信额度**、额度管理、合同管理（若存在政采白名单/预授信结果，需要在本步带出可用的预授信额度/额度提示）。  
4. **客户授权准入（独立阶段，强关联人）**：**客户授权准入组件** → 授权配置管理（含关联人类型）→ 合同管理（授权协议展示）/消息 → **签约组件**（授权签署，覆盖 SELF/法人/实控人/主股东/关联企业等）→ **风控组件**（授权准入规则）；授权不齐备则拒绝/不允许进入授信申请。  
5. **客户发起授信申请（授信进件）**：**授信申请组件**（渠道端）填写授信申请信息并提交，调用**进件管理 createIntake（`intakeType=CREDIT_APPLY`）**创建授信进件，状态 `RECEIVED`。  
6. **股东会决议（对公）**：企业负责人通过**股东会决议组件**发起决议签署；先由**征信与外部数据管理组件**（工商/司法）提供股东成员清单，再由**签约组件**下发股东会决议签署任务；合同管理负责归档/展示内容，消息中心用于短信下发。  
7. **系统进件接收（早同步 + 转线下）**：进件管理在 `RECEIVED` 下按早同步策略初始化客户/评级/授信/用信；执行进件准入风控；政采贷授信进件**全部转线下**，进件自动分配至客户经理队列，工作流启动办贷流程。  
8. **客户经理补录尽调信息**：客户管理组件补录客户建档/尽调字段。  
9. **利率/额度调整与额度激活**：授信管理/额度管理组件发起，工作流审批，利率与费用/额度管理组件生效；政采贷通常要求额度激活与审批留痕。  
10. **手工发起单独授信申请审批**：授信管理组件发起授信审批 → 风控组件执行授信阶段风控规则 → 工作流/审批流组件完成审批 → 合同管理组件生成**授信合同**、签约组件生成授信合同签约任务（本阶段仅走授信流程，后续用信/贷款申请在额度项下单独处理）。  
11. **客户在线签授信合同**：签约完成回调进件管理 **updateIntakeStatus（SIGNED）**，进件管理在 SIGNED 下按策略对客户/授信/用信/评级/合同更新或终态确认。  
12. **额度项下贷款申请（用信进件/贷款进件）**：客户在**有效额度项**下发起贷款申请：贷款申请组件 → 用信管理组件（额度项下校验与申请）→ 风控组件 → 合同管理组件生成**借款合同** → 签约组件生成借款签约任务。必要时创建新的 LOAN_APPLY 进件或在既有授信上下文下创建用信进件（以配置与实现为准）。  
13. **客户在线签借款合同**：签约完成回调进件管理 updateIntakeStatus（SIGNED，借款合同对应的进件/用信上下文）。  
14. **提款/放款与受托支付**：放款组件链路完成放款；政采贷**受托支付常见**，由工作流/审批流对放款进行审批；放款完成回调 updateIntakeStatus（DISBURSED）；还款同通用链路。

## 二、关键节点与组件角色

### 2.1 渠道端：强关联人授权准入

- **授权配置管理组件**：按产品 + 客户类型 + 业务阶段（AUTHORIZATION）返回授权文件清单及适用关联人类型集合（法人/实控人/主股东/关联企业等）。  
- **客户授权准入组件**：负责把“强关联人授权”作为独立阶段组织起来；授权不齐备时必须拦截。  
- **签约组件（授权协议任务）**：支持多主体、多任务的授权签署与回写留痕，供后续进件准入与外部数据拉取审计使用。

### 2.2 进件接收：createIntake（CREDIT_APPLY）与“必转线下”

- **授信申请组件 → 进件管理**：创建授信进件 `CREDIT_APPLY`。  
- **早同步**：`RECEIVED` 即初始化客户/评级/授信/用信（见同步策略表）。  
- **路由**：政采贷授信进件通常配置为“无直通”，进件自动分配至客户经理队列并启动工作流。

### 2.3 额度项下贷款申请：用信/借款合同

- **额度项下贷款申请**的关键点是“额度可用额校验 + 借款合同签署 + 放款申请书（如需）”。  
- 该阶段的进件类型在落地实现上可选：  
  - 新建 `LOAN_APPLY` 进件（与授信进件区分）；或  
  - 建立“用信申请/贷款申请”进件并关联既有授信/额度项（以系统枚举与配置为准）。  
 端到端链路上必须体现：**用信管理校验额度**、合同管理生成借款合同、签约组件签借款合同、进件状态回调与同步策略执行。

- **场景视图**：见《产品流程-组件调用场景视图与配置表》**1.5 长沙银行-政采贷**，其中：

## 四、与税E贷示例的对照（小结）

| 维度 | 政采贷 | 税E贷 |
|------|--------|-------|
| 白名单来源 | **政采平台推送**（外部数据管理 → 预授信） | 客户经理维护/外部推送（可选） |
| 授权 | **强关联人授权**（法人/实控人/主股东/关联企业等） | 对公独立授权（相对关联人要求更弱） |
| 进件形态 | 授信进件 + **额度项下贷款申请（用信/借款合同）** | 以授信进件为主，提款用信 |
| 签约轮次 | 常见为授信合同 + 借款合同 + 放款申请书（视产品） | 授信合同 + 放款申请书 |
| 受托支付 | **更常见**，工作流审批更强 | 按配置 |

政采贷本质是“对公授信 + 额度项下用信”的组合产品，端到端示例需要同时覆盖：**强关联人授权 → 授信进件（早同步、转线下）→ 额度激活 → 额度项下贷款申请与借款合同签署**。

## 五、与 CSV 对齐的补充场景（白名单推送、二合一与单独用信）

> 本节补充 `产品业务流程与组件综合分析.csv` 中政采贷特有的若干阶段：政采平台白名单推送、授信进件二合一模式以及单独用信申请与审批，方便端到端理解“政采白名单 → 授信（评级+授信二合一）→ 额度项下贷款 → 单独用信”的整体闭环。

### 5.1 阶段 1：名单信息接收（政采平台白名单）

- **对应 CSV 行**：营销产品「长沙银行-政采贷」、业务阶段「1、名单信息接收」  
- **前置条件要点**：
  - 对应阶段的规则包 ID 列表已通过授权配置管理/产品配置等确定；
  - 必要输入数据（客户/进件/授信/用信/押品/外部数据等）可获取或通过征信与外部数据管理组件拉取；
  - 外部数据源与风控引擎处于可用状态。
- **进入组件与内部处理逻辑**：
  - 前置入口：**政采平台/办贷端**；
  - 进入组件：**名单管理组件**；
  - 内部调用链：`名单管理组件`（消费政采平台推送的白名单数据，必要时调用风控组件执行申请前规则包）。
- **关键说明（CSV「说明」列）**：
  - 「1、政采平台推送白名单（申请前规则包）」：即政采平台作为外部来源，通过征信与外部数据管理组件将白名单推入名单管理组件，按配置执行申请前规则后入库；
  - 该阶段通常发生在单笔授信申请之前，用作后续查询产品信息与授权准入、授信申请的前置参考与约束。

### 5.2 阶段 6：系统进件接收（评级+授信二合一，不生成用信）

- **对应 CSV 行**：业务阶段「6、系统进件接收」（授信申请组件 → 进件管理组件）  
- **前置条件要点**：
  - 授信进件请求完整（产品、进件类型、客户标识、基本业务字段）；
  - 同步策略表中已为政采贷配置好早同步策略，进件状态机定义完整；
  - 授权配置中，如要求「先授权」，在进件创建或进入关键状态前可校验授权记录。
- **进入组件与内部处理逻辑**：
  - 前置入口：**授信申请组件（渠道端）**；
  - 进入组件：**进件管理组件**；
  - 内部调用链（与 CSV 一致）：  
    `进件管理组件 → 授权配置管理组件（查询进件阶段规则包 ID）→ 客户管理组件（创建）→ 风控组件（准入，规则包由授权配置提供）→ 评级管理组件 → 风控组件（额度）→ 授信管理组件（初始化）→ 进件管理组件（分配）`
- **关键说明（来自 CSV「说明」列）**：
  - **早同步**：进件环节至落地进件信息，同步评级与授信信息；  
  - 「评级+授信二合一模式同步，不生成用信信息」：政采贷在此阶段仅初始化评级与授信，**不生成用信记录**，与后续「额度项下贷款申请」阶段的单独用信同步形成区分；
  - 模型审批不通过直接拒绝；通过则全部转线下客户经理人工处理。

### 5.3 阶段 15～16：单独用信申请与审批

- **对应 CSV 行**：业务阶段「15～16、单独用信申请与审批」  
- **前置条件要点**：
  - 已存在有效授信记录与额度（前文额度激活阶段完成）；
  - 用信管理组件已记录可用额度项及相关合同/订单信息。
- **进入组件与内部处理逻辑**：
  - 前置入口：**办贷端-用信管理组件**；
  - 进入组件：**用信管理组件（办贷端）** + **工作流/审批流组件（办贷端）**；
  - 内部调用链：  
    `用信管理组件（申报）→ 工作流/审批流组件（用信审批流程）→ 合同管理组件（借款合同/用信协议生成）→ 签约组件（借款合同签署任务）`
- **关键说明（来自 CSV「说明」列）**：
  - 「单独用信申报审批，生成借款合同与签约任务」：即在已有授信和额度基础上，发起**单独用信申请**，经审批后生成借款合同与签约任务；
  - 与额度项下贷款申请一起，完成「授信 + 单独用信」的完整用信闭环；
  - 本端到端示例主线中关于「额度项下贷款申请（用信/借款合同）」与「借款合同签署」的描述，即可视为与该 CSV 阶段共同构成完整的用信场景。
