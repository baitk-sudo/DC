# 进件生命周期与同步策略关系分析

> 本文尝试用「一张时间线 + 一张配置表」的视角，解释进件状态（RECEIVED / APPROVED / SIGNED / DISBURSED 等）与「同步策略表」之间的关系，便于后续在评审和落地时对齐口径。

---

## 一、进件生命周期主干：一条时间线

从统一语言与《进件中心-业务数据同步策略设计》的定义出发，一笔进件（Intake）的**主生命周期**可以简化为一条时间线：

- **RECEIVED**：进件已被接收并持久化，通常为入口状态。  
  - 渠道申请（授信/贷款/提额）成功后，调用进件管理组件生成一条进件记录。  
  - 前置调用条件满足（数据完整、产品与进件配置就绪、先授权等）即应当落库，不以风控结果为前提。
- **UNDER_REVIEW**：审批/调查中。  
  - 处于风控准入、线下三合一/审批、补件等阶段，进件在办贷流程中“被处理”。
- **APPROVED**：审批通过。  
  - 授信/用信审批已通过，业务在“可以做”的状态，可生成合同草稿或待签任务。
- **SIGNED**：合同签约完成。  
  - 必要合同/授权协议已签署完成，业务数据通常视为“正式生效”；晚同步产品多在此一次性把客户/授信/用信/额度/合同等落库。
- **DISBURSED**：已放款/用信。  
  - 资金已划付或用信实际发生，进入贷后与还款阶段。
- **REJECTED**：已拒绝。  
  - 任一环节给出拒绝/终止结论，进件流程结束，但进件记录保留用于审计与追溯。

> **关键点**：这条时间线是**跨产品、跨机构统一的“里程碑语义”**，不关心每个产品在 RECEIVED 之前是先预授信、先授权，还是直接申请，只描述“进件本身现在处在什么阶段”。

## 二、同步策略表：在时间线上“每一格要做什么”

进件状态只是时间线节点，要让不同产品、不同机构在这些节点上做出**差异化的数据落库行为**，需要一张**同步策略表**来“填格子”。

### 2.1 概念模型

从《进件中心-业务数据同步策略设计.md》中抽象，同步策略表可以理解为：

- 维度：  
  - **产品 / 机构**：不同产品、不同机构可以有不同策略；  
  - **进件类型**：授信申请 / 贷款申请 / 提额等；  
  - **进件状态**：RECEIVED / APPROVED / SIGNED / DISBURSED / …；  
  - **业务模块**：客户 / 评级 / 授信 / 用信 / 额度 / 合同 / 报表等；  
  - **同步动作**：初始化 / 更新 / 终态确认 / 无动作。

- 每一行大致类似：  
  - （产品 A，进件类型=LOAN_APPLY，状态=RECEIVED，模块=授信，动作=初始化）  
  - （产品 B，进件类型=LOAN_APPLY，状态=SIGNED，模块=授信，动作=初始化+终态确认）

这相当于为每个「产品 × 进件类型 × 状态」配置了一个**“矩阵行”**：在这一个时点，要不要对每个业务模块做什么事情。

### 2.2 早同步 vs 晚同步只是“填表方式不同”

- **晚同步（如湖北模式）**：  
  - 在 `RECEIVED / APPROVED` 行，对客户/授信/用信/额度等模块多配置为「无动作」；  
  - 在 `SIGNED` 行，一次性将客户/授信/用信/额度/合同等模块配置为「初始化 + 终态确认」；  
  - 效果：进件在签约前只是“申请与审批载体”，真正的业务记录在签约时一次性落地。

- **早同步（如湖南/长沙模式）**：  
  - 在 `RECEIVED` 行，就对客户/评级/授信/用信/额度等模块配置为「初始化」；  
  - 在 `APPROVED` / `SIGNED` 行，对上述模块配置为「更新/终态确认」；  
  - 效果：业务模块很早就能看到“草稿/初始”数据，方便贷中、贷后、风险与营销联动，后续状态只是在这些记录上更新与确认。

> **观察**：早同步/晚同步的差异完全可以通过“在同步策略表上填不同行为”体现，而无需在进件管理组件中写死一堆 if-else。

## 三、进件管理组件在状态迁移时如何使用同步策略

结合《进件管理组件 — 单组件详细设计》和《进件中心-业务数据同步策略设计》，进件管理组件在进件生命周期的每个关键节点，主要做两类工作：

1. **维护进件状态机**  
   - 在前置条件满足的前提下，先**持久化进件记录**（通常初始为 `RECEIVED`），后续根据风控/审批/签约/放款结果将状态从 `RECEIVED` → `APPROVED` → `SIGNED` → `DISBURSED` 或 `REJECTED`。  
   - 风控/审批拒绝只改变进件状态为 `REJECTED`，**不删除进件记录**，保证审计与追溯。

2. **在每次状态变更时执行同步策略**  
   - 状态发生变化（如从 NONE→RECEIVED、RECEIVED→APPROVED、APPROVED→SIGNED 等）时：  
     1. 更新进件单状态；  
     2. 发布对应领域事件（如 `IntakeReceived`、`IntakeApproved`、`IntakeSigned`、`IntakeDisbursed`）；  
     3. 根据「产品 + 进件类型 + 当前状态」查询同步策略表，拿到当前状态下需要触发的**业务模块及同步动作列表**；  
     4. 按列表依次调用各业务组件的 `upsertByIntake` 类接口（按进件号幂等写入/更新）；  
     5. 记录同步结果，失败时支持重试或补偿。

从这个角度看：

- **进件生命周期主干**提供了“事件时间线”；  
- **同步策略表**规定了在时间线上的每个节点对业务模块“做或不做、做什么”；  
- **进件管理组件**扮演“状态机 + 调度器”的角色：不关心具体产品逻辑，而是**根据状态变化 + 配置表**去驱动各业务模块的落库与更新。

---

## 四、小结：为什么要把两者分开设计

1. **复用一套进件实现支撑多机构差异**  
   - 通过**统一的生命周期定义** + **可配置的同步策略表**，可以用一套进件实现支撑湖北的“晚同步”、湖南/长沙的“早同步”，以及未来的新模式，而不用为每个机构分叉代码。

2. **职责清晰，组件边界干净**  
   - 进件管理组件只聚焦于“进件状态机 + 调度同步”，不直接持有各业务模块的业务规则；  
   - 客户/授信/用信/额度/合同等模块只需要提供按进件号幂等的接口，具体何时被调用由同步策略表驱动。

3. **利于审计与排查**  
   - 所有进件都以 `RECEIVED` 为起点，后续状态与同步行为可从进件记录与同步日志中完整追溯；  
   - 同步策略表提供了“某产品在某状态下为什么会（或不会）写某个模块”的清晰来源。

> 建议：后续在评审具体产品接入方案时，可以先在**时间线上画出该产品的进件状态流转**，再在同步策略表上“填格子”，确保“时间线语义”与“同步行为”都一目了然。
