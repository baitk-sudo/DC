## 二、做得好的地方（保持即可）

| 方面 | 说明 |
|------|------|
| **建议 vs 结果产物分离** | 「建议与分析/」单独成夹，且《文档清单-建议类与结果产物区分》明确区分用途，阅读时不会把分析稿当正式设计。 |
| **横切设计集中** | 跨段、跨组件的契约（同步策略、Context Map、词汇表、风控总览、场景视图与配置表等）统一在「横切设计/」，和「按段」的组件设计形成清晰分工。 |
| **产品流程 vs 产品业务流程** | 「产品流程/」放分析文档、「产品业务流程/」放原始数据，概念上区分清楚；README 与文档清单中均有说明。 |
| **各子目录有 README** | 产品流程/、建议与分析/ 有 README 说明本目录用途与入口，横切设计虽无 README，但根目录 README 与 sidebar 已列出其下文件。 |
| **命名一致** | 组件设计类以「组件设计-」前缀、段落以「第N段」、横切以主题命名，便于搜索与过滤。 |

## 三、可改进点与建议

### 3.1 根目录文件过多，组件设计类占满一屏

**现状**：根目录下有 14+ 个「组件设计-」开头的 md 文件（按组件索引、调用关系总览、单组件样板×2、第1～11 段），加上 README、文档清单、发布说明、未在网页导航中发布的文档、sidebar/navbar、CSV 等，新人打开仓库容易眼花。

**建议（可选）**：

- **方案 A（保守）**：保持现状，在 **README 最上方**增加一节「根目录文件速览」，用表格列出：根目录下各 md 的用途、首次阅读推荐顺序（例如：README → 总览-核心概念 → 按组件索引/调用关系总览 → 第 4/6/7 段）。不移动文件，仅靠说明降低认知负担。
- **方案 B（收拢）**：新建 **「组件设计/」** 目录，将「组件设计-按组件索引」「组件设计-组件调用关系总览」「组件设计-单组件样板-*」「组件设计-第*段-*」全部移入该目录；根目录只保留 README、文档清单、发布说明、未在网页导航中发布的文档、导航相关（sidebar/navbar、index*.html）、以及「组件说明/工作计划」等少量文件。  
  - 优点：根目录立刻变清爽，按「组件设计」一个入口即可进入所有段落与单组件样板。  
  - 代价：所有指向上述文件的**相对链接**（README、sidebar、文档清单、各段之间的互相引用、按组件索引中的链接）需批量更新路径；Docsify 等若按相对路径配置也需检查。

若采用方案 B，建议在「文档清单」与 README 中明确写清：**组件设计类文档统一位于 `组件设计/`**，并在迁移后做一次链接与路径检查（可参考《文档结构调整后-链接与路径检查报告》的检查项）。

---

### 3.2 「产品流程」相关文档分布在两处

**现状**：

- **产品流程/**：产品流程-组件被调用关系梳理、产品流程-业务步骤级组件分析。
- **横切设计/**：产品流程-组件调用场景视图与配置表。

「产品流程」三个字在两处出现，容易让人误以为「产品流程的文档」都在产品流程/ 下。

**建议**：

- 在 **产品流程/README.md** 中已有一句：「按产品看端到端阶段→主调→被调链见横切设计中的 产品流程-组件调用场景视图与配置表」，这点很好。
- 可在 **根目录 README** 的「产品流程分析」小节中补一句：  
  「按产品×阶段的**场景视图与配置表**在 `横切设计/` 下，与产品流程/ 下的被调用关系梳理、步骤级分析配套使用。」  
  这样读者会自然接受「产品流程视角 = 产品流程/ + 横切设计里那一份场景视图」。

不建议把「产品流程-组件调用场景视图与配置表」从横切设计挪到产品流程/，因其是跨产品、跨阶段的**配置与契约**，归属横切更符合现有「横切设计思路文件清单」的划分。

---

### 3.3 单组件样板与段落设计混在根目录（或未来的「组件设计/」）

**现状**：单组件样板（进件管理、客户授权准入）与「组件设计-第N段」并列；侧栏中「单组件样板」只列了进件管理，**客户授权准入组件的单组件样板未在 sidebar-report 中展示**。

**建议**：

- **短期**：在 **sidebar-report.md（及 sidebar-impl.md 若适用）** 的「组件设计」下，为单组件样板增加一条：「[单组件样板-客户授权准入组件](组件设计-单组件样板-客户授权准入组件.md)」，保证导航完整。
- **长期**：若采用 3.1 方案 B，单组件样板与段落设计会一起进入「组件设计/」，可在该目录下再分子目录（例如 `组件设计/段落设计/`、`组件设计/单组件样板/`），或仍用扁平结构但靠文件名区分（当前已能区分）。子目录会带来更清晰的浏览体验，但同样需要更新所有链接。

---

### 3.4 数据/清单类文件在根目录

**现状**：根目录有「组件说明.csv / 组件说明_v2.csv」「工作计划.csv」；「产品业务流程/」下是各产品流程的 CSV/xlsx。

**建议**：保持现状即可。组件说明与工作计划属于**项目级清单/计划**，放根目录便于一眼看到；产品业务流程数据已集中在「产品业务流程/」，无需再动。若未来 CSV 种类继续增多，可考虑根目录只保留「组件说明_v2.csv」等 1～2 个主清单，其余放入「数据/」或「产品业务流程/」等，目前不必调整。

---

### 3.5 建议与分析目录下的「Untitled」

**现状**：建议与分析/ 下存在名为 **Untitled** 的文件（无扩展名），疑似临时或空文件。

**建议**：若确认为无用，可**删除**或重命名为「Untitled-待分类」并加一句说明；若为占位，建议改为有意义名称（如「待整理分析.md」），避免在清单与列表中造成干扰。

---

### 3.6 横切设计目录缺少 README

**现状**：产品流程/、建议与分析/ 均有 README；横切设计/ 无 README，仅靠根目录 README 与 sidebar 罗列其下 9 个文件。

**建议**：若希望「每个有内容的目录都有一页说明」，可为 **横切设计/** 增加简短 **README.md**，内容包含：本目录用途（跨组件/阶段/产品的设计契约与配置）、文件列表（或直接列 9 个文件名与一句话说明）、与「组件设计段落」「产品流程」的配合关系。非必须，属于锦上添花。

## 四、小结表（是否调整一览）

| 项 | 现状 | 建议 | 优先级 |
|----|------|------|--------|
| 根目录组件设计文件过多 | 14+ 个 md 在根目录 | 可选：方案 A 增加「根目录速览」；或方案 B 建「组件设计/」收拢并更新链接 | 中 |
| 产品流程文档分布两处 | 产品流程/ + 横切设计/ 各有一类「产品流程」文档 | README 中补一句说明场景视图在横切设计，与产品流程/ 配套 | 低 |
| 单组件样板-客户授权准入未进侧栏 | sidebar 仅列进件管理样板 | 在 sidebar-report（及 impl）中增加客户授权准入单组件样板链接 | 高 |
| 建议与分析下的 Untitled | 无名/临时文件 | 删除或重命名/加说明 | 低 |
| 横切设计无 README | 无 | 可选：新增横切设计/README.md 简要说明目录用途与文件列表 | 低 |
| 数据/清单文件位置 | 组件说明、工作计划在根目录；产品数据在产品业务流程/ | 保持 | — |

## 五、结论

- **整体上，DC 的文件夹与文件分类是合理的**：建议与结果产物分离、横切设计集中、产品流程与产品业务流程区分明确、命名一致，阅读条理已经较好。
- **优先建议**：在侧栏中补全「单组件样板-客户授权准入组件」入口；清理或规范「建议与分析/Untitled」。
- **若希望根目录更清爽**：可评估「组件设计/」收拢方案（方案 B），并在执行后做一次全量链接与路径检查。

如需对某一条做具体修改（例如只改 sidebar、只加 README），可指定项号与方案，再按需落地。
