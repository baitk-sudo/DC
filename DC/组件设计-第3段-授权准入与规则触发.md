## 组件设计分析 — 第 3 段：授权准入与规则触发

按**业务办理过程顺序**进行组件分析，本段覆盖流程主线中的**客户授权准入**环节，重点是：

- 客户对征信、公积金、政采等外部数据的使用授权；
- 在不同产品、不同业务阶段（预授信 / 授信申请 / 贷款申请等）下，授权与风控规则的配置与触发；
确保该段内组件对已分析的全部产品支撑不遗漏，并支持未来多产品扩展。

**本段涉及的组件**（办贷管理端 v2 + 渠道端）：

- 客户授权准入组件（渠道）
- 授权配置管理组件（办贷）
- 风控组件（办贷，本段侧重授权准入阶段的规则执行）
- 合同（协议）模版管理组件（办贷）
- 签约组件（渠道 + 办贷）
- 消息中心组件 / 通用安全能力（短信验证码、人脸识别）

**已对照产品**：湖北农信-公积金贷款（线上/线上转线下）、湖南农信-公积金贷款、长沙银行-税E贷、长沙银行-政采贷。

## 一、客户授权准入组件（渠道端）

### 1.1 职责与边界

| 项目 | 说明 |
|------|------|
| **职责** | 为客户提供授权准入的交互入口和流程编排：在产品详情页或申请流程中，展示本阶段所需授权文件（征信、公积金、政采数据使用授权等），引导客户阅读授权协议、完成短信/人脸等认证并签署授权协议，然后发起授权准入规则校验并展示结果。 |
| **边界** | 不自行决定“需要哪些授权文件/执行哪些规则”，而是调用**授权配置管理组件**获取配置；不执行风控规则，仅负责调用**风控组件**或智能决策接口并展示结果；不负责授权协议模板内容，仅通过**合同（协议）模版管理组件**加载。 |

### 1.2 多产品 / 多模式支持点

- **独立授权阶段 vs 申请内授权**：
  - 独立阶段：如长沙银行税E贷、政采贷在授信申请前有单独“客户授权准入”阶段。
  - 申请内授权：如部分公积金贷款，在贷款/授信申请页面内嵌授权流程。
  - 授权准入组件需支持这两种接入方式，通过配置决定在产品详情页/申请页是否展示独立的“授权准入”入口。
- **多授权文件组合**：
  - 支持在同一授权流程中组合多种授权：征信查询授权、公积金查询授权、政采数据授权等。
  - 授权列表由授权配置管理组件按产品+业务阶段返回，授权准入组件仅负责展示与签署。
- **多因子认证**：
  - 可按产品和授权敏感度配置是否需要人脸识别、短信验证码、交易密码等多因子，具体策略由系统管理组件提供，授权准入组件负责调用通用安全能力。

### 1.3 关键接口与上下游关系

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 发起授权准入 | 在产品详情页或申请页中点击“授权”后，初始化授权准入流程 | 上游：渠道页面；下游：授权配置管理组件 |
| 查询需授权文件列表 | 按产品+业务阶段查询需要的授权文件和需执行的规则包 | 下游：授权配置管理组件；上游：本组件 |
| 加载授权协议模板 | 按授权文件编码加载协议模板供客户阅读 | 下游：合同（协议）模版管理组件；上游：本组件 |
| 短信验证码 / 人脸识别 | 在授权签署前调用短信、人脸等安全能力，完成身份核验 | 下游：消息中心组件、人脸识别服务、安全组件；上游：本组件 |
| 签订授权协议 | 通过签约组件完成授权协议的电子签署，并记录签署结果 | 下游：签约组件（及合同管理组件存档）；上游：本组件 |
| 提交授权准入申请 | 携带客户、产品及授权信息，调用风控组件执行授权准入规则 | 下游：风控组件；上游：本组件 |
| 返回授权准入结果 | 展示授权准入通过/拒绝结果，并将结果返回调用方（如产品详情页、授信/贷款申请流程） | 下游：无；上游：调用方 |

### 1.4 对已分析产品的支撑检查

| 产品 | 本段内相关步骤 | 支撑情况 | 是否遗漏 |
|------|----------------|----------|----------|
| 湖北农信-公积金 | 授信/贷款申请页面内“查看需要授权的文件（征信、公积金授权）”、签订授权协议 | 授权准入流程嵌入申请页，授权准入组件可在“贷款申请”阶段调用授权配置并发起授权签署与规则执行 | 无遗漏 |
| 湖南农信-公积金 | 同湖北，授权在贷款申请阶段内 | 同上 | 无遗漏 |
| 长沙银行-税E贷 | 独立“客户授权准入”阶段：产品详情页点击授权申请，查看授权文件、签约、提交授权准入申请、返回结果 | 授权准入组件支持独立阶段，并调用授权配置、签约、风控；与现设计吻合 | 无遗漏 |
| 长沙银行-政采贷 | 同税E贷，独立授权准入阶段 | 同上 | 无遗漏 |

**小结**：授权准入组件需支持「嵌入申请页」与「独立阶段」两种调用模式，通过授权配置管理组件统一获取授权与规则配置，调用签约/安全/风控能力完成授权与准入校验。

## 二、授权配置管理组件（办贷）

### 2.1 职责与边界

| 项目 | 说明 |
|------|------|
| **职责** | 以「产品 + 客户类型 + 业务阶段」为主维度，统一配置：需要客户签署的授权文件（征信、公积金、政采等授权协议）以及在该阶段需要执行的风控规则包（如授权准入规则、预授信规则包等），并提供查询服务给授权准入、进件、签约等环节使用。 |
| **边界** | 不执行授权流程本身（签署、认证等），由客户授权准入组件与签约组件负责；不执行规则，仅返回「需要执行哪些规则包」给风控组件；不存储规则执行结果，仅存配置。 |

### 2.2 多产品 / 多模式支持点

- **产品 + 客户类型 + 业务阶段维度配置**：
  - 客户类型示例：`PERSONAL`（个人）、`CORPORATE`（对公）、`GROUP`（集团客户）等，可扩展；
  - 业务阶段示例：`CUSTOMER_PRE_CREDIT`（客户预授信）、`PRODUCT_PRE_CREDIT`（产品预授信）、`AUTHORIZATION`（独立授权准入）、`LOAN_APPLY`（贷款申请）、`CREDIT_APPLY`（授信申请）、`SIGNING`（签约前授权）等。
  - 每个组合可配置：需签署的授权文件列表 + 需执行的规则包列表。
- **关联人类型（适用对象）配置**：
  - 一笔业务往往不仅需要**客户本人**授权，还需要对若干**关联人**（如配偶、法定代表人、实际控制人、保证人等）进行授权配置；
  - 关联人类型示例（可按机构扩展）：`SELF`（客户本人）、`SPOUSE`（配偶）、`LEGAL_REP`（法定代表人）、`LEGAL_REP_SPOUSE`（法定代表人配偶）、`ACTUAL_CONTROLLER`（实际控制人）、`ACTUAL_CONTROLLER_SPOUSE`（实际控制人配偶）、`GUARANTOR`（保证人）、`GUARANTOR_SPOUSE`（保证人配偶）、`CO_BORROWER`（共同借款人）、`MAJOR_SHAREHOLDER`（主要股东）、`RELATED_COMPANY`（关联企业）、`RELATED_COMPANY_SHAREHOLDER`（关联企业股东）等；
  - 每条授权文件配置需注明**适用对象集合**（客户本人及/或上述一个或多个关联人类型），授权准入组件据此确定在本产品、本阶段需要为哪些主体展示授权协议并检查授权是否齐备。
- **授权检查模式配置**：
   - 对于不同产品/阶段，可配置**授权检查模式**字段（例如：`ENFORCE` 强制校验、`REMIND` 仅提示、`NONE` 仅记录不拦截），用于控制在授权缺失或不完整时的处理方式；
   - 客户授权准入组件在执行前，根据该字段决定：是将缺失授权视为**硬性拦截条件**（不允许继续申请/进件），还是只在前端做**高亮提示/确认**后继续流程，或仅用于后续风控/审计记录。
- **多授权文件类型**：
  - 支持配置多种授权类型：征信、银行流水、公积金、税务、政采等。
  - 授权文件与合同模板通过文件编码与合同（协议）模版管理组件关联。
- **规则包复用**：
  - 规则包由风控组件维护，授权配置管理仅引用规则包 ID，实现多个产品/阶段共享同一规则包（如统一授权准入规则）。

### 2.3 关键接口与上下游关系

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 查询授权配置 | **按产品 + 客户类型 + 业务阶段** 返回本场景下的授权要求：1）需签署的授权文件列表（含适用对象/关联人类型）；2）需执行的风控规则包**编号列表**（rule_package_ids），供调用方直接传给风控组件执行 | 下游：无；上游：客户授权准入组件、进件管理组件、签约组件等 |
| 维护授权文件配置 | 在办贷端维护不同产品、不同客户类型、不同阶段的授权文件（授权协议编码、名称、是否必签、适用对象/关联人类型集合等） | 下游：合同（协议）模版管理组件（模板内容）；上游：业务配置人员 |
| 维护规则包绑定关系 | 为每个产品 + 客户类型 + 阶段绑定需要执行的风控规则包列表（仅存**规则包编号**，规则内容在风控组件域维护），支持同一规则包在多产品/阶段间复用 | 下游：风控组件消费规则包 ID；上游：业务/风控配置人员 |

### 2.4 对已分析产品的支撑检查

| 产品 | 本段内相关步骤 | 支撑情况 |
|------|----------------|----------|
| 湖北/湖南 公积金 | 在贷款申请阶段查看征信、公积金授权文件 | 授权配置管理按产品 + `LOAN_APPLY` 阶段配置征信与公积金授权文件及规则包，授权准入组件查询使用 |
| 长沙银行-税E贷 | 独立授权准入阶段需配置授权文件与规则包 | 授权配置管理按产品 + `AUTHORIZATION` 阶段配置，客户授权准入组件按该阶段查询 |
| 长沙银行-政采贷 | 同上 | 同上 |

**小结**：授权配置管理组件是“授权与规则触发”的配置中枢，新产品或新阶段只需在此配置，而无需修改授权准入或风控代码。

## 三、风控组件（本段：授权准入阶段）

> 风控组件在整个流程中职责很广，这里仅聚焦在“授权准入阶段”的职责。

### 3.1 本段职责与边界

| 项目 | 说明 |
|------|------|
| **本段职责** | 根据授权配置管理组件返回的**规则包编号列表（rule_package_ids）**，在客户完成授权签署后执行相应的授权准入规则（如可否拉取征信、公积金、政采数据，是否满足准入前置条件），并返回通过/拒绝/转人工等结果。 |
| **边界** | 不决定需要哪些授权，由授权配置管理组件负责；不直接对接外部数据，外部数据由征信与外部数据管理组件提供；不改变授权签署状态，仅输出授权准入的风控结论。 |

### 3.2 多产品 / 多模式支持点

- **规则包配置化**：支持为不同产品/阶段配置不同的授权准入规则包（例如针对公积金产品、政采产品的特定授权前置规则）。
- **结果类型**：支持多级结果（通过 / 拒绝 / 转人工 / 补充资料），授权准入组件和进件管理组件可按结果决定下一步流程。

### 3.3 关键接口与上下游关系（本段）

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 执行授权准入规则 | 接收产品、阶段、客户、授权信息及规则包 ID，执行授权准入规则并返回结果 | 下游：征信与外部数据管理组件（如需拉取外部数据）；上游：客户授权准入组件、进件管理组件 |

**小结**：风控组件在本段主要作为“授权准入规则执行引擎”，规则包来源于授权配置管理，数据来源于征信与外部数据管理，结果回传给授权准入或进件。

## 四、合同（协议）模版管理组件 & 签约组件 & 消息中心组件

> 这些组件在第 1、2 段已有基础说明，这里只补充与授权准入密切相关的能力。

### 4.1 合同（协议）模版管理组件

- 职责：维护各类授权协议的模板（征信授权、公积金授权、政采授权等），支持按协议编码、版本查询；提供给授权准入/签约组件加载展示。
- 对本段的支持：授权配置管理中配置的授权文件编码，通过本组件获取具体模板内容。

### 4.2 签约组件（授权场景，公共能力）

- **定位**：签约组件为系统**公共能力组件**，统一支撑授权协议签订与合同签订；本段仅涉及其中「授权协议」签署场景。
- **职责**：承载客户对授权协议的电子签署流程（查看协议、短信/人脸验证、签署、状态回写）；签约入口可为**面客渠道**（客户在 APP/Web 自助签）或**办贷端 PAD 现场签约**（客户经理持 PAD 到场、客户在 PAD 上签授权协议），两种入口共用同一套任务与回写逻辑。
- 对本段的支持：
  - 支持“授权协议”这一类签约任务类型，与授信合同、借款合同区分；
  - 在授权准入阶段可单独生成“授权签署任务”，也可以作为贷款/授信申请内嵌的一部分；
  - 授权协议签署可由面客渠道或办贷 PAD 发起，签署结果统一回写到合同管理组件或授权记录表，以便后续征信/外部数据查询留痕。

### 4.3 消息中心组件 / 通用安全能力

- 职责：发送并验证短信验证码，为授权准入中的关键步骤提供安全校验能力；配合人脸识别、安全组件实现多因子认证。
- 对本段的支持：授权准入组件在展示授权协议前/签署时调用短信、人脸等能力，具体策略由系统管理组件配置。

## 五、本段组件协作关系小结

```text
[渠道]                                  [办贷 v2 / 平台]
   |                                            |
   |  产品详情/申请页点击“授权”                 |
   |--> 客户授权准入组件 ---------------------> 授权配置管理组件
   |                                (查询产品+阶段的授权文件和规则包)
   |
   |  加载授权协议模板                         |
   |--> 合同（协议）模版管理组件              |
   |
   |  短信/人脸校验                            |
   |--> 消息中心组件 / 人脸识别 / 安全组件     |
   |
   |  签订授权协议                             |
   |--> 签约组件  ---------------------------> 合同管理/授权记录
   |
   |  提交授权准入申请                         |
   |--> 风控组件（执行授权准入规则） <------ 授权配置管理组件(规则包)
   |
   |  返回授权准入结果（通过/拒绝/转人工）     |
   |<-- 客户授权准入组件
```

## 六、与前两段的衔接

- 与**第 1 段（营销获客与预授信）**：在预授信阶段，也可能需要征信/公积金授权；授权配置管理组件可为 `CUSTOMER_PRE_CREDIT`/`PRODUCT_PRE_CREDIT` 阶段配置授权文件与规则包，预授信组件调用授权准入流程。
- 与**第 2 段（客户注册与认证）**：授权准入中用到的短信、人脸、密码等认证手段，均为第 2 段中定义的通用安全能力与系统管理策略的复用。
- 与后续**进件与同步段**：进件接收时可校验授权是否有效（例如授权是否在有效期、授权是否覆盖本次查询所需数据源），需要时触发补授权流程，仍由本段组件协作完成。

本文件为「按业务办理过程顺序」组件分析的第 3 段；第 4 段可继续分析**进件接收与风控准入及同步策略在各产品中的落地**。
