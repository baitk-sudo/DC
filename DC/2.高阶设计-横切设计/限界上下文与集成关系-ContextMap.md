# 限界上下文与集成关系（Context Map）

> 目标：从“模块清单”上升一层，给出**几个相对独立的业务子域 + 它们之间的集成关系**，方便在评审架构、划分团队与落系统边界时有统一参照。

---

## 1. 总览：推荐的上下文划分（按业务办理流程顺序）

> 下文各上下文的排列顺序与**业务办理流程**一致：获客与身份 → 产品与营销 → 授权与准入配置 → 外部数据与风控 → 进件与流程 → 授信/签约/放款还款 → 贷后 → 报表与批处理 → 通用能力，便于沿主线阅读。

- **客户与身份上下文**：用户注册、客户管理（临时/正式、对私/对公、多源建档）
- **产品与营销上下文**：产品管理、营销方案管理、预授信（渠道产品页与申请页展示、申请/进件要素）
- **授权与准入配置上下文**：授权配置管理（阶段×产品×授权文件×规则包×检查模式，第3段配置来源）
- **外部数据上下文**：征信与外部数据管理、外部平台推送
- **风控上下文**：规则包执行、结果、标签
- **进件与流程上下文**：进件管理、工作流/审批流 + 同步策略
- **授信/额度/用信上下文**：授信管理、额度管理、用信管理、评级管理、利率与费用
- **签约与合同上下文**：合同实例、合同模板、签约任务与签署结果（含股东会决议等）
- **支付与账务集成上下文**：放款组件、还款组件，与核心/账务系统的接口（还款计息依赖利率与费用）
- **贷后与预警上下文**：贷后管理/预警、押品、贷后规则与工单
- **报表与监管上下文**：监管报送与报表
- **批处理与调度上下文**：批处理调度与批量任务
- **通用安全与消息上下文**：系统管理（认证策略）、安全组件、消息中心、人脸识别

下面按**业务办理流程顺序**逐一说明各上下文的职责、主要数据、对外能力及与其他上下文的集成方式。

## 2. 客户与身份上下文

**核心组件**：用户注册组件（渠道）、客户管理组件（办贷）

- **职责**
  - 用户注册：渠道侧账号注册与登录、对公/对私与第三方对接、显式/自动注册、认证流程编排；与办贷端客户管理对接，驱动临时客户创建与升级为正式客户。
  - 客户管理：维护客户主数据（身份、联系信息、标签、临时/正式、对私/对公）、多源建档（注册、进件同步、外部推送、客户经理补录）、黑名单等；不负责授信/额度/合同等业务数据。
- **持有的数据（主责任）**
  - 客户主数据、客户标签、临时/正式状态、多源标识（external_source + external_customer_id）。
  - 注册与认证结果由系统管理组件记录，本上下文以客户标识为纽带。
- **对外能力**
  - 客户创建/更新/查询、临时客户升级为正式客户。
  - 按客户维度提供基础信息与标签，供风控、授信、进件同步、贷后等消费。
- **与其他上下文的集成方式**
  - **进件与流程上下文**：在 RECEIVED/SIGNED 等状态按同步策略调用本上下文的 upsertByIntake 或等效接口，写入或更新客户。
  - **产品与营销上下文**：营销方案、预授信、授信/贷款申请前依赖客户标识与标签。
  - **通用安全与消息上下文**：认证策略与验证能力由系统管理/安全/消息/人脸提供，本上下文负责流程编排与客户落地。

> 设计重点：客户是贯穿全流程的主数据；**不在此处维护认证策略**，只承载“谁是谁”的建档与标识。

## 3. 产品与营销上下文

**核心组件**：产品管理组件、营销方案管理组件、预授信组件

- **职责**
  - 产品管理：可售产品定义、产品参数（期限、担保、利率结构等）、申请要素与校验规则、机构/客户类型等配置；为进件、授信、用信、合同、营销等提供产品主数据。
  - 营销方案管理：营销产品与可售产品映射、对客展示、申请页/进件查看要素配置、营销溯源（二维码/链接）；为渠道提供产品列表与详情、申请入口，支撑渠道业务办理时产品页面的显示。
  - 预授信：产品预授信（批处理/客户经理代客/外部推送白名单）与客户预授信（客户主动申请）；调用风控执行预授信规则，结果落额度或名单；不直接对接外部系统，白名单由外部数据上下文接入后本组件消费。
- **持有的数据（主责任）**
  - 产品主数据与参数、营销产品与可售产品映射、申请/进件要素配置。
  - 预授信记录（产品预授信/客户预授信、名单与额度）。
- **对外能力**
  - 产品/营销产品查询、申请页要素与进件查看要素、预授信试算与结果查询。
  - 预授信名单与额度写入额度管理（或通过进件同步策略在后续状态落库）。
- **与其他上下文的集成方式**
  - **进件与流程上下文**：进件接收前依赖产品与申请要素；授信/贷款申请、提额入口在本上下文与进件之间衔接。
  - **风控上下文**：预授信阶段由本上下文调用风控执行规则包；规则包 ID 来自产品配置等。
  - **外部数据上下文**：预授信白名单可来自外部推送，经外部数据上下文接入后由本上下文消费。
  - **授权与准入配置上下文**：申请环节所需授权书与风控模型选择由该上下文提供，本上下文不承载授权配置。

> 设计重点：**产品与营销**收拢“卖什么、怎么展示、申请页要素、预授信名单与额度”，对应第 1 段（营销获客与预授信）及渠道产品页与申请入口的显示支撑。

## 4. 授权与准入配置上下文

**核心组件**：授权配置管理组件

- **职责**
  - 按**产品 + 客户类型 + 业务阶段**配置：需签署的授权文件（征信、公积金、政采等）、适用关联人类型、需执行的风控规则包绑定、授权检查模式（强制/仅提示/仅记录）；为授权准入、进件、签约提供“本阶段要签哪些、执行哪些规则”的查询服务。
  - 不执行授权签署或风控规则，仅提供配置与查询；规则包内容在风控上下文，协议模板在签约与合同上下文。
- **持有的数据（主责任）**
  - 授权配置（阶段×产品×客户类型×授权文件列表×适用关联人×规则包 ID 列表×检查模式）。
- **对外能力**
  - 按产品+客户类型+业务阶段查询授权文件列表与规则包 ID 列表，供客户授权准入组件、进件、签约使用。
- **与其他上下文的集成方式**
  - **风控上下文**：仅引用规则包 ID，规则执行由风控上下文完成；授权准入阶段由调用方（客户授权准入组件等）根据本上下文返回的规则包 ID 调用风控。
  - **签约与合同上下文**：授权协议模板由合同模版管理提供，本上下文通过“授权文件编码”与模板关联。
  - **产品与营销上下文**：产品主数据由产品与营销提供，本上下文按产品维度配置授权与规则。
  - **进件与流程上下文**：进件/授权准入流程在需要时查询本上下文，决定是否展示授权入口、是否拦截。

> 设计重点：**授权与准入配置**是第 3 段（授权准入与规则触发）的**配置来源**，负责“业务申请环节所需的授权书与风控模型选择”，不承载流程执行。

## 5. 外部数据上下文

**核心组件**：征信与外部数据管理组件

- **职责**  
  - 统一接入人行/百行征信、公积金、税务、政采平台、公安联网核查、工商司法等外部数据源。  
  - 统一管理授权与调用合规、查询留痕与推送接收。
- **持有的数据（主责任）**  
  - 各外部数据源的查询记录与结果（标准化后）。  
  - 外部推送白名单/客户信息等的接收记录。
- **对外能力**  
  - 标准化的数据查询接口。  
  - 推送数据分发（如外部白名单 → 预授信组件）。
- **与其他上下文的集成方式**  
  - **风控上下文**：规则执行时通过本上下文获取外部数据。  
  - **预授信、贷后上下文**：消费推送名单和外部数据。

## 6. 风控上下文

**核心组件**：风控组件（规则引擎）、与之相关的规则配置域（部分配置在授权与准入配置、产品与营销中）

- **职责**  
  - 统一执行预授信、授权准入、进件准入、三合一节点、签约前、放款前、贷后等各阶段的规则包。  
  - 输出规则结论：通过/拒绝/转人工/预警等级/额度测算结果等。
- **持有的数据（主责任）**  
  - 规则包定义与版本（可与外部风控平台/规则平台协同）。  
  - 规则执行日志（可选）：请求参数摘要、结果、耗时、来源上下文等。
- **对外能力**  
  - 按规则包 ID 执行规则，返回结构化结果（结论、分值、标签、建议处置等）。  
  - 提供规则执行日志查询（用于排查与审核）。
- **与其他上下文的集成方式**  
  - **外部数据上下文**：通过“征信与外部数据管理组件”获取外部数据，自己不直接对接外部接口。  
  - **进件与流程上下文**：被进件管理调用执行准入规则，结果驱动路由与状态。  
  - **授权/签约/放款上下文**：被授权准入组件、签约组件、放款组件在对应阶段调用。  
  - **贷后上下文**：被贷后管理/预警组件或批处理调用执行贷后规则包。

> 设计重点：风控上下文是**纯规则执行者**，不参与业务状态机，与“什么时候执行哪些规则包”解耦。

## 7. 进件与流程上下文

**核心组件**：进件管理组件、工作流/审批流组件、（与之紧耦合的）同步策略表  

- **职责**
  - 统一接收各类进件（授信/贷款/提额），维护进件生命周期与状态机。
  - 根据风控结果与**产品/进件类型/渠道等配置**驱动路由：**R1 直通**（生成签约任务等）或 **R3 转线下**；转线下时**按配置确定办贷任务类型**（三合一 / 二合一(授信+用信) / 二合一(评级+授信) / 单独模式），将路由结果与办贷任务类型（或流程模板 ID）交给工作流，由工作流启动对应办贷流程模板，不写死「转人工 = 三合一」。
  - 在状态迁移时按同步策略表调用其他上下文的业务组件（客户、授信、用信、额度、合同、评级等）的 `upsertByIntake` 类接口。
- **持有的数据（主责任）**
  - 进件单（类型、产品、客户标识、渠道、申请要素等）。
  - 进件状态、**路由结果（R1/R3）**、**办贷任务类型或流程模板 ID**（R3 时）、同步执行日志（成功/失败、目标模块、动作类型等）。
- **对外能力**
  - 进件接收、状态迁移、查询。
  - 按进件维度发起同步（基于策略表）。
  - 触发工作流流程实例：直通分支生成签约任务；转线下分支按**办贷任务类型配置**启动对应办贷流程模板（三合一、二合一、单独评级/授信/用信），审批节点完成时回调进件管理更新状态。
- **与其他上下文的集成方式**
  - **风控上下文**：同步调用风控组件执行进件准入规则包，消费结果（R1/R3/拒绝等）。
  - **签约与合同上下文**：在 R1 或审批通过时生成合同与签约任务；在签约完成时被回调以更新状态为 `SIGNED` 并触发同步。
  - **授信/额度/用信上下文**：在不同状态下按策略调用相应 upsert 接口，实现早/晚同步与多次更新；办贷任务（三合一/二合一/单独）由工作流节点内按模板调用该上下文各组件。
  - **贷后/报表上下文**：通过落地的一致数据模型间接支持（不直接依赖）。

- **办贷任务类型与配置（设计要点）**
  - **办贷任务类型**：三合一（评级+授信+用信）、二合一(授信+用信)、二合一(评级+授信)、单独模式（单独评级/授信/用信），由**产品 + 进件类型 + 渠道（线上转线下 / 纯线下）**等维度配置决定；进件管理根据配置解析出办贷任务类型或流程模板 ID，工作流提供多套办贷流程模板并据此启动实例。
  - **线上转线下**：常见为三合一或二合一(授信+用信)；**纯线下办贷**：可配置三合一、两种二合一、单独模式等，同一套进件与流程上下文通过配置支撑多种办贷形态。

> 设计重点：**进件与流程上下文只依赖“配置 + 统一接口约定”操作其他上下文**，不直接读写它们的数据表；**转线下时的办贷形态由配置驱动**，不写死单一任务类型。

## 8. 授信 / 额度 / 用信上下文

**核心组件**：授信管理组件、额度管理组件、用信管理组件、评级管理组件、利率与费用组件

- **职责**
  - 授信管理：授信申请与结论、授信合同生成/关联、三合一牵头、利率/额度调整和额度激活入口。
  - 额度管理：额度台账维护、额度测算结果落库、额度占用与释放、额度调整/激活。
  - 用信管理：在授信/额度之下记录贷款/用信明细，承接放款与还款。
  - 评级管理：客户/授信维度的评级结果维护，为授信额度与定价提供输入；贷后风险缓释时可下调或更新评级。
  - 利率与费用：按产品与规则配置提供授信定价、用信/合同利率与费用、应还本息费与罚息计算；被授信管理、用信管理、合同管理、还款组件调用。
- **持有的数据（主责任）**
  - 授信记录及其状态（含额度、期限、担保方式、审批结论等）。
  - 额度台账（授信维度/客户维度等可配置的额度维度）。
  - 用信/贷款记录（放款金额、币种、期限、还款方式、当前状态等）。
  - 评级结果（客户/授信维度、版本与生效策略）。
  - 利率与费用规则及计算结果缓存（可选）。
- **对外能力**
  - 授信/用信/额度的查询、初始化（通过进件同步）、审批结果写入、额度调整/激活。
  - 可用额度校验（放款与贷款申请前）。
  - 评级查询与更新（含贷后发起的评级下调）。
  - 利率与费用计算（授信定价、应还本息费/罚息等）。
- **与其他上下文的集成方式**
  - **进件与流程上下文**：通过同步策略在 RECEIVED/APPROVED/SIGNED 等状态下被按进件维度 upsert（含客户、评级、授信、用信、额度）。
  - **签约上下文**：生成授信合同/借款合同；合同生效后确认授信/用信终态。
  - **支付与账务上下文**：放款与还款结果回调后，更新用信状态与额度占用；还款组件调用本上下文的利率与费用计算应还本息费/罚息。
  - **贷后上下文**：提供授信/额度/用信明细作为贷后预警与风险缓释的输入；接收贷后发起的评级下调、额度冻结/下调等风险缓释请求。

> 设计重点：授信/额度/用信（含评级、利率与费用）在**内部维护强一致**（同一聚合内），对外通过进件和放款/还款事件来感知流程。

## 9. 签约与合同上下文

**核心组件**：合同管理组件、合同（协议）模版管理组件、签约组件

三者职责与关系需严格区分，便于详细设计与实现边界清晰：

### 9.1 三组件职责与数据归属（细化）

| 组件 | 唯一职责（核心） | 主责数据 | 不负责 |
|------|------------------|----------|--------|
| **合同（协议）模版管理组件** | 合同/协议的**模板内容与结构**管理：条款正文、变量定义、版本；按类型/编码/版本提供只读模板。 | 模板编码、合同/协议类型、版本、条款结构、变量定义（名称、类型、填充来源或规则）。 | 不生成合同实例；不参与签约流程；不持有客户/授信/用信等业务数据。 |
| **合同管理组件** | 合同**实例**的全生命周期：按业务主键（进件/授信/用信等）**生成**合同实例、关联业务实体、提供可展示内容、接收签署结果并更新状态/归档。 | 合同实例（合同ID、类型、版本、状态、关联进件/客户/授信/用信、填充后内容或引用、签署结果摘要）。 | 不维护签约任务列表与签约流程；不维护模板正文与变量定义（只消费模版组件）；不决定「何时生成哪份合同」（由进件/工作流/授信等触发）。 |
| **签约组件** | **签约流程与任务**的编排与执行：待签任务创建与列表、发起签约、拉取展示内容、调用风控/短信/人脸、提交签署结果并回写合同管理。 | 签约任务（任务ID、任务类型、关联合同ID、进件ID、客户、状态、截止时间、签约入口标识）。 | 不生成或维护合同正文/模板；不决定何时生成哪份合同；需要展示时只向合同管理要「可展示的合同内容」。 |

### 9.2 三者调用与数据流关系

- **合同模版管理** → 仅被调用、不主动调用业务：**合同管理**在生成实例时按合同类型/产品/版本向模版管理拉取模板与变量定义；**签约组件**不直接调模版管理，展示用内容由**合同管理**提供（合同管理内部用模版 + 业务数据生成可展示内容）。
- **合同管理** → 被进件/工作流/授信触发「生成合同实例」；被签约组件「查询合同内容（展示用）」与「接收签署结果并更新状态/归档」；生成实例时内部可调模版管理 + 授信/用信/产品等取填充数据。
- **签约组件** → 被进件/工作流/授信/合同管理触发「创建签约任务」；签约流程中调**合同管理**取合同展示内容，调风控/消息/人脸做校验，签署完成后将结果回写**合同管理**；合同管理更新为已签署/生效后，签约组件通知进件管理或将事件交给进件管理驱动 SIGNED 与同步策略。

### 9.3 上下文级职责与对外能力（汇总）

- **职责**  
  - 合同模版：维护并输出合同/协议模板（类型、编码、版本、变量定义）。  
  - 合同实例：按业务上下文生成合同实例，关联进件/客户/授信/用信，提供展示内容，接收签署结果并完成状态更新与归档。  
  - 签约流程：统一承载待签任务、多入口签约（面客 + 办贷 PAD）、校验与签署结果回写。
- **持有的数据（主责任）**  
  - 合同模板（类型、编码、版本、变量定义）。  
  - 合同实例（与进件/客户/授信/用信关联的具体合同及状态）。  
  - 签约任务（任务类型、渠道、状态、截止时间）。
- **对外能力**  
  - 按合同类型与业务上下文生成合同实例（合同管理）。  
  - 查询待签任务列表、发起签约、提交签署结果（签约组件）。  
  - 将合同状态更新为已签署/生效，并记录归档信息（合同管理）。
- **与其他上下文的集成方式**  
  - **进件与流程上下文**：进件/工作流触发合同生成与签约任务创建；签约完成后回调进件管理更新状态为 `SIGNED`。  
  - **风控/安全上下文**：签约组件在签约前按产品配置调用风控与安全能力（短信、人脸等）。  
  - **授信/额度/用信上下文**：合同实例关联授信/用信记录，用于确认业务关系与后续还款、贷后处理。  

**配置归属（设计约定）**：
- **什么业务签订什么合同**（授信合同、借款合同、放款申请书、股东会决议等）由**合同管理**的配置决定（如产品+进件类型/流程阶段→合同类型与生成时机）。
- **什么业务签什么协议**（征信授权、公积金授权、政采授权等授权协议）由**授权配置管理**决定（产品+客户类型+业务阶段→授权文件列表及适用关联人）。

> 设计重点：签约与合同上下文**不决定业务逻辑**（是否允许签约、签约后写哪些业务表），只负责“合同 + 签署”；**模版管结构、合同管实例、签约管流程与任务**，三者边界清晰；**合同与协议的配置归属分离**，便于详细设计。

## 10. 支付与账务集成上下文

**核心组件**：放款组件（渠道+办贷）、还款组件（渠道+办贷），与核心/账务系统的接口

- **职责**
  - 放款：接收提款/放款指令，校验合同/额度/用信状态与放款前风控、受托支付审批，调用核心执行放款并处理回盘。
  - 还款：接收还款指令（正常/提前结清/部分提前），按授信/额度/用信上下文的**利率与费用组件**计算应还本息费/罚息，调用核心执行扣款与入账，并维护还款计划执行情况。
- **持有的数据（主责任）**
  - 放款/还款指令与状态、核心流水号与对账结果。
  - 还款计划、执行明细（也可与用信管理联合维护）。
- **对外能力**
  - 放款/还款指令受理与状态查询。
  - 与核心/账务的交易对接与对账信息。
- **与其他上下文的集成方式**
  - **授信/额度/用信上下文**：放款/还款成功后更新用信状态与额度占用；还款计息依赖其利率与费用能力。
  - **进件与流程上下文**：放款成功后通知进件管理将状态更新为 `DISBURSED`。
  - **贷后与预警上下文**：将还款/逾期情况作为贷后监控输入。  

---

## 11. 贷后与预警上下文

**核心组件**：贷后管理/预警组件、押品组件

- **职责**  
  - 监控还款行为、押品价值、外部数据变化，执行贷后规则，生成预警记录与贷后工单。  
  - 支持贷后检查、催收、押品重估与风险缓释（额度/授信/评级调整等）。
- **持有的数据（主责任）**  
  - 贷后检查记录、预警记录与工单、处置结果。  
  - 押品信息与估值、价值率等（由押品组件维护）。  
- **对外能力**  
  - 贷后预警规则的配置与执行（可委托给风控规则引擎）。  
  - 贷后工单管理与处置结果记录。  
  - 向评级/授信/额度发起风险缓释操作请求。
- **与其他上下文的集成方式**  
  - **用信/还款上下文**：消费用信与还款/逾期数据。  
  - **外部数据上下文**：获取征信/司法/工商等贷后数据。  
  - **进件与流程上下文**：使用工作流/审批流管理贷后工单与风险缓释审批。  
  - **报表与监管上下文**：输出五级分类、不良贷款、预警分布等指标。

---

## 12. 报表与监管上下文

**核心组件**：监管报送与报表组件

- **职责**  
  - 作为统一数据出口，按监管口径和内部管理口径生成各类报表与数据集，并支持报送与内部分析。
- **持有的数据（主责任）**  
  - 报表定义与版本、报表结果集、报送批次与状态。
- **对外能力**  
  - 报表查询与导出、报送任务管理。  
  - 为外部监管系统和内部管理视图提供数据。
- **与其他上下文的集成方式**  
  - **客户与身份、授信/额度/用信、贷后与预警等上下文**：通过统一视图抽取数据。  
  - **批处理与调度上下文**：由批处理统一调度生成与报送任务。  

---

## 13. 批处理与调度上下文

**核心组件**：批处理组件

- **职责**  
  - 统一调度各类批量任务：批量预授信、进件同步补偿、还款计划与逾期更新、贷后预警批量扫描、报表生成与报送等。
- **持有的数据（主责任）**  
  - 批任务定义与调度配置、执行日志、告警与重试记录。
- **对外能力**  
  - 按任务标识调度对应业务组件批量执行。  
  - 提供任务执行情况查询与重试接口。
- **与其他上下文的集成方式**  
  - 对各业务上下文仅做“调度”，不直接读写业务表，由对应业务组件保证幂等与补偿逻辑。

## 14. 通用安全与消息上下文

**核心组件**：系统管理组件（认证/安全策略）、安全组件、消息中心、人脸识别服务

- **职责**  
  - 定义认证与安全策略，为注册/登录/授权/签约/提款等操作提供统一的认证与安全能力。  
  - 统一提供短信、站内信等消息发送服务。
- **持有的数据（主责任）**  
  - 认证策略、登录安全策略、认证/验证记录与审计日志。  
  - 消息发送记录与通道配置。
- **对外能力**  
  - 认证策略查询、验证码发送与校验、人脸识别调用、消息发送。
- **与其他上下文的集成方式**  
  - 被注册、授权、签约、放款、还款、贷后通知等各上下文在需要时调用，不反向依赖具体业务表。
