# 产品流程 × 组件调用场景视图与配置表

本文档将各产品的业务办理流程与《组件设计-组件调用关系总览》对齐，形成：**按产品的组件调用场景视图** 与 **实施时需维护的配置表**，便于按产品落地实现与配置管理。

**数据源约定**：业务阶段与**说明**列以 `1.产品经理输入：产品业务流程/产品业务流程与组件综合分析.csv` 为**唯一准表**；表中「说明」列（晚同步/早同步、R1/R3、黑名单拦截时机、申请内授权/独立授权、提额可选、白名单维护等）均从该 CSV 反推并保持与设计文档一致。

---

## 一、组件调用场景视图（按产品）

以下按产品列出「业务阶段 → 主调组件 → 被调组件链」的调用关系，与总览中的前置条件一致；表中「主调」为流程入口或编排方，「被调链」为在该阶段内依次或并行调用的组件。

### 1.1 湖北农信-公积金贷款（线上）

| 业务阶段 | 主调/入口 | 组件调用链（主调→被调） | 说明 |
|----------|-----------|--------------------------|------|
| 1、客户注册认证 | 渠道前端 | 用户注册组件 → 客户管理组件（临时客户）、系统管理/消息/人脸/安全 | 先注册认证再申请（个人客户）；客户管理组件（办贷端）提供【临时客户创建接口】 |
| 2、查询产品信息 | 渠道前端 | 营销方案管理组件 → 产品管理组件、利率与费用组件、额度管理组件、合同管理组件、预授信组件 | 展示产品列表/详情、利率优惠券、额度/合同可用额，并通过预授信组件展示预计可贷款金额，通过产品管理组件展示产品最高可贷款金额；本阶段不做黑/白名单拦截，是否可发起申请由后续申请前规则决定 |
| 3、客户发起贷款申请 | 营销方案（产品）管理组件 | 贷款申请组件 → 营销方案管理组件（申请页初始化）→ 风控组件（**申请前规则包：反欺诈、黑/白名单、产品参数准入**，按产品配置）→ 客户管理组件（补录）→ 客户授权准入组件 → 授权配置管理组件 → 合同管理组件（授权协议可展示内容）→ 消息/人脸 → 签约组件（授权）→ 进件管理组件 | 客户需先完成注册认证；申请页初始化由营销方案管理组件提供；提交前按产品配置执行申请前规则包：未通过则在本阶段直接拦截（不创建进件、也不进入授权）；通过后才进入授权准入与后续进件；授权协议展示由合同管理提供；**申请内授权** |
| 4、进件接收 | 贷款申请组件（渠道端） | 进件管理组件 → 授权配置管理组件（查询进件阶段规则包 ID）→ 客户管理组件 → 风控组件（准入，规则包由授权配置提供）→ [R1]签约组件（生成签约任务）/ [R3]进件管理组件（转线下，按配置办贷任务类型） | **晚同步**：进件环节至落地进件信息，不同步/评级/授信/用信信息；按业务环阶段「产品+贷款进件阶段」查询当前需要执行的规则包；**R1 转线下生成签约任务** |
| （可选）客户发起提额申请 | 营销方案（产品）管理组件 | 提额组件 → 营销方案管理组件（申请页初始化）→ 风控组件（申请前规则包）→ 消息/人脸 → 签约组件（授权）→ 进件管理组件 | 客户需先完成注册认证及前置贷款申请；申请前规则包未通过则本阶段直接拦截；**申请内授权** |
| （可选）提额进件接收 | 提额组件（渠道端） | 进件管理组件 → 授权配置管理组件（查询进件阶段规则包 ID）→ 客户管理组件 → 风控组件（准入）→ [R1]签约组件（生成签约任务）/ [R3]进件管理组件（转线下） | 晚同步：进件环节至落地进件信息，不同步/评级/授信/用信信息；需有前置授信申请或贷款申请 |
| 5、客户在线签借款合同 | 渠道前端 | 签约组件 → 合同管理组件（可展示内容）→ 消息/人脸/安全 → 合同管理组件（签署回写）→ 客户/用信/授信/评级/合同管理组件（SIGNED 时同步）→ 进件管理组件（状态） | **晚同步**：合同签订完成，同步/评级/授信/用信信息，更新进件状态；展示内容由合同管理提供 |
| 6、客户发起提款申请 | 渠道前端 | 放款组件（渠道）→ 用信管理/额度管理/合同管理组件 → 安全/人脸/消息 → 签约组件（放款申请书）→ 放款组件（办贷）→ 核心系统 → 进件管理组件 | 一个借款合同项下支持多次提款；提款前风控/验证按产品配置；提款环节需要签订放款申请书；更新进件状态 |
| 7、受托支付审批 | 办贷/工作流 | 工作流/审批流组件 → 放款组件（办贷） | 按产品配置是否启用 |
| 8、客户还款 | 渠道前端 | 还款组件（渠道）→ 用信管理组件/利率与费用组件 → 还款组件（办贷）→ 核心/贷后/监管 | — |

## 三、使用说明

- **组件调用场景视图**：按产品、按业务阶段查看「谁在什么环节调用谁」，与《组件设计-组件调用关系总览》中的前置条件一致；实现时需保证调用顺序与前置条件满足关系一致。
- **配置表**：实施时在系统中维护对应配置（或通过参数表/配置中心落地）；新增产品时在 2.1～2.5 各表中增加一行或若干行，并同步更新《产品规则-回归用例矩阵》与本文档的视图。
- **对公统一前置**：对公客户办理业务需要先在渠道端先完成注册认证；客户类型为对公（CORPORATE）的产品，须在渠道端「用户注册组件 + 客户管理组件」完成注册与基础认证后，才允许调用授信申请/贷款申请/提额等后续组件；2.1 表「申请前是否要求注册登录」对公产品均为「是」。

若某产品需细分子流程（如不同渠道、不同担保方式），可在上述配置表中通过「产品编码+子类型」或扩展列区分，并在场景视图中增加对应列或子表。
