# 汇报目录说明

本文档说明当前文档结构的**汇报目录**设计、合理性分析及优化建议，便于对外汇报或评审时使用。

---

## 一、当前目录结构（与《_sidebar》一致）

| 层级 | 章节 | 对应文档 |
|------|------|----------|
| 1 | **首页** | README |
| 2 | **组件设计（按段落）** | 第1段～第11段（共 11 篇） |
| 3 | **总览与风控** | 组件调用关系总览、风控组件与规则触发总览 |
| 4 | **产品与流程** | 业务步骤级组件分析、组件调用场景视图与配置表、回归用例矩阵 |

| 6 | **发布说明** | 如何发布到服务器 |

## 二、合理性分析

### 2.1 做得好的地方

- **段落完整**：第 1～11 段覆盖从营销获客到批处理与流程衔接，与业务办理顺序一致，便于按流程查阅。
- **总览与风控独立成章**：组件调用关系、风控触发点单独列出，方便架构与风控专题汇报。
- **产品与流程集中**：业务步骤分析、场景视图与配置表、回归矩阵放在一起，便于“产品视角”查阅。
- **进件与同步单列**：同步策略是跨段落的横切设计，单独成节合理。
- **发布说明收尾**：操作类内容放在最后，符合“先设计后使用”的阅读习惯。

### 2.2 可优化点

| 问题 | 说明 | 建议 |
|------|------|------|
| **汇报动线** | 当前顺序为「首页 → 组件设计(11段) → 总览 → 产品与流程 → 进件 → 发布」。汇报时听众更习惯「先见森林再见树木」：先看整体架构与调用关系，再看业务如何跑，最后按段落看细节。 | 将 **「总览与风控」** 提前到 **「组件设计（按段落）」** 之前，形成「首页 → 总览与风控 → 产品与流程 → 组件设计（按段落）→ 进件与同步 → 发布说明」。 |
| **总览的定位** | 总览类文档是理解全貌的入口，放在段落设计之后，容易被忽略。 | 总览紧跟首页，作为“架构与风控”总图。 |
| **命名一致性** | 侧栏写「回归用例矩阵」，文档名为「产品规则-回归用例矩阵」。 | 汇报目录中可写全称「产品规则回归用例矩阵」，或保留简称并在 README 中说明对应文件名。 |
| **组件清单未入目录** | 组件说明_v2.csv 是组件清单的权威来源，当前仅在 README 中链接，未在导航目录体现。 | 若汇报需要“组件清单”入口，可在「总览与风控」或「附录」中增加「组件清单（v2）」链接或说明。 |
| **进件与同步扩展** | 若后续补充《进件管理组件详细设计》，需有明确归属。 | 放在「进件与同步」下，与「进件中心-业务数据同步策略设计」并列。 |
| **发布说明标题** | 当前为「如何发布到服务器」，偏操作步骤。 | 汇报目录可显示为「发布与使用说明」，与《发布说明.md》内容一致。 |

## 三、优化后的汇报目录（推荐）

按「总览先行 → 产品与流程 → 段落设计 → 核心机制 → 使用说明」的汇报逻辑，推荐结构如下：

```
1. 首页
   · README

2. 总览与风控
   · 组件调用关系总览
   · 风控组件与规则触发总览
   · （可选）组件清单说明 / 组件说明_v2 引用

3. 产品与流程
   · 业务步骤级组件分析
   · 组件调用场景视图与配置表
   · 产品规则回归用例矩阵

4. 组件设计（按段落）
   · 第1段 营销获客与预授信
   · 第2段 客户注册与认证
   · 第3段 授权准入与规则触发
   · 第3.5段 授信申请/贷款申请与提额
   · 第4段 进件接收与风控准入
   · 第5段 三合一任务与审批
   · 第6段 签约与合同管理
   · 第7段 放款与受托支付审批
   · 第8段 还款
   · 第9段 贷后管理与预警
   · 第10段 监管报送与报表
   · 第11段 批处理与流程衔接

5. 进件与同步
   · 进件中心-业务数据同步策略设计
   · （预留）进件管理组件详细设计

6. 发布与使用说明
   · 如何发布到服务器
```

---

## 四、使用建议

- **对外汇报 / 评审**：采用「三、优化后的汇报目录」顺序，便于先讲架构与风控、再讲产品与流程、再按段落展开设计、最后讲进件同步与发布。
- **日常开发查阅**：若更习惯按业务段落顺序查文档，可维持当前 _sidebar 顺序（组件设计在前）；或通过 README 的「文档结构」表格快速跳转。
- **Docsify 侧栏**：若希望站点导航与汇报顺序一致，可将 `_sidebar.md` 的章节顺序调整为上述「三」中的顺序；若希望与现有文件名/习惯一致，可保持现有顺序，仅在汇报 PPT 或汇报提纲中采用优化顺序。
