## 组件设计分析 — 第 2 段：客户注册与认证

按**业务办理过程顺序**进行组件分析，本段覆盖流程主线中的**客户注册 / 认证 / 客户信息建档**环节，确保该段内组件对已分析的全部产品支撑不遗漏，并支持「显式注册 + 自动注册」「临时客户 + 正式客户」「对私 + 对公」多种模式。

**本段涉及的组件**（办贷管理端 v2 + 渠道端）：

- 用户注册组件（渠道）
- 客户管理组件（办贷）
- 系统管理组件（办贷，认证策略 / 安全能力的承载者之一）
- 通用安全能力：人脸识别服务、短信验证码（消息中心）、安全组件（交易密码等）

**已对照产品**：湖北农信-公积金贷款（线上/线上转线下）、湖南农信-公积金贷款、长沙银行-税E贷、长沙银行-政采贷。

## 一、用户注册组件（渠道端视角）

### 1.1 职责与边界

| 项目 | 说明 |
|------|------|
| **职责** | 在渠道侧完成客户账号注册与登录入口，包括对公/对私注册、第三方平台对接注册、基础认证（如手机号、证件号、人脸）等，并与办贷端的客户管理对接，驱动「临时客户 → 正式客户」的升级。 |
| **边界** | 不直接维护客户业务主数据（客户名称、证件、联系信息等最终存储），这些由**客户管理组件**负责；不独自定义认证策略，只承载前端交互流程，认证策略与结果记录由**系统管理组件**与客户/认证域负责。 |

### 1.2 多产品 / 多模式支持点

- **支持对私 + 对公注册**：通过配置控制不同渠道/产品是否开放对公注册，对对应字段集与校验规则做可配置（如企业名称、统一社会信用代码、法定代表人等）。
- **支持第三方平台对接注册**：如微信、政务 APP、外部业务平台等，通过单点登录或 OAuth 接入，按第三方返回的身份信息在办贷端创建或映射客户记录。
- **显式注册 + 自动注册**：
  - 显式注册：客户主动点击「去注册」，按流程填写信息并完成认证。
  - 自动注册：在贷款/授信申请过程中，如果检测到客户未注册，可在后台根据申请信息和第三方身份信息自动创建账号，并标记认证状态（如「弱认证」）。
- **外部客户推送自动注册（可配置）**：当征信与外部数据管理组件推送的客户信息满足已配置的「自动注册条件」（如来自高可信平台且已做实名验证）时，可直接触发账号创建与绑定；否则仅在客户管理组件中建档，账号在首次登录或办理业务时再由用户注册组件完成绑定。
- **登录与会话**：已注册客户可通过账号（手机号/证件号/第三方 ID）+ 密码/验证码登录，后续所有申请、预授信、进件等操作都在登录上下文中进行。

### 1.3 关键接口与上下游关系

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 客户注册 | 采集注册信息并进行基础校验，调用办贷端接口创建临时/正式客户 | 下游：客户管理组件（创建/更新客户）；上游：渠道页面 |
| 认证操作 | 调用人脸识别、短信验证码等通用能力，完成实名认证或增强认证 | 下游：人脸识别服务、消息中心、安全组件；上游：渠道页面 |
| 自动注册/认证 | 在进件或申请过程中，由渠道或办贷逻辑决定触发自动注册/认证 | 下游：客户管理组件、系统管理组件（认证记录）；上游：贷款/授信申请流程 |
| 登录与会话管理 | 登录、登出、会话状态管理 | 下游：系统管理组件（认证策略）、安全组件；上游：所有渠道功能 |

### 1.4 对已分析产品的支撑检查

| 产品 | 本段内相关步骤 | 组件支撑 | 是否遗漏 |
|------|----------------|----------|----------|
| 湖北农信-公积金-线上/线下 | 「客户注册」「认证操作」「临时客户创建」 | 显式注册 + 基础认证能力齐备，临时客户创建交由客户管理组件 | 无遗漏 |
| 湖南农信-公积金 | 「客户注册认证（可选操作）」「系统自动注册客户」「系统自动认证客户」 | 显式注册为可选；自动注册/认证由用户注册组件 + 客户管理组件 + 系统管理组件协作 | 无遗漏 |
| 长沙银行-税E贷 | 「客户注册认证（可选操作）」 | 支持可选注册：客户也可从预授信/授权/授信流程进入并在过程中触发注册 | 无遗漏 |
| 长沙银行-政采贷 | 「客户注册认证（可选操作）」 | 同税E贷 | 无遗漏 |

**小结**：用户注册组件需支持对公/对私、多渠道/三方接入，以及「显式 + 自动」两种注册/认证模式；对客户主数据与认证结果的最终落地依赖客户管理与系统管理。

## 二、客户管理组件（办贷视角）

### 2.1 职责与边界

| 项目 | 说明 |
|------|------|
| **职责** | 维护客户主数据（包括对公与对私）、客户状态（临时/正式/黑名单等）、客户标签与关联关系（集团、上下游、股东等）；为进件、授信、用信、预授信、股东会决议等提供统一客户视图。 |
| **边界** | 不直接负责授信结论、额度、评级、合同等业务数据；不承载认证策略决策逻辑，认证记录可关联到客户，但决策本身在系统管理/认证域；不执行风控，只提供基础客户信息与标签给风控组件消费。 |

### 2.2 多产品 / 多模式支持点

- **临时客户 vs 正式客户**：
  - 支持在注册或申请阶段创建「临时客户」记录（信息不完整、尚未完成全部认证/建档）；
  - 在签约、审批通过或专门建档环节，将临时客户升级为「正式客户」。
- **对公 + 对私统一建模**：
  - 提供统一 `customer_id`，下挂个人/企业不同属性集；支持混合场景（如个人股东、实际控制人等）；
  - 对公客户建档支持尽调信息（营业执照、股东结构、司法/税务状况等）扩展。
- **多渠道、多源创建**：
  - 渠道注册、办贷端客户经理建档、外部平台导入/推送（如政采平台等）等均可创建或更新客户；
  - 提供按外部来源标识（如 `external_source`）+ 外部客户 ID、证件号、手机号等多键查找与去重策略，避免重复建档。
- **标签与黑名单**：
  - 标签用于营销与风控（如优质、公积金缴存客户、政采供应商等），标签可配置；
  - 黑名单标记存于客户管理，风控组件在规则中消费。

### 2.3 关键接口与上下游关系

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 创建/更新客户（按注册/申请） | 按注册/申请信息创建或更新客户记录，可指定是否为临时客户 | 下游：客户表、标签表；上游：用户注册组件、进件管理组件 |
| 升级临时客户为正式客户 | 在签约或建档环节，将临时客户状态升级为正式，补全必填信息 | 下游：客户表；上游：签约组件、进件管理组件、客户经理前台 |
| 对公客户建档与补录 | 对公场景下，客户经理补录企业信息、尽调信息 | 下游：客户/尽调相关表；上游：客户经理前台 |
| 客户查询（统一视图） | 提供按 `customer_id`、证件号、外部 ID 等多种查询方式 | 下游：无；上游：进件、预授信、授信、用信、评级、股东会决议等 |
| 黑名单/标签管理 | 维护黑名单状态与标签集合 | 下游：标签/黑名单表；上游：风控、营销、预授信 |

### 2.4 对已分析产品的支撑检查

| 产品 | 本段内相关步骤 | 组件支撑 | 是否遗漏 |
|------|----------------|----------|----------|
| 湖北农信-公积金-线上/线下 | 「临时客户创建」「补充临时客户信息」「创建/更新正式客户」「客户经理个人客户创建」「补录客户信息」 | 临时/正式客户模型、补录与升级能力完整；对公/对私建档通过字段集与扩展满足 | 无遗漏 |
| 湖南农信-公积金 | 「系统创建客户」「客户经理个人客户创建」 | 进件接收时由进件管理调用客户管理创建客户；线下再补录 | 无遗漏 |
| 长沙银行-税E贷 | 「系统创建客户」「客户经理补录客户信息」 | 同上；需支持税务/工商等外部数据补录字段 | 无遗漏 |
| 长沙银行-政采贷 | 「系统创建客户」「客户经理补录客户信息（建档/尽调）」 | 需支持政采类供应商信息、尽调扩展字段；客户管理以扩展字段满足 | 无遗漏 |

**小结**：客户管理组件需清晰建模「临时/正式」「对公/对私」「多源创建 + 客户经理补录 + 自动注册」等能力，现有产品步骤均可映射；后续在贷后与预警段落中还会复用统一客户视图。

## 三、系统管理组件（与注册/认证相关的部分）

> 系统管理组件的完整能力更广，这里仅聚焦与「客户注册与认证」强相关的部分。

### 3.1 职责与边界（本段）

| 项目 | 说明 |
|------|------|
| **本段职责** | 定义与管理认证策略（密码强度、短信/人脸/设备指纹等多因子策略）、登录安全策略（登录失败次数限制、风控前置黑名单检验等）、认证记录与审计日志；为用户注册组件与其他渠道功能提供统一的认证能力配置。 |
| **边界** | 不直接存储客户主数据；不执行业务审批流程（那是工作流组件的职责）；认证流程的前端交互由用户注册组件与渠道承载，系统管理提供策略与结果记录。 |

### 3.2 多产品 / 多模式支持点

- **多渠道、多产品策略**：可按渠道、产品、客户类型配置认证与登录策略（如某些高风险产品强制人脸+短信、多渠道登录策略不同）。
- **风险前置校验**：在登录/注册/关键操作前调用黑名单或风险规则（与风控组件协作）进行前置拦截。
- **认证等级与扩展**：支持将认证结果分为不同等级（如弱认证、强认证），业务组件可按认证等级决定是否允许申请某类产品或走直通流。

### 3.3 关键接口与上下游关系（本段）

| 接口/能力 | 说明 | 上游/下游 |
|-----------|------|-----------|
| 认证策略查询 | 按渠道/产品/操作类型（注册、登录、授信申请等）返回所需认证方式组合 | 下游：无；上游：用户注册组件、渠道登录模块 |
| 认证结果记录 | 将每次认证（短信、人脸、密码验证等）的结果、设备信息等记录下来，供审计与风控使用 | 下游：认证记录表、审计日志；上游：用户注册组件、其它需要认证的功能 |
| 登录安全策略 | 登录失败次数、锁定策略、验证码触发条件等 | 下游：无；上游：登录模块 |

### 3.4 对已分析产品的支撑检查

| 产品 | 本段内相关步骤 | 组件支撑 | 是否遗漏 |
|------|----------------|----------|----------|
| 所有产品 | 「认证操作」「验证短信验证码」「输入提款密码」「人脸识别操作」（跨段通用） | 系统管理提供认证策略与部分安全能力入口；具体短信/人脸/安全能力由消息中心、人脸服务、安全组件提供 | 无明显遗漏，本段主要是策略与记录层 |

## 四、通用安全能力（人脸、短信、安全组件）

> 这些能力在多个业务段落都会使用，这里仅从「注册与认证」角度明确其平台化定位。

### 4.1 职责与边界

| 能力 | 职责 | 边界 |
|------|------|------|
| 人脸识别服务 | 为注册、登录、授权、签约、提款等环节提供统一人脸识别与活体检测能力 | 不理解业务含义，只做比对与风控特征输出；业务是否通过由系统管理/风控/业务组件决定 |
| 消息中心（短信验证码） | 统一发送短信验证码、通知短信等，并提供验证码验证能力 | 不决定业务逻辑，仅根据请求发送/校验验证码；短信内容模板与频控策略可配置 |
| 安全组件（交易密码等） | 提供交易密码校验、设备指纹、风控前置信息收集等 | 不直接修改业务数据，仅提供安全校验结果与风险评分 |

### 4.2 多产品 / 多模式支持点

- 可按产品或业务操作（如提款、授信签约）配置是否需要人脸/短信/密码等多因子组合。
- 可按客户类型（个人/企业）、客户风险等级区分认证强度（如高风险客户每次提款都强制人脸）。

### 4.3 对已分析产品的支撑检查（本段相关）

| 步骤类型 | 出现产品 | 能力支撑 |
|----------|----------|----------|
| 认证操作 | 湖北/湖南/长沙各产品 | 用户注册组件 + 系统管理策略 + 人脸/短信/安全组件平台能力可统一支撑 |
| 验证短信验证码 | 各产品在注册、授权、签约、提款等多处 | 消息中心统一实现，策略在系统管理中配置 |
| 人脸识别操作 | 各产品在注册、签约、提款等多处 | 人脸识别服务统一实现，业务段落调用 |

## 五、本段组件协作关系小结

```text
[渠道]                                [办贷 v2 / 平台]
   |                                         |
   |  注册/登录页面 ---> 用户注册组件  ---->  客户管理组件（创建/更新客户：临时/正式）
   |                               \\-->  系统管理组件（认证策略/记录）
   |                                  \\-> 人脸识别/短信/安全组件（通用能力）
   |
   |  申请/进件过程中的自动注册/认证 ---> 用户注册组件/进件管理组件
   |                                         |
   |                                         --> 客户管理组件（自动创建/升级客户）
   |                                         --> 系统管理组件（记录认证行为）
```

## 六、与其它段落的衔接

- 与**第 1 段（营销获客与预授信）**：注册/登录状态可能影响营销产品可见性与预授信查询（如仅登录客户可看到预授信额度）。
- 与后续**授权准入段**：授权配置管理组件在授权环节可能再次要求身份认证，本段的安全能力在各段重复复用。
- 与**进件与同步策略**：进件接收时若客户尚未存在，可通过「自动注册 + 自动创建客户」打通注册与进件，相关逻辑与本段设计一致。

本文件为「按业务办理过程顺序」组件分析的第 2 段；第 3 段可继续分析**授权准入（客户授权准入组件 + 授权配置管理组件 + 风控）**。
